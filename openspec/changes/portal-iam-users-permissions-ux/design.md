## Context

Раздел «Роли и доступы» уже разнесён на три страницы (change
`portal-iam-split-pages`, ADR-0017). «Роли» приняли образцовый паттерн «таблица/
каталог + кнопка действия в `PageHeader.actions` → `Dialog` для создания;
`ConfirmDialog` для деструктивных действий». Две страницы отстали:

- **`UsersPage`** (инвентаризация — `web/src/pages/iam/UsersPage.tsx`): сейчас
  держит ДВЕ карточки — большую форму назначения роли с пикером (state: `useForm`
  `{subject, role}`, `subjectSearch`+`useDebounced`, локальный `picked`) и карточку
  с `DataTable` субъектов. Назначение — `useAssignRole`, снятие — `ConfirmDialog` +
  `useRevokeRole`. Пикер: `useDirectorySearch(debounced)`, скрыт при 403
  (`directoryForbidden`), индикация при 503 (`directoryUnavailable`). Всё это надо
  целиком перенести в модалку, не меняя поведения.
- **`PermissionsPage`** (`web/src/pages/iam/PermissionsPage.tsx`): модалка создания
  права уже есть (`Dialog` + `useForm` `{action, resource}` через `form id=
  "create-perm-form"`), удаление — `ConfirmDialog`, тоггл плотности `idp-density`,
  `DataTable` с колонками perm/type/actions и `sortValue`. Поля `action`/`resource`
  — свободный `Input`. Фильтра у таблицы нет.

Общие хуки (`web/src/pages/iam/hooks.ts`) и примитивы (`Dialog`, `ConfirmDialog`,
`DataTable`, `PageHeader`, `Input`, `toast`, `IamGuard`, `lib/errors`,
`lib/ui-state`) переиспользуются как есть. Контракт периметра и кодоген
(`web/src/api/*`, `openapi/openapi.yaml`, `web/public/openapi.yaml`) НЕ трогаем —
`gen:check` обязан остаться зелёным. Эндпоинта «справочник доступных действий/
ресурсов» в контракте НЕТ.

## Goals / Non-Goals

**Goals:**
- `UsersPage`: на экране только `DataTable` субъектов + кнопка «Назначить роль» в
  `PageHeader.actions`; назначение — в `Dialog`, с тем же пикером и поведением.
- `PermissionsPage` (создание): подсказки доступных `action`/`resource` из каталога
  с возможностью валидного выбора И ввода новых значений.
- `PermissionsPage` (таблица): клиентский поиск по `action`/`resource` + фильтр по
  типу `system`/`user`, доступный с клавиатуры, с корректным пустым состоянием.
- Полное сохранение поведения прав/ошибок (403 admin/`directory`/`manage`, 503,
  409/422/400), тостов, `zod .parse`, осиротевших, идемпотентности, инвалидаций.
- Инкрементальный рефактор: `tsc --noEmit` и `vitest` зелёные на каждом шаге.

**Non-Goals:**
- Любые изменения бэкенда/контракта: proto, `services/*`, `openapi/openapi.yaml`,
  коды ответов, RBAC, миграции, новые эндпоинты (в т.ч. «доступные действия/
  ресурсы»).
- Смена стека (фреймворк/роутер/стейт-менеджер), SSR, i18n, тяжёлые UI-киты,
  combobox-библиотеки.
- Переделка страниц «Роли» и `Sidebar` (трогаем только при выделении общего
  фильтр-примитива — и тогда сохраняем их тесты зелёными).

## Decisions

### 1. Источник подсказок действий/ресурсов: производные от `listPermissions`

**Решение.** Подсказки = отсортированные УНИКАЛЬНЫЕ `action` и УНИКАЛЬНЫЕ `resource`,
выведенные из уже загруженного каталога `usePermissionsQuery()` (тот же кэш, без
новых сетевых запросов). Источник вычисляется на странице мемоизированно:
`actions = [...new Set(permissions.map(p => p.action))].sort()` и аналогично для
`resource`.

**Альтернативы.** (а) Статический белый список в коде — отвергнут: дублирует знание
бэкенда, устаревает, не отражает реальный каталог. (б) Новый эндпоинт «доступные
значения» — вне scope (меняет контракт). Производные от каталога — единственный
вариант при фиксированном контракте, и он самодостаточен.

### 2. Контрол ввода: нативный `<datalist>` (подсказки + свободный валидный ввод)

**Решение.** Поля `action`/`resource` остаются `<input>` (примитив `Input`) с
привязкой `list={id}` к нативному `<datalist>`, наполненному уникальными
значениями. Это даёт подсказки И разрешает ввод НОВЫХ значений (создание ранее не
встречавшихся `action`/`resource` — нужно для расширения каталога), сохраняет
`react-hook-form` + `zod` без изменений схемы. Нативный `<datalist>` доступен с
клавиатуры и скринридерами бесплатно, не тянет UI-кит (в дизайн-системе нет
combobox-примитива — ADR-0017).

**Альтернативы.** (а) `<select>` + опция «другое» с раскрытием поля — отвергнут:
сложнее, два состояния, хуже a11y, и ломает «выбор ИЛИ ввод» в один шаг.
(б) Сторонний combobox — нарушает «не тянуть тяжёлые UI-киты». Группировку
ресурсов по namespace (`project:*`, `iam:*`) в MVP не вводим — `<datalist>` это не
поддерживает нативно; оставляем как возможное улучшение.

### 3. Фильтр каталога прав: локальный тулбар над таблицей (без расширения DataTable)

**Решение.** Фильтр живёт на `PermissionsPage` как тулбар-полоса НАД `DataTable`:
поле поиска (`Input`, debounce не нужен — фильтрация клиентская и мгновенная) +
сегмент типа (все / системные / пользовательские). Отфильтрованный массив
вычисляется мемоизированно и передаётся в `DataTable` как `rows`. `DataTable` НЕ
меняем — это исключает риск поломки его существующих использований (Роли/
Пользователи) и их тестов, и сохраняет всю клиентскую сортировку/плотность/
пагинацию (они работают поверх переданных строк).

Пустой результат фильтра показывается штатным `emptyMessage` `DataTable` — задаём
динамический текст «Ничего не найдено» при активном фильтре vs «Каталог прав пуст»
без фильтра, переиспользуя единый empty-state примитива.

**Альтернативы.** (а) Расширить `DataTable` тулбар-слотом/пропом фильтра —
привлекательно для будущего переиспользования на «Ролях», но сейчас раздувает
scope и рискует регрессией на трёх вызовах примитива; решено НЕ делать в этом
change (зафиксировано как улучшение на потом). (б) URL-driven `?q=` — не вводим:
фильтр каталога эфемерный, шеринг тут малоценен, а добавление `useSearchParams`
усложняет тест-матрицу; оставляем как возможное улучшение.

### 4. Модель модалки назначения роли на `UsersPage`

**Решение.** Состояние пикера и формы остаётся ВНУТРИ `UsersPage` (как сейчас:
`useForm` `{subject, role}`, `subjectSearch`, `useDebounced`, `picked`), но весь
текущий JSX формы переносится в `<Dialog>`, открываемый кнопкой «Назначить роль»
из `PageHeader.actions`. Кнопка submit живёт в `Dialog.footer` через
`form="assign-role-form"` (паттерн `RolesPage`/`PermissionsPage`). При успешном
назначении (`onSuccess`) форма сбрасывается (`reset()`, `setPicked(null)`,
`setSubjectSearch("")`) и модалка закрывается; при закрытии модалки (ESC/overlay/
крестик/Отмена) — тот же сброс, чтобы повторное открытие было чистым. Возврат
фокуса на кнопку-инициатор обеспечивает примитив `Dialog`. Инвалидация списка
субъектов и идемпотентность назначения не меняются (логика в `useAssignRole`).

Поведение прав в модалке сохраняется буквально: `directoryForbidden` (403
`iam:directory`) → пикер скрыт с отказом; `directoryUnavailable` (503) → индикация,
поле поиска остаётся; 403 admin на чтении субъектов по-прежнему ловится `IamGuard`
на уровне страницы (модалка открывается только когда раздел доступен).

### 5. Матрица «страница → примитивы / эндпоинты»

| Страница | Примитивы | Хуки/эндпоинты (без изменений) |
|---|---|---|
| Users | `PageHeader`(actions), `Dialog`(назначение), `ConfirmDialog`(снятие), `DataTable`, `Input`, `IamGuard`, `toast` | `useRolesQuery`(listRoles), `useSubjectsInfiniteQuery`(listSubjects), `useDirectorySearch`(searchDirectorySubjects), `useAssignRole`(assignRole), `useRevokeRole`(revokeRole), `useDebounced` |
| Permissions | `PageHeader`(actions), `Dialog`(создание+`<datalist>`), `ConfirmDialog`(удаление), `DataTable`, `Input`(поиск+поля), `IamGuard`, `toast` | `usePermissionsQuery`(listPermissions), `useCreatePermission`(createPermission), `useDeletePermission`(deletePermission) |

Что НЕ трогаем: `web/src/api/*`, `openapi/openapi.yaml`, `web/public/openapi.yaml`,
`web/src/pages/iam/hooks.ts` (переиспользуем), примитивы `ui/*`, страница «Роли»,
`Sidebar`.

## Risks / Trade-offs

- **Регрессия `DataTable` при добавлении фильтра** → решено НЕ менять примитив;
  фильтрация локальна на странице, поверх неизменного `rows`. Тесты Ролей/
  Пользователей на `DataTable` не затрагиваются.
- **Пустые подсказки при пустом каталоге** (`listPermissions` вернул `[]`) →
  `<datalist>` пуст, но `<input>` остаётся полностью функциональным (свободный
  ввод) — поведение деградирует мягко, валидация `zod` не меняется.
- **`<datalist>` разрешает любые значения** → это намеренно (создание новых
  `action`/`resource`); валидность по-прежнему проверяет периметр (400/409/422),
  ошибки — штатными тостами через `CATALOG_OVERRIDES`.
- **Сброс состояния модалки назначения** → единый сброс и при success, и при
  close; иначе повторное открытие показало бы прежний выбор. Покрыть тестом
  «закрытие сбрасывает форму».
- **Эквивалентность тест-покрытия** → существующие тесты `UsersPage` ожидают форму
  на экране сразу; их надо обновить под «открыть модалку → действовать», сохранив
  все проверки (debounce, канонический `sub`, 403/503, валидация, снятие,
  пагинация). Покрытие не сокращается.

## Migration Plan

Чисто фронтенд, без миграций данных и без отката провизии ресурсов. Инкрементально,
с зелёными `tsc --noEmit` + `vitest` на каждом шаге:

1. `PermissionsPage`: добавить `<datalist>`-подсказки к полям создания (вывод
   уникальных `action`/`resource` из каталога). Обновить/дополнить тесты.
2. `PermissionsPage`: добавить локальный тулбар фильтра (поиск + тип) над
   `DataTable`, динамический `emptyMessage`. Тесты на фильтр.
3. `UsersPage`: перенести форму назначения в `Dialog` за кнопкой в
   `PageHeader.actions`; единый сброс при success/close. Обновить тесты под модалку.
4. Прогон полного `vitest` + `tsc --noEmit`; сверка `gen:check` (контракт/кодоген
   не изменены).

Apply начинается с ветки `change/portal-iam-users-permissions-ux` от `master`
(прямые коммиты в `master` запрещены); PR с зелёным CI; затем `/opsx:archive`
отдельным PR sync+archive.
