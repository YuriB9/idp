## Context

Аудит `golang-code-audit` (отчёт `docs/audits/2026-06-golang-audit.md`, PR #78)
зафиксировал чистый baseline и 6 находок. ST1 уже исправлена в аудит-PR. Этот
change закрывает оставшиеся 5 (C1, P1, R1, D1, D2) одним проходом. Все правки —
внутренние (конкурентность/производительность клиента Keycloak) и
инфраструктурные (CI/Dependabot/Makefile); внешнее поведение не меняется.

Затронутый код сосредоточен в одном файле
`services/idm/internal/identity/keycloak.go` (C1/P1/R1). Модуль `services/idm`
уже зависит от `golang.org/x/sync` (используется `singleflight` в
`internal/usecase`), поэтому `errgroup` и `singleflight` доступны без новой
внешней зависимости графа (возможен лишь tidy в пределах модуля).

## Goals / Non-Goals

**Goals:**
- Сделать govulncheck-гейт воспроизводимым (D1).
- Сгруппировать Dependabot и ввести согласованный `make tidy-all` (D2).
- Убрать удержание мьютекса на время I/O при выдаче токена (C1), учтя caveat
  `singleflight` (R1).
- Сделать `Resolve` ограниченно-параллельным (P1).
- Сохранить fail-closed, внешние контракты и зелёный CI.

**Non-Goals:**
- Изменение внешнего поведения, контрактов, кодов ответов.
- Рефакторинг RBAC-пути `CheckAccess` (он не вызывает Keycloak — вне находок).
- Пересмотр ADR.

## Decisions

**C1 — токен вне критической секции.** Под `mu` держим только чтение/запись
полей `token`/`exp`. Сетевой запрос выполняется вне лока; для дедупликации
одновременных промахов — `singleflight.Group` (один in-flight запрос токена на N
конкурентных). Альтернатива (double-checked без singleflight) допустима, но
singleflight убирает thundering herd при массовом истечении — предпочтительнее,
паттерн уже знаком по `usecase`.

**R1 — устойчивость к отмене лидера.** Загрузку токена/резолв под singleflight
вести так, чтобы отмена контекста первого вызывающего не валила ведомых: при
необходимости — `context.WithoutCancel` для фоновой выдачи токена (таймаут
HTTP-клиента всё равно ограничивает время).

**P1 — ограниченно-параллельный Resolve.** `errgroup.Group` + `SetLimit(k)`
(небольшой k, напр. 8), запись результата в предвыделенный срез по индексу
субъекта (без гонок — каждая горутина пишет свой индекс). Семантика
«осиротевшего» (`Found=false`) и `ErrUnavailable` сохраняется (первая фатальная
ошибка отменяет группу).

**D1 — пин govulncheck.** Предпочтительно добавить govulncheck в пинованный
набор `tools` и запускать из `tools/bin` (единый паттерн с golangci-lint/goose);
запасной вариант — явный `@vX.Y.Z` в джобе. Версию выбрать = текущей стабильной
на момент apply.

**D2 — Dependabot groups + tidy-all.** Добавить `groups:` для gomod (общие
runtime-зависимости в одну группу), чтобы общие бампы шли одним PR. `make
tidy-all` прогоняет `go mod tidy` (`GOWORK=off`) по модулям в порядке
зависимостей (`pkg` → `projects` → `devinfra-worker` → `gateway`/`idm`/`e2e`/
`tools`), снимая кросс-модульную хрупкость (memory `gowork-cross-module-tidy`).

## Risks / Trade-offs

- **[C1/P1 — регрессия конкурентности]** → покрыть `go test -race` пакета
  identity; добавить/проверить тест на параллельный резолв и на отсутствие
  сериализации при выдаче токена.
- **[P1 — частичные ошибки]** → семантику `Found=false` vs `ErrUnavailable`
  зафиксировать тестом; errgroup отменяет группу на первой фатальной ошибке.
- **[D1 — пин устаревает]** → Dependabot/обновление tools подхватит новые версии;
  важна воспроизводимость, а не «вечная свежесть».
- **[D2 — кросс-модульный tidy]** → `make tidy-all` и прогон tidy по всем
  затронутым модулям в одном PR; не повторять PR #75.
- **[новая зависимость в idm]** → `errgroup` из уже присутствующего
  `golang.org/x/sync`; tidy согласовать (`GOWORK=off`).
