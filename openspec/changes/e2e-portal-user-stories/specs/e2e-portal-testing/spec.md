## ADDED Requirements

### Requirement: Сквозное покрытие 4 user stories через периметр

Набор E2E ДОЛЖЕН (MUST) покрывать все четыре user stories платформы через REST-периметр (ADR-0009), вызывая операции `createService`, `getService`/`listServices`, `setServiceOwners`, `decommissionService`, `transferService` ровно так, как их видит портал. Каждая user story ДОЛЖНА (MUST) иметь как минимум один happy-path-сценарий, доводящий соответствующий Temporal-workflow до терминального ожидаемого статуса.

#### Scenario: Создание сервиса доходит до active

- **GIVEN** поднят docker-compose-стенд с моками GitLab/Vault/Harbor
- **WHEN** тест вызывает `createService` для нового имени в проекте и поллит `getService`
- **THEN** запись фиксируется со статусом `creating`, затем переходит в `active` в пределах таймаут-бюджета, а Saga-activity (GitLab repo/CI-vars, Vault policy/AppRole, Harbor project/robot) завершаются успешно

#### Scenario: Изменение владельцев отражается в каталоге и ролях

- **GIVEN** существует активный сервис с известной `owners_version`
- **WHEN** тест вызывает `setServiceOwners` с полным желаемым набором владельцев и текущей `owners_version`
- **THEN** `getService` возвращает обновлённый набор владельцев и инкрементированную `owners_version`, а синхронизация ролей в IDM (ADR-0011) отражает diff add/remove

#### Scenario: Decommission переводит в decommissioned (soft delete)

- **GIVEN** существует активный сервис и нагрузка снята
- **WHEN** тест вызывает `decommissionService` с `load_drained=true`
- **THEN** статус переходит в `decommissioned`, запись каталога сохраняется (не purge), доступы во внешних системах отзываются

#### Scenario: Перенос сервиса меняет проект-владельца

- **GIVEN** существует активный сервис в исходном проекте
- **WHEN** субъект с правами `transfer` и `transfer_in` вызывает `transferService` в целевой проект
- **THEN** статус проходит `active→transferring→active`, проект записи меняется на целевой, владельцы переезжают вместе с записью (ADR-0013)

### Requirement: Аутентификация через реальный OIDC (Keycloak + Oauth2-Proxy)

E2E ДОЛЖЕН (MUST) аутентифицироваться по реальному OIDC-пути, а НЕ обходом периметра через `AUTH_DISABLED`. Токен ДОЛЖЕН (MUST) запрашиваться у Keycloak (password-grant клиента `idp-portal`, пользователи realm `dev`/`alice`/`bob`), а запросы периметра ДОЛЖНЫ (MUST) идти через `oauth2-proxy` (:4180) на gateway с включённой проверкой JWT.

#### Scenario: Запрос с валидным Bearer проходит периметр

- **WHEN** тест получает `access_token` у Keycloak и шлёт запрос через oauth2-proxy с заголовком `Authorization: Bearer <token>`
- **THEN** oauth2-proxy проксирует запрос на gateway, gateway валидирует JWT (issuer/audience/JWKS) и применяет RBAC по `sub` субъекта

#### Scenario: Локальная AUTH_DISABLED-сборка не ломается

- **WHEN** запускается базовый `docker compose up` без E2E-override
- **THEN** gateway работает с `AUTH_DISABLED=true`, портал ходит напрямую, существующий `conformance`-таргет остаётся рабочим

### Requirement: Детерминизм прогона и ожидание статусов без sleep

Ожидание переходов статусов ДОЛЖНО (MUST) выполняться ретраи-поллингом `getService` с конечным таймаут-бюджетом, а НЕ фиксированным `sleep`. Моки управляемых систем ДОЛЖНЫ (MUST) быть расширены так, чтобы все activity четырёх воркфлоу доходили до конца предсказуемо. Обнаружение терминального `failed` при ожидании `active` ДОЛЖНО (MUST) немедленно завершать сценарий с диагностикой, не выжидая весь бюджет.

#### Scenario: Поллинг до целевого статуса с бюджетом

- **WHEN** тест ожидает перехода `creating→active`
- **THEN** он периодически опрашивает `getService` до достижения статуса или исчерпания бюджета, и красный результат содержит читаемое диагностическое сообщение (последний статус, истёкшее время)

#### Scenario: Готовность стенда проверяется по health перед тестами

- **WHEN** запускается набор E2E
- **THEN** до первого сценария проверяется готовность периметра (gateway `/healthz`), доступность токен-эндпоинта Keycloak и здоровье Temporal, и тесты не стартуют, пока стенд не готов

### Requirement: Изоляция и очистка между сценариями

Сценарии ДОЛЖНЫ (MUST) использовать уникальные имена сервисов для исключения межтестовых гонок при `t.Parallel()`. Набор НЕ ДОЛЖЕН (MUST NOT) зависеть от purge-очистки каталога: проверки опираются на собственную запись сценария, а идемпотентность проверяется намеренными повторными вызовами.

#### Scenario: Параллельные истории не мешают друг другу

- **WHEN** несколько story-тестов выполняются с `t.Parallel()`
- **THEN** каждый оперирует уникальным именем сервиса (и, как следствие, детерминированным, но различным `WorkflowID`), и результат одного не зависит от другого

#### Scenario: Идемпотентность повторного create

- **GIVEN** запущено создание сервиса
- **WHEN** тест повторяет `createService` для того же `(project, name)` до завершения первого
- **THEN** периметр отвечает 409 (конфликт по детерминированному `WorkflowID`/занятому имени), а исходный workflow не дублируется

### Requirement: Покрытие ключевых ошибок и компенсаций

Набор ДОЛЖЕН (MUST) проверять ключевые ошибочные ветки контракта периметра и наблюдать политику отказов Saga/PONR без изменения контракта. Переходы статусов ДОЛЖНЫ (MUST) трактоваться как guarded-CAS с ожидаемым исходным статусом (ADR-0004).

#### Scenario: Конфликт версии владельцев

- **WHEN** `setServiceOwners` вызывается с устаревшей `owners_version`
- **THEN** периметр отвечает 409, а набор владельцев не меняется

#### Scenario: Decommission с невыполненным предусловием

- **WHEN** `decommissionService` вызывается с `load_drained=false` либо для сервиса не в статусе `active`
- **THEN** периметр отвечает 422 (предусловие не выполнено), статус не меняется (ADR-0012, чек-флаг при отсутствии K8s worker)

#### Scenario: Идемпотентный повтор decommission и transfer

- **GIVEN** сервис уже `decommissioned` (или уже перенесён)
- **WHEN** соответствующая операция вызывается повторно
- **THEN** периметр отвечает 200 без повторного запуска побочных эффектов

#### Scenario: Saga-откат создания при отказе внешней системы

- **GIVEN** мок настроен возвращать non-retryable ошибку на шаге Vault для маркированного сервиса
- **WHEN** запущено создание такого сервиса
- **THEN** выполняются компенсации (ADR-0005), статус переходит в `failed`, в логах присутствует alert, и нет молчаливого перехода в `active` без ресурсов
