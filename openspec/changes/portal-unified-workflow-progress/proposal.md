## Why

Сейчас UX асинхронных операций над сервисом рассогласован: создание показывает ход
работ панелью-строкой на странице сервиса (`ServiceProgressPage`), а смена владельцев,
вывод из эксплуатации и перенос выполняются мутацией с тостом в правом нижнем углу
(`OwnersCard`/`DecommissionCard`/`TransferCard`). Пользователь видит разные паттерны для
однотипных по сути операций (запуск Temporal-воркфлоу → ожидание терминального статуса),
не понимает, из каких шагов состоит операция и что произошло при ошибке (был ли откат).
Все три плеча DevInfra-воркфлоу уже реальные (ADR-0019/0020/0021), порядок активностей
зафиксирован в воркерах — пора показать его пользователю единым образом.

Изменение соответствует docs/IDP_MVP_План.md (этап «Портал/UX») и развивает ADR-0017
(дизайн-система портала). Опирается на ADR-0001 (Temporal как источник шагов), ADR-0004
(guarded-CAS статусов), ADR-0005 (Saga-откат при ошибке), ADR-0008 (split воркфлоу),
ADR-0011/0012/0013 (owners/decommission/transfer), ADR-0018 (E2E-харнесс).

## What Changes

- Вводится переиспользуемый UI-примитив **ступенчатого прогресса** (stepper) в
  дизайн-системе портала (ADR-0017): шаги с состояниями `pending|running|done|failed`,
  доступность (`role`/`aria-live`), motion-safe-анимации, единый стиль с остальными
  примитивами.
- Вводится **хук поллинга** статуса сервиса (инкапсулирует логику нынешней
  `ServiceProgressPage`: react-query с `refetchInterval` до терминального статуса, без
  `sleep`), переиспользуемый всеми операциями.
- Вводится **клиентская модель шагов на операцию** (создание / смена владельцев /
  decommission / transfer): фиксированный порядок шагов = порядок активностей
  соответствующего Temporal-воркфлоу; текущая фаза выводится из грубого статуса сервиса
  и доменных полей (`decommissioned_at`, `owners_version`). Контракт периметра НЕ
  меняется (выбран вариант B; обоснование и отвергнутый вариант A — в design.md/adr.md).
- **ВСЕ** операции приводятся к единому ступенчатому прогрессу:
  - создание — однострочная панель заменяется степпером;
  - смена владельцев / decommission / transfer — «тост-при-успехе» заменяется тем же
    степпером, отражающим ход асинхронной операции.
- **Судьба тостов**: тост остаётся ТОЛЬКО для мгновенных синхронных ошибок (валидация,
  4xx прав/конфликта/предусловия при запуске операции — через `lib/errors`). Ход и исход
  асинхронной операции (running/done/failed + факт отката Saga) показывает степпер, а не тост.
- Особые случаи отражаются в UI: смена владельцев — короткий воркфлоу (одна успешная смена
  на сервис) → вырожденный степпер; decommission/transfer — пометка точки невозврата;
  `failed` — индикация неуспеха операции и факта компенсирующего отката (Saga, ADR-0005).
- Все подписи шагов — на русском; обновляются web-тесты затронутых страниц/карточек и
  добавляются тесты нового компонента и хука.

### Modified Capabilities
- `portal-ui`: единый ступенчатый прогресс асинхронных операций над сервисом. Меняются
  требования «Экран прогресса с поллингом статуса», «Отображение и изменение владельцев»,
  «Действие „Вывести из эксплуатации“», «Действие „Перенести сервис“» и связанные
  требования обработки ошибок/индикации статуса (тост → степпер для асинхронного хода);
  добавляется требование переиспользуемого примитива ступенчатого прогресса и хука
  поллинга в дизайн-системе.

## Impact

- **Код (web/src)**: новый `components/ui/stepper.tsx` (примитив) и `hooks/`
  (хук поллинга статуса) + клиентская модель шагов; рефактор `pages/ServiceProgressPage.tsx`,
  `pages/CreateServicePage.tsx`, `components/{Owners,Decommission,Transfer}Card.tsx` на
  единый прогресс. Обновление/добавление тестов `web/src/**/*.test.tsx`.
- **Контракт периметра**: НЕ меняется (вариант B). `openapi/openapi.yaml`, `web/src/api/*`,
  `web/public/openapi.yaml` не трогаются руками; `gen:check` остаётся зелёным без
  перегенерации.
- **Бэкенд/воркеры/Temporal**: НЕ меняются. Порядок активностей
  (`services/projects/{provisioning,changeowners,decommission,transfer}`) используется как
  справочник подписей шагов, без изменения семантики.
- **Границы gRPC/Temporal**: не затрагиваются; провизия ресурсов не меняется, поэтому
  отдельный план отката на уровне инфраструктуры не требуется (Saga-откат самих воркфлоу —
  без изменений, лишь отражается в UI).
- **E2E (tests/e2e)**: при необходимости — ассерты единообразного отображения прогресса
  поверх существующего харнесса (поллинг без `sleep`, ADR-0018); контрактных/бэкенд-тестов
  не добавляется (контракт не меняется).
