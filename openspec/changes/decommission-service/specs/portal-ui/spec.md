# portal-ui Specification (delta)

## ADDED Requirements

### Requirement: Действие «Вывести из эксплуатации» с подтверждением

Портал ДОЛЖЕН (MUST) предоставлять действие «Вывести из эксплуатации» на странице
сервиса (по образцу карточки владельцев/формы создания: react-hook-form + zod,
TanStack Query). Действие ДОЛЖНО (MUST) требовать явного подтверждения с вводом
имени сервиса и явно сообщать, что это decommission (soft-delete, необратимость для
доступа), а не purge. Действие ДОЛЖНО (MUST) отправлять явное предусловие снятой
нагрузки (`load_drained`). Кнопка/действие ДОЛЖНЫ (MUST) быть недоступны или
заблокированы для сервиса, который уже `decommissioned` или находится в
недопустимом статусе. Ответы API ДОЛЖНЫ (MUST) валидироваться через zod (`.parse`).

#### Scenario: Подтверждение и успешный вывод

- **GIVEN** активный сервис на странице портала
- **WHEN** пользователь выбирает «Вывести из эксплуатации», вводит имя сервиса для
  подтверждения и подтверждает (со снятой нагрузкой)
- **THEN** отправляется `POST .../decommission`, по успеху статус в UI отражается
  как `decommissioned`

#### Scenario: Блокировка повторного/недопустимого действия

- **GIVEN** сервис уже в статусе `decommissioned` (или `creating`/`failed`)
- **WHEN** отображается страница сервиса
- **THEN** действие «Вывести из эксплуатации» недоступно/заблокировано

### Requirement: Обработка ошибок и индикация статуса в портале

Портал ДОЛЖЕН (MUST) корректно обрабатывать ответы операции вывода из эксплуатации:
`403` (нет права) — сообщение об отказе доступа; `409` (конкурентный конфликт) —
сообщение о конфликте с предложением перечитать состояние; `422` (предусловие не
выполнено, нагрузка не снята) — понятное сообщение о невыполненном предусловии.
Статус `decommissioned` ДОЛЖЕН (MUST) индицироваться бейджем статуса в списке и на
странице сервиса; `decommissioned_at` ДОЛЖЕН (MUST) отображаться для выведенных
сервисов.

#### Scenario: Отказ доступа (403)

- **GIVEN** у пользователя нет права `decommission`
- **WHEN** он пытается вывести сервис из эксплуатации
- **THEN** UI показывает сообщение об отказе доступа, статус не меняется

#### Scenario: Предусловие не выполнено (422)

- **GIVEN** нагрузка не снята
- **WHEN** API возвращает `422`
- **THEN** UI показывает понятное сообщение о невыполненном предусловии (нагрузка
  не снята)

#### Scenario: Конфликт (409)

- **GIVEN** статус сервиса изменился конкурентно
- **WHEN** API возвращает `409`
- **THEN** UI показывает сообщение о конфликте и предлагает перечитать состояние
