## Context

DevInfra worker исполняет четыре воркфлоу (создание/смена владельцев/decommission/transfer)
через activities, ходящие к GitLab/Vault/Harbor по HTTP за узкими интерфейсами
(`internal/integrations`). Все три реализованы как WireMock-моки (`deploy/mocks/mappings/*`)
на приватных адресах с `SSRF_DISABLED=true`. HTTP-клиент GitLab писался под мок, поэтому
расходится с реальным GitLab API (инвентаризация `http.go`):

- `call()` (`http.go:66`) ставит только `Content-Type` — **нет аутентификации**.
- `CreateRepo` (`http.go:111`) шлёт `{"name","namespace":<строка>}` — реальный
  `POST /api/v4/projects` ждёт `namespace_id` (число) или путь группы.
- `SyncMembers` (`http.go:146`) шлёт `{"username","access_level":"40"}` — реальный
  `POST /projects/:id/members` ждёт `user_id` (число).
- Идемпотентные коды (409 при create, 404 при delete/transfer) угаданы под мок.
- URL-кодирование пути `project%2Fname` (archive/unarchive/transfer/variables) под реальный
  GitLab не проверялось.

Каталог создаёт услугу со статусом `creating`, WorkflowID детерминирован
(см. `e2e-change-owners-single-shot`). Имена сервисов в тестах уникальны (`uniqueName`,
`harness_test.go:225`), что снимает коллизии повторного создания репозитория.

Стенд ADR-0018 уже даёт образец тяжёлого override-профиля (`docker-compose.e2e.yml`),
харнесс `tests/e2e` (получение токена, polling `waitForStatus` без sleep, изоляция
уникальными именами, `requireStack`/`TestMain`) и Makefile-цели `e2e-up/test/down`.

Ограничения: не менять контракт периметра/proto/кодоген (gen:check зелёный); комментарии
только на русском; SSRF-guard в проде включён; токен не хардкодить сверх фикстур; GitLab
не в CI и не в дефолтной локалке (тяжёлый образ); инкрементально, каждый шаг зелёный.

## Goals / Non-Goals

**Goals:**
- Довести GitLab-клиент до совместимости с реальным GitLab CE API (auth, `namespace_id`/
  `user_id` resolution, коды идемпотентности), не ломая in-memory и стаб-против-моков.
- Поднять реальный GitLab CE в отдельном compose-override с детерминированным сидом и
  health-gate, не затрагивая локалку/e2e-override/conformance.
- Доказать интеграционными тестами (тег `integration`, ассерт через GitLab API), что
  воркфлоу реально создают/переносят/архивируют репозитории и компенсируют сбои.
- Дать Makefile-цели локального ручного прогона по образцу `e2e-*`.

**Non-Goals:**
- Реальные Vault и Harbor (остаются моками — граница MVP).
- Любые изменения контракта периметра, proto, web/src/api, сгенерированного кода.
- GitLab в CI; CI-runner и пайплайны в GitLab; Kubernetes; прод-GitLab/SaaS; SSE.

## Decisions

### Решение 1 — Аутентификация: `PRIVATE-TOKEN` (root PAT через env)
Worker→GitLab аутентифицируется заголовком `PRIVATE-TOKEN: <token>` на КАЖДОМ запросе к
GitLab. Токен берётся из нового env `GITLAB_TOKEN` (`integrations.Config.GitLabToken`),
читается в `main.go`; в compose-override прокидывается тем же детерминированным значением,
что сид GitLab кладёт как root PAT.
- **Почему PRIVATE-TOKEN, а не Bearer/OAuth:** оба эквивалентны для GitLab REST, но
  `PRIVATE-TOKEN` со статическим PAT проще сидировать детерминированно (без OAuth-обмена) и
  не требует refresh; OAuth/PAT-Bearer избыточен для одного сервис-актора.
- **Сидирование токена:** root PAT задаётся при инициализации GitLab детерминированной
  строкой через переменную окружения образа (`GITLAB_OMNIBUS_CONFIG` →
  `gitlab_rails['initial_root_password']` + предсоздание PAT через `rails runner` в
  init-команде/seed-контейнере). Значение — тест-фикстура (не секрет прода), но в код не
  хардкодится сверх фикстур стенда: и сид, и `GITLAB_TOKEN` worker берут одно env.
- **Реализация в `call()`:** добавить в `doer`/`gitLabHTTP` хранение токена и установку
  заголовка только для GitLab-запросов (Vault/Harbor не трогаем). Заголовок не логируется.
- Альтернатива (отклонена): OAuth password-grant к GitLab — лишний обмен и нестабильность.

### Решение 2 — Маппинг `namespace`→`group_id`: предсозданные группы + lookup с кэшем
Каталог использует `project` как namespace (`demo`/`demo2`). Сид предсоздаёт в GitLab группы
с путями `demo`/`demo2`. Клиент при `CreateRepo`/`TransferRepo` резолвит путь группы в
`group_id` через `GET /api/v4/groups/:path` (URL-encoded) и шлёт `namespace_id`. Результат
кэшируется в клиенте (`map[string]int` под мьютексом) — группы стабильны в пределах прогона.
- **Почему предсозданные группы, а не автосоздание:** namespace = проект каталога; проекты
  сидируются отдельно (IDM 0003-0005), GitLab лишь отражает их; автосоздание размывало бы
  границу и требовало прав на создание групп в воркфлоу.
- Альтернатива (отклонена): хранить `group_id` в конфиге env-картой — хрупко при добавлении
  проектов; lookup по пути самодостаточен.

### Решение 3 — Маппинг владелец→`user_id`: lookup по предсозданным тест-пользователям
Владельцы (`subjAlice`/`subjBob`) — UUID Keycloak, в GitLab их нет. Для `SyncMembers` сид
создаёт тест-пользователей GitLab, а маппинг UUID→GitLab-логин задаётся фикстурой стенда
(env-карта или конвенция). Клиент резолвит логин в `user_id` через
`GET /api/v4/users?username=<login>` и шлёт `user_id` + числовой `access_level` (40).
- **Что ассертить для SyncMembers:** членство пользователя в проекте через
  `GET /projects/:id/members/all/:user_id` (200 = член). Если маппинг владельца не задан —
  пропускаем (warn), членство не обязательно для happy-path create/transfer.
- **Почему не один root для всех:** членство root неинформативно (owner группы и так имеет
  доступ); отдельные пользователи позволяют ассертить реальную синхронизацию состава.
- Альтернатива (отклонена): заводить в GitLab пользователей с UUID-логинами — GitLab
  ограничивает формат username; явная фикстура-маппинг проще и читаема.

### Решение 4 — Идемпотентность под реальные коды GitLab
Пересмотр `okExtra` под реальный GitLab:
- `CreateRepo`: повторное создание → `400` с сообщением «has already been taken» (GitLab НЕ
  отдаёт 409). Трактуем как успех по признаку «уже существует» — либо предварительный
  `GET` проекта по пути, либо разбор тела 400. Выбираем **GET-then-create** (проверить
  существование по `GET /projects/:path%2Fname`, при 200 — успех-no-op), как детерминированный
  и не зависящий от текста ошибки.
- `DeleteRepo`: `404` (уже удалён) и `202` (принято к удалению) — успех.
- `Archive`/`Unarchive`: `201`/`200` норма; повторно — идемпотентно (GitLab возвращает
  текущее состояние 201/200), `404` при отсутствии — на стенде успех.
- `TransferRepo`: `200` норма; повторный transfer в ту же группу → `400` — трактуем через
  предварительную проверку текущего namespace (GET) как no-op-успех.
- `InjectVariables`: GitLab отдаёт `400` «has already been taken» при существующей переменной
  → перейти на upsert: `PUT /projects/:id/variables/:key` (создаёт/перезаписывает) вместо
  POST+409. Значения не логируем.
- **Почему GET-then-act вместо разбора текста ошибки:** тексты GitLab не контракт и зависят
  от версии; коды надёжнее, а GET даёт детерминированный признак идемпотентности.

### Решение 5 — Выбор реализации клиента в рантайме
В `main.go` выбор по env: если задан `GITLAB_TOKEN` (профиль реального GitLab) — собирается
GitLab-клиент с auth+resolution; иначе текущий стаб-против-моков. Vault/Harbor всегда моки.
In-memory (`memory.go`) остаётся для дефолтного прогона (юнит/temporal-testsuite) без сети.
- **Почему по наличию токена, а не отдельный флаг:** реальный GitLab без токена не
  заработает, токен — естественный переключатель; меньше env-поверхности.
- SSRF: для стендового GitLab на приватном адресе по http — `SSRF_DISABLED=true` в
  override (как у моков); в проде guard включён (https/публичные адреса).

### Решение 6 — Где тесты: ОТДЕЛЬНЫЙ интеграционный набор воркфлоу-уровня в `tests/e2e`
Добавляем новый файл в пакет `tests/e2e` (тег `integration`), переиспуя харнесс (токен,
`callAPI`, `waitForStatus`, `uniqueName`, `requireStack`), но с НОВЫМ gate-env
(`GITLAB_API_URL`+`GITLAB_TOKEN`): тест гоняет воркфлоу через периметр И дополнительно
ассертит фактическое состояние в GitLab через его API.
- **Почему отдельный набор, а не расширение существующих историй:** существующие
  `stories_test.go` гоняются против стенда с моками (`make e2e`); добавлять в них GitLab-
  ассерты сломало бы мок-прогон. Новый файл активируется только при заданном `GITLAB_API_URL`
  (свой `requireGitLab`), не мешая `make e2e`. Переиспользование харнесса исключает
  дублирование (цель 4).
- **Почему в пакете `tests/e2e`, а не в `devinfra-worker`:** ассерт «воркфлоу реально
  создало репо» сквозной (периметр→Temporal→GitLab) — это уровень e2e, харнесс уже там.

### Решение 7 — Компенсации/PONR против реального GitLab
- Создание (ADR-0005): при инъецированном сбое позднего шага наблюдаем компенсацию
  `DeleteRepo` — после `failed` репозитория в группе нет (ассерт GitLab API 404).
- Перенос (ADR-0013): `TransferRepo` — точка невозврата; ассертим, что после успеха репо в
  целевой группе и отсутствует в исходной; частичную необратимость не «откатываем».

## Risks / Trade-offs

- [GitLab CE стартует 3-5 мин, образ ~2.5GB] → отдельный override-профиль, НЕ в дефолтной
  локалке/CI; health-gate в compose (`/-/readiness` или `/-/health`) + `depends_on:
  condition: service_healthy`; харнесс ждёт готовности через расширенный бюджет
  (`GITLAB_STATUS_TIMEOUT`), polling без sleep.
- [Текст ошибок GitLab меняется между версиями] → идемпотентность строим на кодах и
  GET-then-act, не на разборе сообщений; образ GitLab пинуется по тегу версии.
- [Детерминизм сида root PAT] → PAT создаётся фиксированной строкой через `rails runner` в
  init-шаге; и сид, и worker берут токен из одного env стенда.
- [Повторное создание репо при детерминированном WorkflowID] → тесты используют уникальные
  имена (`uniqueName`); GET-then-create делает `CreateRepo` идемпотентным и для ретраев.
- [Утечка токена/сырых ошибок GitLab] → `PRIVATE-TOKEN` не логируется; клиент маппит ошибки
  в канонические (`httpclient.MapStatus`), сырой текст GitLab не уходит как контракт.
- [Дрейф мок-стаба и реального клиента] → общий `doer`/интерфейс; стаб-против-моков и
  реальный путь делят код `call()`, расходятся только в auth/resolution/кодах.

## Migration Plan

1. Ветка `change/devinfra-real-gitlab` от master.
2. Клиент: добавить `GitLabToken` в `Config`, auth в `call()`/`gitLabHTTP`, resolution
   (group/user) с кэшем, GET-then-act идемпотентность, upsert переменных — юнит-тесты на
   формирование запросов (стаб-сервер) зелёные; дефолтный прогон не меняется.
2. `main.go`: чтение `GITLAB_TOKEN` и выбор реализации; без токена поведение прежнее.
3. compose-override `docker-compose.gitlab.yml` + сид (группы/PAT/пользователи) + health-gate.
4. Makefile-цели `gitlab-up/gitlab-test/gitlab-down/gitlab` по образцу `e2e-*`.
5. Интеграционный тест-файл в `tests/e2e` (тег `integration`, `requireGitLab`, GitLab-ассерты).
6. PR с зелёным CI (gen:check без изменений — контракт не тронут); merge; затем
   `/opsx:archive` отдельным PR sync+archive (по образцу #55/#57).

**Rollback:** override-профиль и `GITLAB_TOKEN` не задействованы в дефолтной локалке/CI —
откат сводится к невключению профиля; код-путь реального клиента не активируется без токена.

- Нужен ли отдельный ADR: да — «GitLab auth-модель и маппинг namespace/owner→GitLab»
  (Решения 1-3) значимы; оформляется артефактом `adr.md` этого change.
