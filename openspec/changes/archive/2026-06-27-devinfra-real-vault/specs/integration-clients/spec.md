## ADDED Requirements

### Requirement: Исходящая аутентификация запросов к реальному Vault

Все исходящие HTTP-запросы Vault-клиента к реальному Vault ДОЛЖНЫ (MUST) нести аутентификацию
заголовком `X-Vault-Token` со значением из конфигурации (`integrations.Config.VaultToken`, env
`VAULT_TOKEN` или `VAULT_TOKEN_FILE`). Токен НЕ ДОЛЖЕН (MUST NOT) логироваться. Запросы к GitLab и
Harbor этой аутентификацией НЕ затрагиваются (заголовок `X-Vault-Token` не протекает на другие
интеграции).

#### Scenario: Vault-запрос несёт токен

- **GIVEN** сконфигурированный `VAULT_TOKEN`
- **WHEN** Vault-клиент выполняет любой запрос к Vault API
- **THEN** запрос содержит заголовок `X-Vault-Token` с этим значением, и значение токена не попадает в
  логи

#### Scenario: Токен не применяется к другим интеграциям

- **WHEN** клиент GitLab или Harbor выполняет запрос
- **THEN** заголовок `X-Vault-Token` Vault не добавляется к этим запросам

### Requirement: Семантика реального Vault API в Vault-клиенте

Vault-клиент в реальном режиме ДОЛЖЕН (MUST) использовать семантику реального HashiCorp Vault API:
секреты по KV v2 (`/v1/secret/data/...` с обёрткой `{"data":{...}}` на запись и разбором `data.data`
на чтение); AppRole-роли и per-service ACL-политики (`/v1/sys/policies/acl/...`,
`/v1/auth/approle/role/...`); identity-entity для владельцев (`/v1/identity/entity/name/...`).
Идемпотентность ДОЛЖНА (MUST) опираться на коды ответов Vault и при необходимости на предварительную
проверку существования (GET-then-act), а НЕ на разбор текста ошибок: запись (политика/роль/секрет/
entity) трактуется как успех по 2xx (`200`/`204`), чтение/удаление отсутствующего (`404`) — как успех
(no-op). Клиент НЕ ДОЛЖЕН (MUST NOT) светить сырые тексты ошибок Vault как контракт.

#### Scenario: AppRole и политика создаются идемпотентно

- **GIVEN** реальный Vault с включённым AppRole-движком
- **WHEN** воркфлоу провизии вызывает создание AppRole и политики повторно
- **THEN** повторная запись трактуется как успех (upsert/2xx) без дубликата и без зависимости от текста
  ошибки, а `role-id` читается из `data.role_id`

#### Scenario: Миграция секретов по KV v2

- **GIVEN** секреты по исходному пути `secret/data/<src>`
- **WHEN** клиент мигрирует пути source→target
- **THEN** чтение разбирает `data.data`, запись шлёт `{"data":{...}}` по целевому пути, создаётся
  целевая политика, исходный путь очищается (`404` при очистке — успех)

### Requirement: Семантика немедленного отзыва доступа Vault

Операция немедленного отзыва доступа Vault (`RevokeSecretID`, ADR-0012) ДОЛЖНА (MUST) отзывать доступ через перечисление и уничтожение secret-id, а НЕ через несуществующий «destroy all» одним вызовом.
Клиент ДОЛЖЕН (MUST) перечислить активные `secret-id`-accessors роли (`LIST`/`GET ?list=true`) и
уничтожить каждый через `secret-id-accessor/destroy`. Операция ДОЛЖНА (MUST) быть идемпотентна: отсутствие активных
secret-id (пустой список / `404`) трактуется как успех (no-op). Результат отзыва ДОЛЖЕН (MUST) быть
наблюдаем через Vault API (после отзыва набор активных accessors пуст).

#### Scenario: Отзыв уничтожает активные secret-id

- **GIVEN** AppRole-роль с выданными активными secret-id
- **WHEN** вызывается немедленный отзыв доступа
- **THEN** клиент перечисляет secret-id-accessors и уничтожает каждый, после чего набор активных
  accessors роли пуст

#### Scenario: Повторный отзыв идемпотентен

- **GIVEN** роль без активных secret-id (уже отозваны)
- **WHEN** отзыв вызывается повторно
- **THEN** операция завершается успешно как no-op (`404`/пустой список — не ошибка)

### Requirement: Выбор реализации Vault-клиента по конфигурации

`devinfra-worker` ДОЛЖЕН (MUST) выбирать реализацию Vault-клиента по конфигурации: при заданном
`VAULT_TOKEN` (или `VAULT_TOKEN_FILE`) собирается клиент против реального Vault (с аутентификацией
`X-Vault-Token` и семантикой реального API); иначе сохраняется поведение по умолчанию (клиент против
WireMock-мока). In-memory стаб ДОЛЖЕН (MUST) оставаться доступным для дефолтного прогона тестов без
сети. GitLab и Harbor сохраняют свой выбор реализации независимо от выбора Vault-клиента.

#### Scenario: Без токена — поведение по умолчанию

- **WHEN** `VAULT_TOKEN`/`VAULT_TOKEN_FILE` не заданы
- **THEN** `devinfra-worker` использует Vault-клиент против мока, как прежде, без обращения к реальному
  Vault

#### Scenario: С токеном — реальный Vault

- **WHEN** `VAULT_TOKEN` задан (профиль реального Vault)
- **THEN** `devinfra-worker` использует Vault-клиент с аутентификацией `X-Vault-Token` и семантикой
  реального API против реального Vault
