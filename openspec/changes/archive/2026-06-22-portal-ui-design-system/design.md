## Context

Портал `./web` — SPA (React 19 + Vite + TS + TanStack Query + react-router; формы
react-hook-form + zod; рантайм-валидация ответов через zod `.parse`). Текущее
состояние (инвентаризация `web/src`):

- **Примитивы** `web/src/components/ui/*`: `button.tsx`, `badge.tsx`, `card.tsx`,
  `input.tsx` — shadcn-подход на `cva` + Tailwind-токены. Иконки — `lucide-react`.
- **Токены/тема**: `web/src/index.css` — CSS-переменные (oklch, zinc/Geist),
  Tailwind `@theme inline`, `:root`/`.dark`; `web/src/lib/theme.tsx` —
  `ThemeProvider`/`ThemeToggle`, ключ localStorage `idp-theme`, дефолт — тёмная.
- **Каркас**: `web/src/layouts/GlobalLayout.tsx` — статичный sidebar (`<aside>`)
  фиксированной ширины + header + `Outlet`; навигация плоская (`NavLink`).
- **Страницы** `web/src/pages/*`: `ServiceListPage` (данные карточками/списком,
  состояния loading/empty/error инлайново), `CreateServicePage` (форма на странице),
  `ServiceProgressPage` (поллинг + карточки действий), `IamPage` (роли/права/
  субъекты списками, инлайновые формы, пикер пользователя из справочника,
  обработка 403/503/«осиротевших»).
- **Действия** `web/src/components/*`: `DecommissionCard`, `TransferCard`,
  `OwnersCard` — инлайновые карточки с формами и подтверждением вводом имени;
  `StatusBadge` — маппинг `ServiceStatus` (цвет+иконка+текст уже частично есть).
- **API** `web/src/api/*`: `client.ts` (zodios, кодоген из OpenAPI), `schema.ts`,
  `index.ts`. Пагинация контракта — keyset/opaque-курсоры
  (`next_page_token`/`next_cursor`, ADR-0009); коды 400/403/404/409/422/503
  стабилизированы.

Ограничения (обязательны): не менять `openapi/openapi.yaml`, `web/src/api/*`,
`web/public/openapi.yaml` (`gen:check` без diff); не понижать zod 3.25.76; формы —
react-hook-form + zod; ответы — zod `.parse`; не ломать тему и маршруты; сырые
ошибки наружу не показывать; все комментарии в коде — на русском; `tsc --noEmit` и
vitest зелёные на каждом шаге.

## Goals / Non-Goals

**Goals:**

- Единый слой дизайн-токенов и семантика статусов/критичности (цвет+иконка+текст).
- Переиспользуемые a11y-примитивы: DataTable (курсорная пагинация ADR-0009,
  состояния, сортировка, плотность), collapsible Sidebar, Dialog/ConfirmDialog,
  Toast с единым маппингом кодов периметра.
- Инкрементальный рефактор страниц на примитивы с сохранением поведения и прав
  (включая IAM-пикер и обработку 403/503/«осиротевших»).
- WCAG AA в обеих темах, управление фокусом, клавиатура, ARIA,
  `prefers-reduced-motion`.

**Non-Goals:**

- Любые изменения бэкенда: proto, `services/*`, `openapi/openapi.yaml`, коды
  ответов, RBAC, миграции, кодоген `web/src/api/*`.
- Новые бизнес-фичи/разделы/данные, которых нет в контракте.
- Смена фреймворка/роутера/стейт-менеджера, SSR, i18n (тексты — на русском),
  редизайн входа/oauth2-proxy.

## Decisions

### Решение 1: Расширять существующие shadcn-примитивы, а не вводить параллельный слой

Дизайн-система реализуется КАК расширение текущего `web/src/components/ui/*`
(тот же `cva` + Tailwind-токены), а не отдельным параллельным слоем компонентов.
Новые примитивы (`data-table`, `sidebar`, `dialog`, `confirm-dialog`, `toast`,
`page-header`, семантика статусов/критичности) добавляются в `components/ui` и
рядом, переиспользуя `Button`/`Input`/`Card`/`Badge`. Документация токенов и правил
— в README/комментариях слоя.

- **Почему**: минимум дублей, единый визуальный язык, существующие страницы уже
  завязаны на `components/ui`; параллельный слой создал бы две системы и риск
  рассинхронизации.
- **Альтернатива (отклонена)**: отдельный `design-system/` слой — лишняя миграция и
  дубли примитивов без выгоды для MVP-объёма.

### Решение 2: Источник токенов — существующие CSS-переменные + Tailwind `@theme`

Токены остаются на CSS-переменных (`:root`/`.dark` в `index.css`) и пробрасываются
в Tailwind через `@theme inline`. Расширяем палитру состояний/критичности
(семантические токены поверх базовых), не ломая контракт `ThemeProvider`.
Семантические уровни критичности — фиксированный набор токенов (например,
`--criticality-{low,medium,high,critical}` + foreground), используемых только через
компонент бейджа.

- **Почему**: тема уже работает на этих переменных; добавление семантических токенов
  не трогает механику переключения и FOUC-защиту.
- **Альтернатива (отклонена)**: JS-объект токенов/CSS-in-JS — дублировал бы
  источник правды темы и усложнил dark/light без выгоды.

### Решение 3: Семантика статусов/критичности — единый компонент «цвет+иконка+текст»

Маппинг `ServiceStatus` и уровней критичности — единый источник правды (расширение
`StatusBadge` + новый бейдж критичности), кодирующий значение тремя признаками
одновременно (colorblind-safe). Неизвестный статус деградирует в нейтральный бейдж с
исходной строкой (поведение уже заложено в `StatusBadge`).

- **Почему**: разрозненная семантика — корневая проблема; единый компонент
  гарантирует консистентность и a11y (не только цвет).

### Решение 4: DataTable — собственная лёгкая реализация (без TanStack Table)

DataTable реализуется СВОИМ компонентом на семантической разметке (`<table>` с
`role`/заголовками/`aria-sort`), а не через TanStack Table.

- **Почему**: объём данных в MVP небольшой, нужна курсорная (не offset) пагинация
  ADR-0009 и полный контроль a11y/состояний; собственная реализация легче по бандлу
  и проще тестируется vitest. Клиентская сортировка достаточна там, где набор
  загружен целиком.
- **Альтернатива (отклонена)**: TanStack Table — вес и модель пагинации заточены под
  offset/самостоятельную модель; избыточно для текущего объёма.
- **Курсоры**: компонент принимает «загрузить ещё» по непрозрачному курсору
  (`next_page_token` для `/iam/subjects`, `/projects/.../services`; `next_cursor`
  для `/iam/directory/*`); пустой курсор → конец, управление скрыто. Интеграция с
  TanStack `useInfiniteQuery` (как уже сделано для субъектов в `IamPage`).

### Решение 5: Модалки/тосты — собственные лёгкие примитивы, без тяжёлых UI-китов

Dialog/ConfirmDialog и Toast реализуются СВОИМИ примитивами (focus-trap, ESC,
возврат фокуса, блокировка скролла, ARIA) без добавления тяжёлых зависимостей;
допустимо использовать нативный `<dialog>` либо контролируемый портал. Если для
надёжного focus-trap потребуется лёгкая утилита — выбирать минимальную, не понижая
zod и не таща UI-кит целиком.

- **Почему**: ограничение «не тянуть тяжёлые UI-киты без нужды»; полный контроль
  a11y; тестируемость фокуса/ESC в vitest/jsdom.
- **Альтернатива (отклонена)**: Radix/MUI целиком — избыточный вес ради нескольких
  примитивов.

### Решение 6: Единый маппинг кодов периметра в UI-сообщения

Вводится единая утилита маппинга `HTTP-код → сообщение` (400/403/404/409/422/503),
переиспользуемая тостами и блоками ошибок. Существующие `httpStatusOf` и локальные
`mutationMessage`/`catalogMessage` из `IamPage`/карточек консолидируются в общий
модуль (`web/src/lib/*`). Сырые внутренние ошибки наружу не идут; zod `.parse`
сохраняется как страж дрейфа контракта.

- **Почему**: сейчас маппинг разрознён по страницам; единый модуль убирает дубли и
  гарантирует консистентные сообщения.

### Решение 7: Состояние навигации/плотности в localStorage

Ключи localStorage согласуются со схемой темы (`idp-theme`): `idp-sidebar`
(свёрнуто/развёрнуто) и `idp-density` (comfortable/compact) — стабильные строковые
значения с безопасным чтением (try/catch, деградация к дефолту при недоступности).
SSR нет (SPA), но защита от рассинхронизации — валидация прочитанного значения по
белому списку.

- **Почему**: единообразие с темой; устойчивость к приватному режиму.

### Решение 8: Инкрементальная стратегия рефактора (страница за страницей)

Порядок: сначала токены/семантика + примитивы (DataTable, Sidebar, Dialog,
ConfirmDialog, Toast, PageHeader) с unit-тестами; затем страницы по одной
(`ServiceListPage` → `ServiceProgressPage`+карточки → `CreateServicePage` →
`IamPage`), сохраняя зелёные `tsc --noEmit` и vitest на каждом шаге. Каждая страница
переключается на примитивы атомарно, чтобы не оставлять полусломанных состояний.

**Матрица «страница → новые примитивы»:**

| Страница / компонент | Примитивы |
| --- | --- |
| `GlobalLayout` | Sidebar (collapsible, icon-only, off-canvas), PageHeader, ThemeToggle |
| `ServiceListPage` | DataTable (курсор `next_page_token`, состояния, сортировка), StatusBadge/критичность, кнопка «Создать» → модалка/маршрут |
| `ServiceProgressPage` | PageHeader, StatusBadge, действия → Dialog/ConfirmDialog, Toast |
| `DecommissionCard`/`TransferCard`/`OwnersCard` | ConfirmDialog (ввод имени, предусловия), Toast (маппинг 403/409/422) |
| `CreateServicePage` | Dialog/форма (react-hook-form + zod), Toast (409) |
| `IamPage` | DataTable (роли/права/субъекты, курсор `next_page_token`), Dialog/ConfirmDialog (create/delete/attach/detach), Toast (маппинг 400/403/404/409/422/503), пикер (debounce, `next_cursor`), сохранение 403/503/«осиротевших» |

**Что НЕ трогаем**: `openapi/openapi.yaml`, `web/src/api/*`,
`web/public/openapi.yaml` (кодоген; `gen:check` без diff), маршруты и права.

### Решение 9: ADR-0017

Вводится ADR-0017 «Дизайн-система и UI-архитектура портала», фиксирующий: слой
токенов на CSS-переменных + Tailwind; собственные DataTable/Dialog/Toast вместо
тяжёлых зависимостей; курсорную пагинацию таблицы по ADR-0009; единый маппинг кодов
периметра; ключи localStorage. См. `adr.md`.

## Risks / Trade-offs

- **Регрессия темы/FOUC при расширении токенов** → менять только дополняя
  переменные, сохранить синхронное применение `.dark` и дефолт; покрыть
  переключение темы тестом.
- **Потеря поведения IAM (пикер, 403/503/«осиротевшие») при переносе на примитивы**
  → переносить с сохранением логики `IamPage`, закрыть vitest-тестами все ветки
  (403 справочник/админка, 503, `found=false`, debounce-пикер) до и после.
- **Курсорная пагинация перепутана с offset** → DataTable принимает только
  непрозрачный курсор, пробрасывает без интерпретации; тест на «пустой курсор → нет
  следующей страницы».
- **Самописный focus-trap хрупок** → покрыть тестами фокус/ESC/возврат фокуса;
  при необходимости — минимальная утилита, без тяжёлого UI-кита и без понижения zod.
- **Контраст WCAG AA в обеих темах** → подобрать семантические токены под AA,
  проверить вручную/линтером; статусы — не только цветом (иконка+текст).
- **Случайный diff кодогена (`gen:check` красный)** → не править `web/src/api/*` и
  `*/openapi.yaml`; при возникновении потребности в правке контракта — остановиться,
  это сигнал выхода за scope.

## Migration Plan

1. Ветка `change/portal-ui-design-system` от master.
2. Токены + семантика статусов/критичности (расширение `index.css`,
   `StatusBadge`/бейдж критичности) + примитивы (DataTable, Sidebar, Dialog,
   ConfirmDialog, Toast, PageHeader) с unit-тестами — зелёный web-test.
3. Единый модуль маппинга кодов (`web/src/lib`), консолидация `httpStatusOf`/
   сообщений.
4. Рефактор страниц по матрице, по одной, с зелёными `tsc`/vitest на каждом шаге.
5. PR с полным зелёным CI (включая `gen:check` без diff); merge → отдельный PR
   sync+archive (`/opsx:archive`).

Откат: изменения изолированы в `./web` и в git-ветке; откат = revert ветки/PR,
бэкенд и контракт не затронуты.

## Open Questions

  при реализации семантики, проверив WCAG AA в обеих темах.
- Командная палитра (Cmd/Ctrl-K), хлебные крошки, страница 404/forbidden,
  переключатель плотности — включать по обоснованию; базово предусмотрены ключи
  localStorage и `PageHeader`, остальное — опционально в рамках scope.
