# portal-ui Specification (delta)

## ADDED Requirements

### Requirement: Слой дизайн-токенов портала

Портал ДОЛЖЕН (MUST) определять единый слой дизайн-токенов (цвет, типографика,
шкала отступов, радиусы, тени) поверх CSS-переменных и Tailwind `@theme`, общий
для всех разделов. Токены ДОЛЖНЫ (MUST) быть согласованы для светлой и тёмной тем
и переключаться через существующий `ThemeProvider`/`ThemeToggle` без регрессий;
существующий ключ localStorage `idp-theme` и платформенный дефолт (тёмная тема)
ДОЛЖНЫ (MUST) сохраняться. Примитивы UI ДОЛЖНЫ (MUST) использовать только токены
(не «магические» значения), чтобы редизайн оставался консистентным.

#### Scenario: Переключение темы сохраняет согласованность токенов

- **GIVEN** портал открыт в тёмной теме
- **WHEN** пользователь переключает тему через `ThemeToggle`
- **THEN** все примитивы перерисовываются по токенам светлой темы, выбор
  сохраняется в localStorage (`idp-theme`), мигания темы (FOUC) нет

#### Scenario: Дефолт темы при отсутствии сохранённого выбора

- **GIVEN** в localStorage нет ключа `idp-theme`
- **WHEN** портал загружается
- **THEN** применяется платформенный дефолт — тёмная тема — на токенах
  дизайн-системы

### Requirement: Семантика статусов и критичности (цвет + иконка + текст)

Портал ДОЛЖЕН (MUST) отображать статус сервиса (`ServiceStatus`:
creating/active/decommissioned/failed/transferring) и уровень критичности системы
консистентным бейджем, кодируя значение ОДНОВРЕМЕННО цветом, иконкой и текстом
(colorblind-safe), а НЕ только цветом. Маппинг статусов и уровней критичности
ДОЛЖЕН (MUST) быть единым источником правды (одна таблица/компонент), переиспользуемым
во всех разделах. Неизвестный статус ДОЛЖЕН (MUST) деградировать в нейтральный
бейдж с исходной строкой, не ломая отрисовку.

#### Scenario: Бейдж статуса кодирует значение тремя признаками

- **GIVEN** сервис в статусе `failed`
- **WHEN** отображается бейдж статуса
- **THEN** показаны соответствующий цвет, иконка и текстовая подпись «Ошибка»
  одновременно

#### Scenario: Неизвестный статус деградирует безопасно

- **GIVEN** периметр прислал статус вне известного набора
- **WHEN** отображается бейдж
- **THEN** показан нейтральный бейдж с исходной строкой и нейтральной иконкой,
  отрисовка не ломается

### Requirement: Переиспользуемый компонент DataTable

Портал ДОЛЖЕН (MUST) предоставлять переиспользуемый компонент таблицы (DataTable)
с доступной разметкой (`role="table"`, заголовки колонок, `aria-sort`),
выравниванием колонок по типу данных, регулируемой плотностью строк и опциональной
«липкой» шапкой. DataTable ДОЛЖЕН (MUST) поддерживать единые состояния: loading
(скелет строк), empty (единый пустой экран) и error (единый блок ошибки). Там, где
данных мало и они загружены целиком, ДОЛЖНА (MUST) поддерживаться клиентская
сортировка по колонкам с отражением направления в `aria-sort`.

#### Scenario: Состояние загрузки показывает скелет

- **GIVEN** запрос данных таблицы в процессе
- **WHEN** DataTable рендерится без данных
- **THEN** показывается скелет строк (loading), а не пустой экран и не ошибка

#### Scenario: Пустой и ошибочный результат

- **GIVEN** запрос завершился пустым набором либо ошибкой
- **WHEN** DataTable рендерится
- **THEN** при пустом наборе показан единый empty-state, при ошибке — единый
  error-блок (без раскрытия сырых внутренних деталей)

#### Scenario: Клиентская сортировка с aria-sort

- **GIVEN** таблица с полностью загруженным малым набором строк
- **WHEN** пользователь активирует сортировку по колонке (мышью или с клавиатуры)
- **THEN** строки сортируются по этой колонке, а заголовок отражает направление
  через `aria-sort`

### Requirement: Курсорная пагинация DataTable по ADR-0009

DataTable ДОЛЖЕН (MUST) поддерживать КУРСОРНУЮ (keyset/opaque), а НЕ offset,
пагинацию: курсор (`next_page_token` для `/iam/subjects` и `/projects/.../services`,
`next_cursor` для `/iam/directory/*`) пробрасывается без интерпретации (ADR-0009).
Пустой/отсутствующий курсор ДОЛЖЕН (MUST) означать конец выборки и скрывать
управление «дальше». Управление пагинацией ДОЛЖНО (MUST) быть доступно с клавиатуры.

#### Scenario: Переход на следующую страницу по курсору

- **GIVEN** ответ периметра содержит непустой `next_page_token`
- **WHEN** пользователь запрашивает следующую страницу
- **THEN** DataTable передаёт непрозрачный курсор в следующий запрос без его
  интерпретации и дозагружает строки

#### Scenario: Конец выборки скрывает «дальше»

- **GIVEN** ответ периметра вернул пустой курсор (`next_page_token`/`next_cursor`)
- **WHEN** DataTable обрабатывает страницу
- **THEN** управление «дальше» скрыто/заблокировано, повторный запрос не уходит

### Requirement: Раздвигающееся левое меню (collapsible sidebar)

Портал ДОЛЖЕН (MUST) предоставлять раздвигающееся левое меню с группировкой
разделов и подсветкой активного маршрута (react-router). Меню ДОЛЖНО (MUST)
поддерживать свёрнутый icon-only режим с тултипами, сохранять состояние
свёрнуто/развёрнуто в localStorage (по аналогии с темой, отдельный стабильный
ключ), быть управляемым с клавиатуры и иметь корректные ARIA-роли навигации. На
узких экранах меню ДОЛЖНО (MUST) переходить в off-canvas режим. Недоступность
localStorage НЕ ДОЛЖНА (MUST NOT) ломать меню (деградация к дефолту).

#### Scenario: Свернуть/развернуть с сохранением состояния

- **GIVEN** меню развёрнуто
- **WHEN** пользователь сворачивает его
- **THEN** меню переходит в icon-only режим с тултипами, состояние сохраняется в
  localStorage и восстанавливается при следующей загрузке

#### Scenario: Подсветка активного маршрута

- **GIVEN** открыт маршрут «Роли и доступы»
- **WHEN** отрисовано меню
- **THEN** соответствующий пункт подсвечен как активный (`aria-current`),
  остальные — нет

#### Scenario: Клавиатурная навигация по меню

- **GIVEN** фокус на первом пункте меню
- **WHEN** пользователь перемещается по пунктам с клавиатуры и активирует пункт
- **THEN** фокус видим, активация переходит на маршрут, порядок фокуса логичен

#### Scenario: Недоступность localStorage не ломает меню

- **GIVEN** localStorage недоступен (приватный режим)
- **WHEN** меню инициализируется
- **THEN** применяется дефолтное состояние, ошибок нет, меню работает

### Requirement: Модальные окна (Dialog) для действий

Портал ДОЛЖЕН (MUST) предоставлять единый примитив модального окна (Dialog) с
focus-trap (фокус заперт внутри окна), возвратом фокуса на инициатор после
закрытия, закрытием по ESC и по клику на overlay, и блокировкой прокрутки фона на
время открытия. Формы внутри модалок ДОЛЖНЫ (MUST) оставаться на react-hook-form +
zod (источник правды валидации), а ответы API — проходить рантайм-валидацию zod
`.parse`. Окно ДОЛЖНО (MUST) иметь корректные ARIA (`role="dialog"`,
`aria-modal`, связь заголовка через `aria-labelledby`).

#### Scenario: Открытие захватывает фокус

- **GIVEN** пользователь нажал кнопку действия
- **WHEN** модальное окно открывается
- **THEN** фокус перемещается внутрь окна и заперт в нём, фон не прокручивается

#### Scenario: Закрытие по ESC возвращает фокус

- **GIVEN** открыто модальное окно
- **WHEN** пользователь нажимает ESC (или кликает overlay)
- **THEN** окно закрывается, фокус возвращается на элемент-инициатор

#### Scenario: Сабмит формы в модалке

- **GIVEN** в модалке форма на react-hook-form + zod
- **WHEN** пользователь отправляет валидные данные
- **THEN** выполняется мутация, по успеху окно закрывается; невалидный ввод
  блокируется zod без отправки запроса

### Requirement: Confirm-диалог для деструктивных действий

Портал ДОЛЖЕН (MUST) предоставлять отдельный `ConfirmDialog` для деструктивных
операций (удаление роли, удаление права, открепление права, decommission,
transfer). Деструктивное действие НЕ ДОЛЖНО (MUST NOT) выполняться без явного
подтверждения в диалоге. Для особо рискованных/необратимых операций (decommission,
transfer) подтверждение ДОЛЖНО (MUST) сохранять текущие гарантии (ввод имени
сервиса и явные предусловия, например снятая нагрузка для decommission). Диалог
ДОЛЖЕН (MUST) наследовать a11y-гарантии модалки (focus-trap, ESC, возврат фокуса).

#### Scenario: Деструктивное действие требует подтверждения

- **GIVEN** пользователь инициировал удаление пользовательской роли
- **WHEN** открывается `ConfirmDialog`
- **THEN** действие выполняется только после явного подтверждения; отмена/ESC
  закрывает диалог без мутации

#### Scenario: Необратимая операция сохраняет усиленное подтверждение

- **GIVEN** активный сервис и действие «Вывести из эксплуатации»
- **WHEN** открыт confirm-диалог
- **THEN** требуется ввод имени сервиса и отметка снятой нагрузки (`load_drained`);
  кнопка подтверждения активна только при их выполнении

### Requirement: Система уведомлений (toast) с маппингом кодов периметра

Портал ДОЛЖЕН (MUST) показывать результат действий (успех/ошибка) через единую
систему уведомлений (toast), а не разрозненным инлайновым текстом. Ошибки ДОЛЖНЫ
(MUST) переводиться единым UI-маппингом кодов периметра (400/403/404/409/422/503)
в понятные стабильные сообщения, БЕЗ раскрытия сырых внутренних ошибок сервера.
Рантайм-валидация ответов zod `.parse` ДОЛЖНА (MUST) сохраняться. Тосты ДОЛЖНЫ
(MUST) быть доступны для скринридеров (`role="status"`/`aria-live`) и уважать
`prefers-reduced-motion`.

#### Scenario: Тост успеха после действия

- **GIVEN** пользователь выполнил действие (например, назначил роль)
- **WHEN** мутация завершилась успешно
- **THEN** показывается тост успеха, объявляемый ассистивным технологиям

#### Scenario: Маппинг кода ошибки в понятное сообщение

- **GIVEN** периметр вернул `403` (или `404`/`409`/`422`/`503`/`400`)
- **WHEN** обрабатывается ошибка действия
- **THEN** показывается тост с понятным сообщением, соответствующим коду, без
  сырых внутренних деталей сервера

#### Scenario: 503 как деградация (retryable)

- **GIVEN** периметр вернул `503` (внешняя зависимость недоступна)
- **WHEN** обрабатывается ошибка
- **THEN** тост сообщает о временной недоступности и возможности повторить, не
  ломая остальной UI

### Requirement: Доступность портала (WCAG AA, фокус, клавиатура, reduced-motion)

Портал ДОЛЖЕН (MUST) соответствовать доступности как требованию первого класса:
контраст текста/иконок ДОЛЖЕН (MUST) удовлетворять WCAG AA в ОБЕИХ темах;
интерактивные элементы (меню, таблицы, диалоги, формы) ДОЛЖНЫ (MUST) быть
управляемы с клавиатуры с видимым фокусом и корректными ARIA-ролями; статусы и
критичность ДОЛЖНЫ (MUST) кодироваться не только цветом; анимации ДОЛЖНЫ (MUST)
уважать `prefers-reduced-motion`. Управление фокусом ДОЛЖНО (MUST) быть корректным
при открытии/закрытии модалок и навигации.

#### Scenario: Полная клавиатурная управляемость

- **GIVEN** пользователь работает без мыши
- **WHEN** он проходит по меню, таблице, открывает модалку и отправляет форму
- **THEN** все интерактивные элементы достижимы и активируемы с клавиатуры, фокус
  видим и логичен

#### Scenario: Контраст в обеих темах

- **GIVEN** портал в светлой и в тёмной теме
- **WHEN** проверяется контраст текста и иконок (включая бейджи статусов)
- **THEN** контраст удовлетворяет WCAG AA в обеих темах

#### Scenario: Уважение prefers-reduced-motion

- **GIVEN** в системе включён `prefers-reduced-motion`
- **WHEN** появляются тосты/модалки/анимации меню
- **THEN** переходы редуцируются/отключаются, информация остаётся доступной

### Requirement: Рефактор страниц на единые примитивы с сохранением поведения

Портал ДОЛЖЕН (MUST) перевести существующие страницы на новые примитивы, СОХРАНИВ
текущее поведение, маршруты и права: каталог сервисов и деталь сервиса —
DataTable + бейджи статусов/критичности + действия (создать/decommission/transfer/
смена владельцев) через модалки/`ConfirmDialog`; создание сервиса — форма
(react-hook-form + zod) в едином паттерне; «Роли и доступы» — таблицы ролей/прав/
субъектов, структурные действия (создать/удалить роль/право, attach/detach) в
модалках/confirm, тосты вместо инлайнового текста. Чтение и мутации ДОЛЖНЫ (MUST)
идти через существующий сгенерированный из OpenAPI zodios-клиент с рантайм-
валидацией ответов (zod `.parse`); контракт и кодоген НЕ меняются.

#### Scenario: Каталог сервисов на DataTable сохраняет поведение

- **GIVEN** периметр вернул валидный список сервисов
- **WHEN** открыт каталог
- **THEN** сервисы показаны в DataTable с именем, владельцами и бейджем статуса;
  переход к детали сервиса работает как раньше; ответы проходят zod `.parse`

#### Scenario: Действие сервиса перенесено в модалку

- **GIVEN** деталь активного сервиса
- **WHEN** пользователь инициирует decommission/transfer/смену владельцев
- **THEN** действие выполняется через модалку/`ConfirmDialog` с сохранением текущих
  гарантий (подтверждение, предусловия, optimistic-concurrency владельцев), а не
  инлайновой формой

#### Scenario: Результат действия — через тост

- **GIVEN** действие на отрефакторенной странице завершилось успехом или ошибкой
- **WHEN** обрабатывается результат
- **THEN** показывается тост (успех/маппинг кода ошибки), инлайновый разрозненный
  текст не используется

### Requirement: Сохранение поведения IAM-пикера и обработки 403/503/осиротевших при редизайне

Редизайн раздела «Роли и доступы» ДОЛЖЕН (MUST) сохранить пикер пользователя из
справочника Keycloak (поиск с debounce по `GET /iam/directory/subjects`,
отображение `username`/`email`, назначение по каноническому ключу `sub`) и всю
обработку деградации: при `403` на справочник (`read, iam:directory`) пикер/поиск
скрыт и заменён сообщением об отказе (просмотр и снятие существующих ролей
остаются); при `503` поле поиска остаётся (запрос retryable); «осиротевшие»
субъекты (`found=false`) показываются как raw `sub` с пометкой «нет в каталоге».
При `403` на саму админку (`read, iam:global`) содержимое каталога НЕ показывается
(fail-closed на UI). Эти гарантии НЕ ДОЛЖНЫ (MUST NOT) деградировать при переносе
на новые примитивы.

#### Scenario: Пикер сохраняется после редизайна

- **GIVEN** есть право `(read, iam:directory)` и справочник доступен
- **WHEN** пользователь ищет и выбирает пользователя в новой подаче формы
  назначения
- **THEN** назначение выполняется по каноническому `sub`, поведение пикера
  (debounce, отображение `username`/`email`) сохранено

#### Scenario: 403 на справочник скрывает пикер, не ломая роли

- **GIVEN** нет права `(read, iam:directory)` (справочник отвечает `403`)
- **WHEN** открыт раздел «Роли и доступы» в новой подаче
- **THEN** пикер/поиск скрыт с сообщением об отказе, просмотр и снятие
  существующих ролей остаются доступны

#### Scenario: 503 каталога — деградация с сохранением поиска

- **GIVEN** `/iam/directory/subjects` отвечает `503`
- **WHEN** пользователь ищет в пикере
- **THEN** показана индикация «каталог недоступен» (через единый UI), поле поиска
  остаётся, управление существующими ролями работает

#### Scenario: 403 на админку — fail-closed на UI

- **GIVEN** нет права `(read, iam:global)` (периметр отвечает `403`)
- **WHEN** пользователь открывает раздел
- **THEN** содержимое каталога не отображается, показан отказ доступа, сырые
  ошибки не раскрываются

#### Scenario: Осиротевший субъект помечен

- **GIVEN** субъект с `found=false`
- **WHEN** открыт список субъектов на DataTable
- **THEN** показан raw `sub` с пометкой «нет в каталоге», действия управления его
  ролями доступны
