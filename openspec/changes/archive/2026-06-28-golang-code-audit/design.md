## Context

Монорепо IDP содержит пять Go-модулей в go.work (`pkg`,
`services/{gateway,idm,projects,devinfra-worker}`), модуль `tests/e2e` и
изолированный `tools` (вне go.work, `GOWORK=off`). Go 1.26.4. Качество уже
охраняется CI (`.github/workflows/ci.yml`): `go test -race -shuffle=on` по
матрице модулей; `golangci-lint` v2.12.2; `govulncheck` (blocking, `GOWORK=off`);
`go mod tidy --check` (`GOWORK=off`); integration-тесты на Postgres 17. Корневой
`.golangci.yml` — единая конфигурация линтеров.

Этот change — аналитический. Цель — не переписать код, а получить
прослеживаемую картину техдолга и расхождений с инженерными стандартами
(`openspec/config.yaml`, `docs/IDP_MVP_План.md` BLOCK 0, ADR 0001..0023),
которые CI-гейты по своей природе не выявляют (архитектурные слои, идиоматичность,
покрытие критичных путей, дизайн надёжности). Аудит должен опираться на вывод
существующих инструментов как на baseline и искать то, что они НЕ покрывают,
а не дублировать их.

Учитываются нюансы из памяти: `vault-real-api-divergences` (gosec G101,
реальный Vault vs mock), `e2e-real-oidc-wiring`, `e2e-change-owners-single-shot`,
и хрупкость кросс-модульного `tidy` (`gowork-cross-module-tidy`: bump общей
зависимости ломает `tidy`/`govulncheck` в зависимом `devinfra-worker` через
`replace ../projects`).

## Goals / Non-Goals

**Goals:**
- Прогнать 10 доменных скиллов `samber/cc-skills-golang@*` под оркестратором
  `golang-how-to` по Go-коду в границах и собрать находки.
- Выпустить `docs/audits/2026-06-golang-audit.md`: сводная таблица severity →
  10 доменных разделов → приоритизированный бэклог ремедиаций.
- Разделить бэклог на «чинить в этом change» (низкорисковые механические правки)
  и follow-up changes по доменам.
- Опционально применить правки категории «сейчас», сохранив зелёный CI.

**Non-Goals:**
- Широкое переписывание кода, поведенческие изменения, новые фичи.
- Изменения контрактов (`.proto`/OpenAPI) и ручная правка сгенерированного кода.
- Пересмотр принятых ADR (значимое выносится отдельным ADR).
- Дублирование того, что уже ловит CI (race, lint-набор, vuln, tidy).
- Любые правки фронтенда `./web`.

## Decisions

**Решение 1: Baseline-first методология.**
Сначала фиксируем фактическое состояние инструментов (что уже зелёное/что
шумит), затем аудит ищет пробелы поверх baseline. Альтернатива — аудит «с нуля»
без учёта CI — отвергнута: дублирует находки линтеров и раздувает отчёт.

**Решение 2: Один отчёт, сгруппированный по 10 доменам.**
Единый markdown-файл с устойчивым форматом находки
(`severity | file:line | описание | правило/скилл | рекомендация`) и сводной
таблицей сверху. Альтернатива — отдельный файл на домен — отвергнута: труднее
обозревать и приоритизировать сквозь домены.

**Решение 3: Двухкатегорийный бэклог.**
«Чинить сейчас» = только механические, не меняющие поведение правки
(modernize/style/очевидный safety), проходящие весь CI. Всё остальное — в
follow-up changes с явным именем и доменом. Критерий разделения: меняет ли
правка поведение/контракт/публичный API или требует доменного решения.

**Решение 4: Скиллы как источник правил.**
Каждый домен запускает соответствующий скилл и применяет его чек-лист; находка
ссылается на конкретное правило скилла. Это обеспечивает прослеживаемость
рекомендаций и единый язык severity.

**Решение 5: Применяемые правки изолированы по модулю.**
Любая правка прогоняется через `golangci-lint` и `go test -race` своего модуля;
при касании общих зависимостей — `go mod tidy` (`GOWORK=off`) во всех затронутых
модулях, включая `devinfra-worker`, чтобы не повторить PR #75.

## Risks / Trade-offs

- **Шум/дубликаты с линтерами** → находку включаем только если она вне покрытия
  `.golangci.yml`/`govulncheck`; явно отмечаем «уже ловится CI» где уместно.
- **Subjective severity** → фиксируем критерии severity в шапке отчёта
  (blocker = ломает безопасность/корректность; high = риск инцидента; medium =
  техдолг с обходом; low = стиль/косметика).
- **Соблазн «починить всё»** → жёсткий критерий категории (а): только
  механические правки; спорное уходит в follow-up.
- **Поломка кросс-модульного `tidy`** → правки зависимостей избегаем в этом
  change; если неизбежно — повторный `tidy` по всем зависимым модулям (memory
  `gowork-cross-module-tidy`).
- **Ложноположительные англокомментарии** → проверка «комментарии только на
  русском» отделяет строковые литералы/идентификаторы от собственно комментариев.
- **Объём** → при большом числе находок отчёт перечисляет репрезентативные
  примеры на правило + счётчик, а не каждую строку, чтобы оставаться обозримым.
