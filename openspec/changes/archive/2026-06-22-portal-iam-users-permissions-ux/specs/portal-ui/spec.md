## MODIFIED Requirements

### Requirement: Страница «Права» (каталог прав)

Страница «Права» (`/iam/permissions`) ДОЛЖНА (MUST) показывать каталог прав на
DataTable (сортировка, состояния loading/empty/error). Создание права ДОЛЖНО (MUST)
выполняться в модалке (Dialog) с react-hook-form + zod (поля `action`, `resource`);
при вводе пользователю ДОЛЖНЫ (MUST) предлагаться доступные значения — РАЗЛИЧНЫЕ
`action` и РАЗЛИЧНЫЕ `resource`, выведенные из уже загруженного каталога
(`listPermissions`, тот же кэш, без новых эндпоинтов), при этом ввод НОВОГО валидного
значения ДОЛЖЕН (MUST) оставаться возможным (нативный `<datalist>` — без тяжёлых
UI-китов). Каталог прав ДОЛЖЕН (MUST) поддерживать КЛИЕНТСКУЮ фильтрацию: поиск
подстрокой по `action`/`resource` и фильтр по типу (`system`/`user`), доступный с
клавиатуры, с корректным пустым состоянием «ничего не найдено» и сохранением текущей
сортировки, плотности (`idp-density`) и пагинации. Удаление пользовательского права —
через ConfirmDialog. Системные права (`system=true`) ДОЛЖНЫ (MUST) показываться
read-only (бейдж «системное», кнопка удаления скрыта/заблокирована). Мутации ДОЛЖНЫ
(MUST) инвалидировать `["iam","permissions"]` (и права ролей, где уместно) и
показывать результат тостом с единым маппингом кодов (403 `manage`/409/422/400).
Ответы ДОЛЖНЫ (MUST) проходить `zod .parse`; сырые внутренние ошибки наружу НЕ
показываются.

#### Scenario: Создание права c подсказками действий/ресурсов

- **GIVEN** в каталоге есть права с различными `action`/`resource`
- **WHEN** пользователь открывает модалку создания и фокусирует поле `action`/`resource`
- **THEN** предлагаются доступные значения, выведенные из каталога; при выборе или
  вводе валидного значения и отправке выполняется мутация создания, каталог
  инвалидируется и обновляется, показывается тост

#### Scenario: Создание права с новым значением

- **WHEN** пользователь вводит ранее не встречавшееся валидное `action`/`resource`
  (которого нет в подсказках) и отправляет форму
- **THEN** ввод не блокируется на клиенте, выполняется мутация создания; результат
  периметра (успех/409/422/400) показывается тостом без раскрытия сырых ошибок

#### Scenario: Фильтрация каталога по подстроке

- **GIVEN** каталог прав длиннее нескольких строк
- **WHEN** пользователь вводит подстроку в поле поиска и/или выбирает тип
  `system`/`user`
- **THEN** таблица показывает только совпадающие права, сортировка/плотность/
  пагинация сохраняются

#### Scenario: Пустой результат фильтра

- **WHEN** под фильтр не подпадает ни одно право
- **THEN** показывается единый empty-state «ничего не найдено» (а не состояние
  ошибки/загрузки), фильтр остаётся доступным для изменения

#### Scenario: Удаление пользовательского права

- **GIVEN** пользовательское право в каталоге
- **WHEN** пользователь удаляет его через ConfirmDialog
- **THEN** выполняется мутация удаления, каталог обновляется, показывается тост

#### Scenario: Системное право read-only

- **GIVEN** право с `system=true`
- **WHEN** отрисован каталог прав
- **THEN** показан бейдж «системное», кнопка удаления скрыта/заблокирована

### Requirement: Страница «Пользователи» (субъекты, назначение роли, пикер)

Страница «Пользователи» (`/iam/users`) ДОЛЖНА (MUST) показывать на экране ТОЛЬКО
список субъектов с их ролями на DataTable с КУРСОРНОЙ пагинацией (`next_page_token`,
ADR-0009); форма назначения роли НЕ ДОЛЖНА (MUST NOT) располагаться карточкой над
таблицей. Назначение роли ДОЛЖНО (MUST) запускаться кнопкой «Назначить роль» в
`PageHeader.actions` и выполняться в МОДАЛЬНОМ окне (Dialog). Назначение ДОЛЖНО (MUST)
выполняться ТОЛЬКО выбором пользователя из справочника Keycloak через пикер с поиском
(debounce): ввод запускает `GET /iam/directory/subjects?search=`, показывает
`username`/`email`, при выборе используется канонический `subject` (`sub` UUID).
Модалка ДОЛЖНА (MUST) сбрасывать состояние пикера и формы при закрытии (ESC/overlay/
крестик/Отмена) и после успешного назначения, а фокус ДОЛЖЕН (MUST) возвращаться на
кнопку-инициатор. Снятие роли ДОЛЖНО (MUST) подтверждаться ConfirmDialog.
«Осиротевшие» субъекты (`found=false`) ДОЛЖНЫ (MUST) показываться как raw `sub` с
пометкой «нет в каталоге», не ломая управление их ролями. Мутации ДОЛЖНЫ (MUST)
инвалидировать запрос субъектов и показывать результат тостом; назначение
идемпотентно. Ответы ДОЛЖНЫ (MUST) проходить `zod .parse`.

#### Scenario: Открытие модалки назначения из заголовка

- **GIVEN** раздел доступен (есть `read, iam:global`)
- **WHEN** пользователь нажимает «Назначить роль» в `PageHeader`
- **THEN** открывается Dialog с пикером и выбором роли; на странице вне модалки
  остаётся только таблица субъектов

#### Scenario: Назначение роли выбором из справочника

- **GIVEN** в открытой модалке введена строка, справочник вернул пользователя
- **WHEN** пользователь выбран и подтверждено назначение роли
- **THEN** выполняется назначение с `subject` = канонический `sub` выбранного
  пользователя, список субъектов обновляется, показывается тост, модалка
  закрывается

#### Scenario: Сброс состояния модалки при закрытии

- **GIVEN** в модалке выбран пользователь и/или роль
- **WHEN** пользователь закрывает модалку без отправки и открывает её снова
- **THEN** пикер и форма очищены (прежний выбор не сохранён), фокус возвращён на
  кнопку-инициатор

#### Scenario: Валидация ввода в модалке

- **WHEN** роль выбрана, но пользователь из справочника не выбран (или наоборот)
- **THEN** zod-валидация react-hook-form блокирует отправку и показывает сообщение,
  запрос на периметр не уходит

#### Scenario: Снятие роли с подтверждением

- **GIVEN** субъект с назначенной ролью в таблице
- **WHEN** пользователь снимает роль и подтверждает в ConfirmDialog
- **THEN** выполняется мутация снятия, список обновляется, показывается тост

#### Scenario: Осиротевший субъект помечен

- **GIVEN** субъект с `found=false`
- **WHEN** открыт список субъектов
- **THEN** показан raw `sub` с пометкой «нет в каталоге», кнопки управления ролями
  доступны

#### Scenario: Курсорная пагинация субъектов

- **GIVEN** список субъектов длиннее одной страницы
- **WHEN** пользователь запрашивает следующую страницу
- **THEN** загрузка идёт по `next_page_token`, новые строки дописываются без потери
  предыдущих

#### Scenario: Нет права на справочник в модалке (403)

- **GIVEN** у субъекта нет `(read, iam:directory)` (справочник отвечает `403`)
- **WHEN** пользователь открывает модалку назначения
- **THEN** пикер скрыт с явным отказом, сырые ошибки не раскрываются

#### Scenario: Каталог справочника недоступен в модалке (503)

- **GIVEN** справочник отвечает `503`
- **WHEN** пользователь вводит строку поиска в модалке
- **THEN** показывается индикация «недоступен», поле поиска остаётся (запрос
  ретраебелен)
