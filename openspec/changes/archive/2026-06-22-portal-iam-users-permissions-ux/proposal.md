## Why

После разнесения раздела «Роли и доступы» на три страницы (change
`portal-iam-split-pages`, ADR-0017) две страницы остались эргономически
недоведёнными: на «Пользователях» форма назначения роли с пикером висит большой
карточкой НАД таблицей (выбивается из паттерна «таблица + кнопка → модалка»,
принятого на «Ролях»/«Правах»), а на «Правах» создание права требует от
пользователя угадывать значения двух свободных текстовых полей (`action`,
`resource`) и не даёт фильтровать растущий каталог. Это чистая UX-консолидация
фронтенда: контракт периметра (OpenAPI), proto и поведение бэкенда НЕ меняются.

Соответствие: ADR-0017 «Дизайн-система и UI-архитектура портала»;
docs/IDP_MVP_План.md (этап портала/IAM-админка). Архитектурно затрагивается ТОЛЬКО
фронтенд `./web` — gRPC/Temporal-границы и сервисы не задействованы.

## What Changes

- **Страница «Пользователи»** (`web/src/pages/iam/UsersPage.tsx`): убрать форму-
  карточку назначения роли над таблицей. На экране остаётся ТОЛЬКО `DataTable`
  субъектов; назначение роли переезжает в модальное окно (`Dialog`), открываемое
  кнопкой «Назначить роль» в `PageHeader.actions` — по образцу «Роли»/«Права».
  Внутри модалки — тот же пикер пользователя из справочника Keycloak (debounce,
  `username`/`email`, канонический `sub`) и выбор роли. Снятие роли
  (`ConfirmDialog`), пометка осиротевших (`found=false`), курсорная пагинация —
  без изменений.
- **Страница «Права», форма создания**: вместо двух свободных текстовых полей
  показывать пользователю доступные ДЕЙСТВИЯ и РЕСУРСЫ как подсказки, сохранив
  возможность ввести валидное (в т.ч. новое) значение. Источник подсказок —
  РАЗЛИЧНЫЕ `action`/`resource`, выведенные из уже загруженного каталога
  `listPermissions` (отдельного эндпоинта в контракте НЕТ — см. design).
- **Страница «Права», таблица каталога**: добавить КЛИЕНТСКУЮ фильтрацию (поиск
  подстрокой по `action`/`resource` + фильтр по типу `system`/`user`) с корректным
  empty-state «ничего не найдено» и сохранением сортировки/плотности/пагинации.
- Поведение прав и обработка ошибок (403 admin → fail-closed через `IamGuard`; 403
  `iam:directory` → пикер скрыт; 503 каталога → индикация, поиск остаётся; 403
  `manage`/409/422/400 на каталоге; тосты; `zod .parse`) — полностью СОХРАНЯЮТСЯ.
- Контракт и кодоген НЕ трогаем: `openapi/openapi.yaml`, `web/src/api/*`,
  `web/public/openapi.yaml` — без изменений (`gen:check` остаётся зелёным).

### Modified Capabilities
- `portal-ui`: MODIFIED требования «Страница «Пользователи» (субъекты, назначение
  роли, пикер)» (назначение роли → в модалке, страница = таблица + кнопка в
  `PageHeader`) и «Страница «Права» (каталог прав)» (подсказки доступных
  `action`/`resource` при создании + клиентский фильтр таблицы). Эквивалентные
  гарантии по 403/503/осиротевшим/пикеру сохраняются.

## Impact

- Затрагиваемый код (только `./web`): `web/src/pages/iam/UsersPage.tsx`,
  `web/src/pages/iam/PermissionsPage.tsx`, при обосновании —
  `web/src/components/ui/data-table.tsx` (тулбар-слот фильтра); тесты
  `web/src/pages/iam/{UsersPage,PermissionsPage}.test.tsx`.
- Переиспользуются без изменений: `web/src/pages/iam/hooks.ts` (запросы/мутации/
  `useDebounced`/overrides), `IamGuard`, примитивы `Dialog`/`ConfirmDialog`/
  `DataTable`/`PageHeader`/`Input`/`toast`, `lib/errors.ts`, `lib/ui-state.ts`.
- API/контракт/proto/сервисы/RBAC/миграции — НЕ затрагиваются. gRPC/Temporal-
  границы не задействованы. Отката провизии ресурсов нет (изменение чисто UI).
- Риски: не сломать существующие использования `DataTable` (Роли/Пользователи) и
  их тесты при расширении примитива; сохранить эквивалентное покрытие vitest.
