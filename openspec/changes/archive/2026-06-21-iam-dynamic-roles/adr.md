## Новые ADR

- **ADR-0015 — Динамический каталог IAM: полномочие manage, защита системных ролей,
  инвалидация поколением**
  (`docs/adr/0015-iam-dynamic-catalog-manage-and-system-protection.md`):
  структурные мутации каталога RBAC (создание/удаление ролей и прав, attach/detach
  прав роли) защищаются ОТДЕЛЬНЫМ привилегированным действием `manage` на
  горизонтальном ресурсе `iam:global` (отдельно от `read`/`write`, без неявного
  наследования; `CheckAccess` перед КАЖДОЙ ручкой, fail-closed); мутирующий контракт
  выносится в новый сервис `IamCatalogService` (`proto/idm/v1`), сообщения
  `Role`/`Permission` аддитивно расширяются полем `system`; системные (сидированные)
  роли/права помечаются колонкой `system` (обратимая миграция, backfill всех
  существующих) и защищены от удаления и правки набора прав (попытка →
  `FailedPrecondition` → `422`); любая структурная мутация → ШИРОКАЯ инвалидация
  кэша поколением `idm:cache:gen` (точечной по субъекту недостаточно — затронуты все
  носители роли), assign/revoke остаются точечными; удаление роли «в использовании»
  — каскад (`subject_roles`/`role_permissions` FK) + широкая инвалидация; коды:
  create-дубль→409, attach/detach идемпотентны→200, delete-нет→404; scope MVP
  исключает иерархии/wildcard/ABAC, аудит-лог, реестр пользователей.

Канонический ADR-0015 (для публикации при apply):

```markdown
# ADR-0015: Динамический каталог IAM — полномочие manage, защита системных ролей, инвалидация поколением

**Status:** Accepted
**Date:** 2026-06-21
**Change:** iam-dynamic-roles

## Context

IAM-админка фазы 1 (ADR-0014) дала просмотр каталога ролей/прав и назначение
существующих ролей субъектам (assign/revoke под `(write, iam:global)`). Сам каталог
статичен: роли, права и связки `role_permissions` заводятся ТОЛЬКО миграциями
(ADR-0010/0011). Нужна динамическая правка каталога из портала/периметра.
Структурная правка опаснее назначения: меняется сам набор прав; правка
`role_permissions`/удаление роли затрагивает ВСЕХ носителей роли разом, поэтому
точечной инвалидации кэша по субъекту недостаточно; удаление сидированных ролей
(`iam-admin`, `owner:project:*`, `project-creator`) или их прав сломало бы платформу.
Опирается на ADR-0003 (fail-closed), ADR-0009 (периметр/OpenAPI/TS), ADR-0010
(RBAC и кэш решений: поколение `idm:cache:gen` + `InvalidateSubject`), ADR-0011
(контракт ролей), ADR-0012/0013 (маппинг ошибок), ADR-0014 (модель админки
`read`/`write` на `iam:global`, обобщённый `authorizeResource`).

## Decision

- **Привилегированное действие `manage` на `iam:global`.** Все структурные мутации
  каталога (create/delete роли, create/delete права, attach/detach прав роли)
  авторизуются действием `manage` — отдельным от `read` (чтение) и `write`
  (assign/revoke). Укладывается в существующую модель (action + resource,
  strict-match, deny-by-default). Менять сам каталог прав привилегированнее, чем
  назначить существующую роль → три уровня доверия (аудитор `read`, оператор
  назначений `write`, администратор каталога `manage`). Неявного наследования НЕТ
  (`manage` не подразумевает `write`/`read` в модели); гранты роли админки выдаются
  явно сидированием (`iam-admin` получает `read`+`write`+`manage`). gateway зовёт
  `CheckAccess(subject, "iam:global", "manage")` перед КАЖДОЙ структурной ручкой
  (fail-closed → 403); helper — существующий `authorizeResource`.
- **Мутирующий контракт — новый сервис `IamCatalogService`** в `proto/idm/v1`
  (отдельно от читающего `IamAdminService` и от `RoleAdminService` привязок):
  `CreateRole`, `DeleteRole`, `CreatePermission`, `DeletePermission`,
  `AttachPermission`, `DetachPermission`. Отдельный сервис делает границу полномочий
  (`manage`) и blast-radius (широкая инвалидация) явными на уровне контракта.
  Изменение аддитивно/wire-совместимо: новый сервис + новые сообщения; читающие RPC
  и assign/revoke переиспользуются. Идентификатор роли — `name`, право — пара
  `action`/`resource`; `id` (UUID) наружу не отдаётся.
- **Признак `system` (колонка, не deny-list).** Колонки `roles.system` и
  `permissions.system` (`boolean NOT NULL DEFAULT false`); сообщения
  `Role`/`Permission` аддитивно расширяются полем `bool system`. Обратимая миграция
  backfill'ом помечает `system=true` ВСЕ существующие роли/права (в текущем
  состоянии все сидированы миграциями; правило «существует до этого change →
  системное»). Создаваемые через API — `system=false`. Колонка — единый источник
  правды (виден в данных и контракте), в отличие от хрупкого deny-list имён.
- **Защита системных → `FailedPrecondition` (422), не 403.** Через API запрещены:
  `DeleteRole` системной роли, `DeletePermission` системного права, attach/detach на
  системную роль (её состав фиксирован сидированием). Прикрепление СИСТЕМНОГО права к
  ПОЛЬЗОВАТЕЛЬСКОЙ роли разрешено (защищается роль, не право). Код `422`, т.к.
  вызывающий авторизован (`manage`) — отказывает предусловие «ресурс не системный»,
  а не авторизация (согласовано с ADR-0012/0013). Rename вне scope.
- **Широкая инвалидация кэша поколением.** Любая структурная мутация ПОСЛЕ commit
  вызывает `InvalidateAll` (инкремент `idm:cache:gen`) — правка
  `role_permissions`/удаление роли/права затрагивает все решения носителей роли;
  точечной `InvalidateSubject` недостаточно (оставила бы устаревший `allow` у прочих
  носителей). Это O(1)-бамп счётчика; цена — холодный кэш после редких структурных
  правок. assign/revoke остаются точечными; чтение — без эффектов на кэш; при откате
  транзакции кэш не трогается.
- **Удаление роли «в использовании» — каскад.** FK `subject_roles.role_id` и
  `role_permissions.role_id` уже `ON DELETE CASCADE` → удаление роли снимает её у
  всех носителей и убирает связки прав, затем широкая инвалидация. Системная роль
  всё равно защищена (422), поэтому каскад применяется лишь к пользовательским.
  Проще и предсказуемее guard'а (не оставляет «залипших» неудаляемых ролей);
  документируется явно.
- **Идемпотентность и коды.** create роли/права НЕ идемпотентны (дубль →
  `AlreadyExists` → 409). attach/detach ИДЕМПОТЕНТНЫ (повтор → 200, `ON CONFLICT DO
  NOTHING`/`DELETE`). delete роли/права несуществующего → `NotFound` → 404 (явный
  администраторский акт, в отличие от идемпотентной привязки revoke). Создание права
  на произвольный `action`/`resource` допустимо (открытый каталог; валидация
  непустоты/utf8/без NUL → 400; матчинг strict — «мусорное» право безвредно). Тела
  ответов несут актуальное состояние (роль/право/набор прав роли) для zod `.parse` и
  TanStack-кэша. Маппинг — существующий `httpFromGRPC` (reuse ADR-0012/0013):
  `PermissionDenied→403`, `NotFound→404`, `AlreadyExists/Aborted→409`,
  `FailedPrecondition→422`, `InvalidArgument→400`; новых правил не требуется.
- **REST-ручки (ADR-0009), все под `manage`.** `POST/DELETE /iam/roles`,
  `POST/DELETE /iam/roles/{role}/permissions` (attach — тело `{action,resource}`,
  detach — query `action`/`resource`), `POST/DELETE /iam/permissions` (delete —
  query). Все коды (200/201/400/403/404/409/422) документируются в OpenAPI
  (Spectral + Schemathesis-конформанс зелёные).
- **Локальный seed.** Обратимые goose-миграции IDM: `0007` (колонки `system` +
  backfill), `0008` (право `(manage, iam:global)` роли `iam-admin`; `demo-user`
  получает транзитивно). Только стенд docker-compose, не прод.

## Consequences

**Положительные:** структурная правка каталога защищена отдельным более
привилегированным правом `manage` (fail-closed); системные роли/права защищены от
удаления/правки колонкой `system` (видна в данных и контракте); широкая инвалидация
поколением исключает устаревший `allow` у всех носителей роли; аддитивный
wire-совместимый контракт; каскадное удаление предсказуемо; маппинг ошибок
переиспользуется; UI получает признак `system` для read-only.

**Отрицательные:** структурная мутация холодит весь кэш решений (приемлемо —
структурные правки редки); горизонтальный `iam:global` — единая грубая точка
полномочий; открытый каталог прав допускает «мусорные» права (безвредны при strict;
без чистки накапливаются); добавление сервиса требует синхронной регенерации Go/TS;
backfill «всё существующее → системное» консервативен (если в каталоге уже есть
пользовательские роли — они тоже станут системными; в текущем состоянии всё
сидировано миграциями, поэтому корректно).

**Альтернативы (отклонены):** переиспользовать `write` для структурных мутаций
(стирает границу «назначить роль» vs «переписать каталог»; нарушает наименьшие
привилегии); неявное наследование `manage ⊃ write ⊃ read` (привилегия утекла бы
скрыто, против strict-match); deny-list имён вместо колонки `system` (хрупок,
невидим в данных, не отдаётся UI); защита системных кодом `403` (смешивает «нет
права» и «ресурс защищён»); точечная инвалидация по субъекту при правке
`role_permissions` (оставит устаревший `allow` у прочих носителей); guard удаления
роли «в использовании» (оставляет залипшие неудаляемые роли); идемпотентный delete
роли (200 на отсутствующую — структурный акт лучше сигналить 404); добавление
мутаций в `IamAdminService` (смешивает read/manage в одном сервисе, размывает
границу полномочий); иерархии/wildcard/ABAC и аудит-лог (вне MVP-scope).

**Связь:** реализует Этап 3 docs/IDP_MVP_plan.md (RBAC/IAM); опирается на
ADR-0003/0009/0010/0011/0012/0013/0014; goose из `./tools` (ADR-0007).
```

## Реализуемые существующие ADR

- **ADR-0003 — Аутентификация/авторизация (fail-closed):** КАЖДАЯ структурная
  мутирующая ручка под `CheckAccess(manage, iam:global)`; недоступность IDM → 403;
  внутренние ошибки наружу не раскрываются.
- **ADR-0009 — Форма ресурсов периметра (REST/OpenAPI/TS):** новые структурные
  `/iam/*` ручки; OpenAPI — источник правды; TS-клиент/zod + `public/openapi.yaml`
  регенерируются (`gen:check`), Spectral + Schemathesis-конформанс зелёные.
- **ADR-0010 — Модель RBAC и кэш решений:** новое действие `manage` в той же модели
  (strict-match, deny-by-default); структурные мутации → широкая инвалидация
  поколением `idm:cache:gen`; чтение без эффектов на кэш.
- **ADR-0011 — Контракт ролей и синхронизация:** assign/revoke переиспользуются
  (точечная `InvalidateSubject`), структурные мутации каталога — отдельный контракт.
- **ADR-0012/0013 — Маппинг ошибок:** `AlreadyExists/Aborted→409`,
  `FailedPrecondition→422`, `NotFound→404`, `InvalidArgument→400`,
  `PermissionDenied→403` через существующий `httpFromGRPC` (без новых правил).
- **ADR-0014 — IAM-админка (фаза 1):** переиспользуются обобщённый
  `authorizeResource`, читающий `IamAdminService` (теперь отдаёт `system`),
  ресурс `iam:global`, раздел портала «Роли и доступы».
