# portal-ui Specification (delta)

## ADDED Requirements

### Requirement: Пикер пользователя из справочника при назначении роли

Раздел портала «Роли и доступы» ДОЛЖЕН (MUST) назначать роль ТОЛЬКО выбором
пользователя из справочника через пикер с поиском (debounce): ввод строки запускает
`GET /iam/directory/subjects?search=` (с задержкой/debounce), показывает совпадения с
`username`/`email`, и при выборе пользователя в качестве `subject` для назначения
роли используется КАНОНИЧЕСКИЙ ключ (`subject` из `SubjectIdentity` = `sub` UUID).
Отдельного поля ручного ввода строки subject в UI быть НЕ ДОЛЖНО (MUST NOT)
(назначение по произвольной строке остаётся доступным на периметре для
совместимости, но не предлагается в портале). Формы — react-hook-form + zod; ответы
API ДОЛЖНЫ (MUST) проходить рантайм-валидацию zod `.parse`; сырые внутренние ошибки
наружу НЕ показываются.

#### Scenario: Назначение роли выбором из справочника

- **GIVEN** в пикере введена строка, справочник вернул пользователя
- **WHEN** пользователь выбран и подтверждено назначение роли
- **THEN** выполняется назначение с `subject` = канонический `sub` выбранного
  пользователя, список пользователей-с-ролями обновляется

#### Scenario: Без выбранного пользователя назначение не уходит

- **WHEN** роль выбрана, но пользователь из справочника не выбран
- **THEN** клиентская валидация (zod) блокирует отправку и просит выбрать
  пользователя; запрос на периметр не уходит

#### Scenario: Валидация ввода поиска

- **WHEN** в поле поиска пустая/слишком короткая строка
- **THEN** запрос к справочнику не отправляется (или ответ `400` обрабатывается без
  сырой ошибки), пикер показывает подсказку о вводе

### Requirement: Отображение идентичностей и обработка осиротевших субъектов

Раздел «Роли и доступы» ДОЛЖЕН (MUST) показывать рядом с субъектами их
`username`/`email` (из обогащённого `GET /iam/subjects`), а «осиротевшие» субъекты
(`found=false` — роль есть, в каталоге нет) ДОЛЖЕН (MUST) показывать как raw `sub` с
пометкой «нет в каталоге», не ломая управление их ролями. Поля идентичности в
zod-схеме ответа ДОЛЖНЫ (MUST) быть опциональными (могут отсутствовать при
деградации/без права).

#### Scenario: Отображение имени и почты

- **GIVEN** субъект обогащён идентичностью (`found=true`)
- **WHEN** открыт список субъектов-с-ролями
- **THEN** рядом с субъектом показаны `username`/`email`

#### Scenario: Осиротевший субъект помечен

- **GIVEN** субъект с `found=false`
- **WHEN** открыт список
- **THEN** показан raw `sub` с пометкой «нет в каталоге», кнопки управления ролями
  доступны

### Requirement: Индикация недоступности каталога и отказа доступа

Портал ДОЛЖЕН (MUST) при недоступном каталоге (`503` от `/iam/directory/*` или
отсутствие идентичностей в `/iam/subjects`) показывать индикацию «каталог
недоступен», НЕ ломая просмотр и снятие ролей у уже известных субъектов. При `503`
поле поиска ДОЛЖНО (MUST) оставаться (запрос ретраебелен). При `403` (нет
`(read, iam:directory)`) пикер/поиск ДОЛЖЕН (MUST) быть скрыт и заменён сообщением об
отказе доступа к каталогу (назначение новой роли через пикер недоступно), при этом
просмотр и снятие существующих ролей остаются доступными. Сырые внутренние ошибки
наружу НЕ показываются.

#### Scenario: Каталог недоступен

- **GIVEN** `/iam/directory/subjects` отвечает `503`
- **WHEN** пользователь ищет в пикере
- **THEN** показана индикация «каталог недоступен», поле поиска остаётся, просмотр и
  снятие существующих ролей работают

#### Scenario: Нет права на справочник → 403

- **GIVEN** у субъекта нет `(read, iam:directory)`
- **WHEN** открыт раздел «Роли и доступы»
- **THEN** пикер/поиск скрыт и заменён сообщением об отказе, имена/почта не
  показываются, просмотр и снятие существующих ролей работают
