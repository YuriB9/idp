## MODIFIED Requirements

### Requirement: Форма создания сервиса

Портал ДОЛЖЕН (MUST) предоставлять форму создания сервиса на react-hook-form,
где источником правды валидации входных данных служит zod-схема из кодогена.
Форма ДОЛЖНА (MUST) содержать поле имени и ОБЯЗАТЕЛЬНОЕ поле владельцев (минимум
один владелец), используя для разбора ввода тот же `parseOwners` (разделители —
перевод строки и запятая, trim/dedup/sort), что и карточка владельцев. Отправка
формы ДОЛЖНА (MUST) вызывать `POST /api/projects/{project}/services` с телом
`{name, owners}` и по успеху переводить пользователя на экран прогресса
создаваемого сервиса. Подписи шагов единого прогресса создания
(`OPERATION_STEPS.create`) ДОЛЖНЫ (MUST) соответствовать порядку активностей
воркфлоу создания (вариант B), включая шаги назначения владельцев в
GitLab/Vault/IDM. Форма МОЖЕТ (MAY) предлагать текущего пользователя как дефолтное
значение владельцев (UX), но НЕ ДОЛЖНА (MUST NOT) делать его обязательным или
добавлять неявно при отправке.

#### Scenario: Успешная отправка с владельцами

- **GIVEN** заполнено валидное имя сервиса и хотя бы один владелец
- **WHEN** пользователь отправляет форму
- **THEN** портал шлёт `POST` с `{name, owners}`, получает `id`/`status:"creating"`
  и переходит на экран прогресса

#### Scenario: Клиентская валидация имени и владельцев

- **GIVEN** имя пустое/не проходит zod-схему ИЛИ владельцы пусты после разбора
- **WHEN** пользователь пытается отправить форму
- **THEN** форма показывает ошибку валидации соответствующего поля и НЕ шлёт запрос

#### Scenario: Конфликт имени с сервера

- **GIVEN** периметр вернул 409 (имя занято)
- **WHEN** обрабатывается ответ
- **THEN** форма показывает понятное сообщение о конфликте без раскрытия
  внутренних деталей сервера

### Requirement: Клиентская модель шагов Temporal-воркфлоу (вариант B)

Портал ДОЛЖЕН (MUST) держать статическую упорядоченную модель шагов по операции
(`OPERATION_STEPS`) в чистом модуле без побочных эффектов, где порядок и русские
подписи соответствуют порядку активностей соответствующего воркфлоу `projects`.
Поскольку периметр отдаёт только грубый статус сервиса (без per-activity
прогресса), фаза операции выводится клиентски из статуса и доменных полей, а шаги
показываются ОДНОЙ фазой (без ложного указания, какой именно шаг идёт/упал). Шаги
операции `create` ДОЛЖНЫ (MUST) отражать вариант B воркфлоу создания: создание
репозитория GitLab → назначение владельцев в GitLab → создание проекта образов
Harbor → настройка секретов Vault → выдача доступа владельцам в Vault → инъекция
секретов в CI → активация сервиса → назначение ролей владельцев в IDM.

#### Scenario: Шаги создания отражают назначение владельцев

- **WHEN** инспектируется `OPERATION_STEPS.create`
- **THEN** список шагов включает назначение владельцев в GitLab, выдачу доступа
  владельцам в Vault и назначение ролей владельцев в IDM в порядке активностей
  воркфлоу создания

#### Scenario: Фаза создания выводится из грубого статуса

- **GIVEN** статус сервиса `creating`/`active`/`failed`
- **WHEN** разрешается прогресс операции создания
- **THEN** все шаги создания показываются единой фазой: `creating`→running,
  `active`→done, `failed`→failed
