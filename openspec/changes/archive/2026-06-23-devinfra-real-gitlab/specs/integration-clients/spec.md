## ADDED Requirements

### Requirement: Исходящая аутентификация запросов к реальному GitLab

Все исходящие HTTP-запросы GitLab-клиента к реальному GitLab ДОЛЖНЫ (MUST) нести
аутентификацию заголовком `PRIVATE-TOKEN` со значением из конфигурации
(`integrations.Config.GitLabToken`, env `GITLAB_TOKEN`). Токен НЕ ДОЛЖЕН (MUST NOT)
логироваться. Запросы к Vault и Harbor этой аутентификацией НЕ затрагиваются.

#### Scenario: GitLab-запрос несёт токен

- **GIVEN** сконфигурированный `GITLAB_TOKEN`
- **WHEN** GitLab-клиент выполняет любой запрос к GitLab API
- **THEN** запрос содержит заголовок `PRIVATE-TOKEN` с этим значением, и значение токена не
  попадает в логи

#### Scenario: Токен не применяется к другим интеграциям

- **WHEN** клиент Vault или Harbor выполняет запрос
- **THEN** заголовок `PRIVATE-TOKEN` GitLab не добавляется к этим запросам

### Requirement: Резолвинг namespace и владельца в идентификаторы GitLab

GitLab-клиент ДОЛЖЕН (MUST) резолвить путь группы (проект каталога) в числовой `namespace_id`
и логин владельца в числовой `user_id` через GitLab API, и использовать числовые
идентификаторы в телах запросов (`namespace_id` при создании репозитория, `user_id` при
синхронизации участников). Результаты резолвинга МОГУТ (MAY) кэшироваться в пределах процесса.
Если маппинг владельца на пользователя GitLab не задан, синхронизация этого участника ДОЛЖНА
(MUST) безопасно пропускаться (не приводить к ошибке воркфлоу).

#### Scenario: Создание репозитория использует namespace_id

- **GIVEN** предсозданная группа с известным путём
- **WHEN** клиент создаёт репозиторий в этом namespace
- **THEN** путь группы резолвится в `namespace_id`, и создание шлёт `namespace_id` (число), а
  не строку namespace

#### Scenario: Синхронизация участников использует user_id

- **GIVEN** владелец, отображённый на существующего пользователя GitLab
- **WHEN** клиент добавляет участника в репозиторий
- **THEN** логин резолвится в `user_id`, и запрос шлёт `user_id` (число) с числовым
  `access_level`

#### Scenario: Незаданный маппинг владельца пропускается

- **GIVEN** владелец без отображения на пользователя GitLab
- **WHEN** выполняется синхронизация участников
- **THEN** этот владелец безопасно пропускается без ошибки воркфлоу

### Requirement: Выбор реализации GitLab-клиента по конфигурации

`devinfra-worker` ДОЛЖЕН (MUST) выбирать реализацию GitLab-клиента по конфигурации: при
заданном `GITLAB_TOKEN` собирается клиент против реального GitLab (с аутентификацией и
резолвингом идентификаторов); иначе сохраняется поведение по умолчанию (клиент против
WireMock-моков). In-memory стаб ДОЛЖЕН (MUST) оставаться доступным для дефолтного прогона
тестов без сети. Vault и Harbor ОСТАЮТСЯ моками независимо от выбора GitLab-клиента.

#### Scenario: Без токена — поведение по умолчанию

- **WHEN** `GITLAB_TOKEN` не задан
- **THEN** `devinfra-worker` использует GitLab-клиент против моков, как прежде, без обращения
  к реальному GitLab

#### Scenario: С токеном — реальный GitLab

- **WHEN** `GITLAB_TOKEN` задан (профиль реального GitLab)
- **THEN** `devinfra-worker` использует GitLab-клиент с аутентификацией и резолвингом
  идентификаторов против реального GitLab

## MODIFIED Requirements

### Requirement: Идемпотентность операций провизии и компенсации

Операции клиентов интеграций и их компенсации ДОЛЖНЫ (MUST) быть идемпотентны (создание
репозитория, директории, Robot Account, политик, AppRole, инъекция переменных). Для GitLab
идемпотентность ДОЛЖНА (MUST) опираться на коды ответов реального GitLab и при необходимости
на предварительную проверку существования (GET-then-act), а НЕ на разбор текста сообщений об
ошибке (текст не контракт и зависит от версии): повторное создание уже существующего
репозитория ДОЛЖНО (MUST) трактоваться как успех (no-op), повторное удаление отсутствующего
(`404`/`202`) — как успех, повторная архивация/перенос в уже целевое состояние — как успех.
Повторный вызов операции провизии при уже существующем ресурсе НЕ ДОЛЖЕН (MUST NOT) приводить
к ошибке или дубликату; повторный вызов компенсации при уже удалённом ресурсе ДОЛЖЕН (MUST)
считаться успешным (no-op).

#### Scenario: Повторная провизия существующего ресурса

- **GIVEN** GitLab-репозиторий, уже созданный предыдущей попыткой
- **WHEN** операция создания вызывается повторно
- **THEN** клиент по предварительной проверке существования возвращает успех без создания
  дубликата и без зависимости от текста ошибки GitLab

#### Scenario: Повторная компенсация удалённого ресурса

- **GIVEN** Harbor-директория, уже удалённая компенсацией
- **WHEN** компенсация удаления вызывается повторно
- **THEN** операция завершается успешно как no-op (ресурс отсутствует — это не ошибка)

#### Scenario: Повторный перенос в уже целевую группу

- **GIVEN** репозиторий, уже находящийся в целевой группе
- **WHEN** операция переноса вызывается повторно
- **THEN** клиент трактует это как успех (no-op) по проверке текущего namespace, не возвращая
  ошибку

### Requirement: Инъекция секретов в CI/CD-переменные GitLab

После создания секретов в Vault и Robot Account в Harbor клиент интеграции ДОЛЖЕН (MUST)
инъектировать выданные токены (Vault AppRole RoleID/SecretID и Harbor Robot-токен) в CI/CD-
переменные GitLab-репозитория. Секреты НЕ ДОЛЖНЫ (MUST NOT) логироваться в открытом виде.
Операция инъекции ДОЛЖНА (MUST) быть идемпотентна через upsert переменной (создание или
перезапись по ключу), совместимый с реальным GitLab, не создавая дубликатов и не завися от
текста ошибки «переменная существует».

#### Scenario: Токены попадают в переменные репозитория

- **GIVEN** созданные Vault AppRole и Harbor Robot Account
- **WHEN** выполняется инъекция секретов
- **THEN** в CI/CD-переменных GitLab-репозитория появляются соответствующие значения, и сами
  значения не пишутся в логи в открытом виде

#### Scenario: Повторная инъекция идемпотентна

- **GIVEN** переменные уже установлены предыдущей попыткой
- **WHEN** инъекция выполняется повторно
- **THEN** переменные перезаписываются (upsert) без ошибки и без дублирования
