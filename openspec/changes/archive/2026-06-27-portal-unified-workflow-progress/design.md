## Context

Портал управляет сервисами через четыре асинхронные операции, каждая = запуск
Temporal-воркфлоу в сервисе projects с фиксированным порядком активностей (источник
«шагов от темпорала»):

- **Создание** (`CreateServiceWorkflow`): GitLab create repo → Harbor create →
  Vault setup → инъекция секретов → переход `active`. Статус: `creating → active|failed`.
- **Смена владельцев** (`ChangeOwnersWorkflow`): GitLab sync members → Vault sync
  owners → запись каталога (`owners_version++`) → IDM sync owner roles. Статус сервиса
  НЕ меняется (остаётся `active`); сигнал завершения — рост `owners_version`. Одна
  успешная смена на сервис (см. память e2e-change-owners-single-shot).
- **Decommission** (`DecommissionWorkflow`): ensure load drained → GitLab archive →
  Harbor read-only → Vault revoke secret-id → запись каталога. Статус:
  `active → decommissioned|failed` (промежуточного статуса НЕТ).
- **Transfer** (`TransferServiceWorkflow`): catalog begin (`active→transferring`) →
  GitLab transfer repo → Vault migrate paths → Harbor metadata → catalog commit
  (`transferring→active` в target) → перенос ролей IDM. Статус:
  `active → transferring → active|failed`.

Текущий UX рассогласован: создание показывает однострочную панель на
`ServiceProgressPage`; остальные операции — мутация + тост успеха/ошибки в углу
(`OwnersCard`/`DecommissionCard`/`TransferCard`). Контракт периметра отдаёт только
ГРУБЫЙ статус (`ServiceStatus: [creating, active, decommissioned, failed,
transferring]`) + `owners_version` + `decommissioned_at`; per-activity прогресса
Temporal в контракте НЕТ. Дизайн-система (ADR-0017) уже даёт toast/ConfirmDialog/
Dialog/StatusBadge и единый маппинг ошибок (`lib/errors`), но НЕ имеет примитива
ступенчатого прогресса.

Ограничения: комментарии в коде — только русский; ветка
`change/portal-unified-workflow-progress` от master; зелёный CI (включая `gen:check`,
`web-test`); доступность ADR-0017 (aria-live/role, motion-safe, без сырых ошибок);
поллинг без `sleep` (ADR-0018).

## Goals / Non-Goals

**Goals:**

- Переиспользуемый примитив ступенчатого прогресса (`ui/stepper.tsx`) с состояниями
  `pending|running|done|failed`, доступностью и motion-safe-анимациями.
- Хук поллинга статуса (`hooks/useServiceStatus`), инкапсулирующий нынешнюю логику
  `ServiceProgressPage` (react-query, `refetchInterval` до терминала, без `sleep`).
- Клиентская модель шагов на операцию (порядок = порядок активностей воркфлоу),
  current-фаза выводится из грубого статуса + доменных полей.
- Единый ступенчатый прогресс для ВСЕХ операций; тост — только для синхронных ошибок
  запуска (4xx/валидация).
- Корректное отражение особых случаев: вырожденный степпер для смены владельцев;
  пометка точки невозврата для decommission/transfer; `failed` показывает факт
  Saga-отката.

**Non-Goals:**

- Замена поллинга на SSE/WebSocket (остаётся поллинг).
- Изменение семантики воркфлоу/активностей и реальных бэкендов DevInfra
  (ADR-0019/0020/0021) — без изменений.
- Истинная per-activity гранулярность Temporal в UI (требует варианта A — см. ниже).
- Изменение контракта периметра, кодогена, `openapi.yaml`.
- Редизайн остального портала; Kubernetes/прод.

## Decisions

### Решение 1 (КЛЮЧЕВОЕ): источник шагов — вариант B (вывод из грубого статуса), не A

**Выбор: B.** Шаги — фиксированная клиентская модель на операцию (известный порядок
активностей), текущая фаза выводится из `status` + доменных полей; контракт НЕ
меняется.

**Почему B, а не A (реальные per-activity шаги Temporal через новое поле/эндпоинт
периметра):**

- *Цена A непропорциональна цели.* Цель задачи — УНИФИЦИРОВАТЬ UX (один компонент
  вместо панель+тост). Это достигается полностью на фронте. A требует:
  путь данных worker → projects (репорт прогресса в каталог ИЛИ Temporal Query/
  DescribeWorkflowExecution) → новое поле `steps[]` в openapi → чистая перегенерация
  кодогена (openapi-typescript/zod) → бэкенд+контрактные+e2e-тесты. Это backend+contract
  работа, пересекающая Saga-откат (ADR-0005), guarded-CAS (ADR-0004), split воркфлоу
  (ADR-0008), точки невозврата (ADR-0012/0013).
- *`gen:check` остаётся зелёным без перегенерации* — снимается главный риск регрессии
  CI. A менял бы `gen:check` (требует чистой перегенерации, без ручной правки
  `web/src/api/*` и `web/public/openapi.yaml`).
- *Быстрый инкремент* в серии завершённых работ; A — естественное продолжение
  отдельной задачей, когда понадобится истинная гранулярность.

**Честная цена B (фиксируется в spec и UI, не маскируется):** периметр несёт только
грубый статус, поэтому B НЕ может правдиво показать, какая именно активность сейчас
идёт или какая упала. Для создания и transfer есть промежуточные статусы
(`creating`/`transferring`) → можно различить «идёт/завершено/упало» на уровне фазы.
Для decommission и смены владельцев промежуточного статуса НЕТ (статус не меняется до
терминала) → степпер для них показывает «выполняется → завершено/ошибка» без
посегментной разбивки. UI честно отражает это: пока идёт — шаги в состоянии `running`
(без ложного `done` на конкретных), на терминале — все `done` либо операция `failed`
с сообщением об откате.

**Если в будущем выбираем A** (вне этой задачи): минимальное расширение — поле
`steps[]` элементами `{key, label, state: pending|running|done|failed}` в
`ServiceSummary` (или отдельный under-status-эндпоинт), наполняемое воркером (heartbeat
с прогрессом в каталог ИЛИ запрос Temporal из projects), затем чистая перегенерация
кодогена. Saga-откат отражается пометкой `failed` на упавшем шаге + `done`-компенсации.

### Решение 2: примитив `ui/stepper.tsx` — презентационный, управляемый пропсами

Компонент принимает `steps: { key; label; state }[]` и не знает про поллинг/домен —
только отрисовка. Состояние → цвет+иконка по образцу `StatusBadge` (done: CheckCircle2/
success; running: Loader2/animate-spin под motion-safe; failed: XCircle/destructive;
pending: muted). Контейнер: `role="list"` + обёртка с `aria-live="polite"`, чтобы смена
текущего шага объявлялась без перехвата фокуса. Анимация спиннера — через утилиту
motion-safe (Tailwind `motion-safe:animate-spin`), уважая `prefers-reduced-motion`.
Опциональный флаг `irreversible` (пометка точки невозврата) рисует предупреждающую
подпись. Альтернатива (степпер сам поллит) отвергнута: смешивает данные и презентацию,
мешает тестируемости.

### Решение 3: хук `useServiceStatus(project, name)` — единый поллинг

Выносит текущую логику `ServiceProgressPage` (queryKey `["service", project, name]`,
`refetchInterval` → `false` на терминале `active|failed|decommissioned`, иначе
`POLL_INTERVAL_MS=1500`). Возвращает `{ data, status, isLoading, isError }`. НЕ
использует `sleep`. Переиспользуется страницей и всеми карточками. Инвалидация
`["service", ...]` после успешного запуска мутации запускает поллинг до нового терминала.

### Решение 4: модель шагов `lib/workflow-steps.ts` — чистая функция

Чистая функция на операцию: `buildSteps(operation, status, domain) → Step[]`. Порядок и
русские подписи зашиты по активностям воркфлоу (справочник —
`services/projects/*`). Маппинг текущей фазы:

- `create`: `creating` → все шаги `running`; `active` → все `done`; `failed` → операция
  `failed` (+ сообщение об откате).
- `transfer`: `active` (до запуска) → pending; `transferring` → шаги `running`;
  `active` (после) → `done`; `failed` → `failed`.
- `decommission`: запущено и `active` → `running`; `decommissioned` → `done`;
  `failed` → `failed`.
- `change-owners`: запущено → `running`; завершение по росту `owners_version`/успеху
  мутации → `done` (вырожденный одно-/коротко-шаговый степпер).

Чистая функция тривиально тестируется (вход → ожидаемые состояния).

### Решение 5: судьба тостов — тост только для синхронных ошибок запуска

Тост (ADR-0017) остаётся для мгновенных 4xx/валидации при ЗАПУСКЕ операции (через
`lib/errors`, с существующими override-сообщениями 403/409/422). Успех-тосты
(`toast.success`) в `Owners/Decommission/Transfer Card` УБИРАЮТСЯ: ход и исход
асинхронной операции показывает степпер на странице сервиса. После принятой мутации
карточка инвалидирует `["service", ...]`, страница (через хук) поднимает степпер
соответствующей операции до терминала. Альтернатива (полностью убрать тосты) отвергнута:
синхронный отказ прав/конфликта до старта воркфлоу нужно показать немедленно, степпер
для этого не подходит (асинхронная операция не началась).

### Идемпотентность и RetryPolicy (со стороны воркфлоу — без изменений)

Изменение чисто клиентское; воркфлоу не трогаются. Для полноты: детерминированные
`WorkflowID` (`provisioning/transfer/decommission/changeowners.WorkflowID(project,
name)`) и RetryPolicy/таймауты активностей уже заданы в `services/projects/*`
(ADR-0001/0005). Клиент опирается на guarded-CAS-статусы (ADR-0004) как на источник
истины и лишь читает их поллингом; повторный запуск/двойной клик защищён на стороне
сервиса (CAS → 409, показывается тостом).

### Поток данных (sequence)

```
Пользователь        Card (mutation)      Периметр/projects        Temporal worker        Страница (hook+stepper)
   |  подтвердил        |                      |                        |                        |
   |------------------->| POST .../op          |                        |                        |
   |                    |--------------------->| guarded-CAS статуса     |                        |
   |                    |   202/200 | 4xx      |   старт воркфлоу ------->| activity1..N (GitLab/  |
   |                    |<---------------------|                        |  Vault/Harbor)         |
   |  4xx -> toast      |  invalidate(service) |                        |  переходы статуса CAS   |
   |                    |---------------------------------------------------------------------->| poll GetService
   |                    |                      |                        |                        |  buildSteps(status)
   |                    |                      |                        |  active/failed/decomm   |  -> Stepper(running.. )
   |<------------------ ступенчатый прогресс до терминала -------------------------------------- |  терминал -> done/failed
```

## Risks / Trade-offs

- [Грубая гранулярность B: нельзя показать конкретный идущий/упавший шаг] → Честно
  отражаем в UI и spec; для create/transfer используем промежуточные статусы как
  максимум доступной фазовости; путь к A задокументирован для будущей задачи.
- [`failed` без атрибуции шага вводит в заблуждение «всё упало»] → Формулировка
  сообщения: «операция не завершилась — выполнен откат (Saga)», без ложного указания
  конкретного шага; опираемся на `lib/errors`.
- [Смена владельцев не меняет статус → трудно поймать «завершено»] → Сигнал завершения
  — рост `owners_version` в ответе поллинга (и/или успешная мутация); вырожденный
  степпер не претендует на посегментность. Учитываем single-shot (вторая смена
  маскируется периметром как 200 no-op).
- [Двойной запуск/гонка при поллинге] → guarded-CAS на сервере (ADR-0004) вернёт 409 на
  конкурентный запуск (тост); поллинг идемпотентен (только чтение).
- [Регрессия CI `gen:check`] → Вариант B не трогает контракт/кодоген — риск снят;
  `openapi.yaml`, `web/src/api/*`, `web/public/openapi.yaml` не редактируются.
- [Доступность: спам объявлений aria-live при поллинге каждые 1.5с] → Объявляем только
  смену состояния шага (мемоизация), не каждый ответ поллинга; `aria-live="polite"`.

## Migration Plan

Чисто аддитивно во фронте, обратной несовместимости нет:

1. Ветка `change/portal-unified-workflow-progress` от master.
2. Добавить `ui/stepper.tsx`, `hooks/useServiceStatus.ts`, `lib/workflow-steps.ts` (+ их
   тесты).
3. Рефактор `ServiceProgressPage` на хук + степпер создания; затем `Owners/Decommission/
   TransferCard` — убрать success-тост, поднять степпер операции через инвалидацию;
   `CreateServicePage` без изменений потока (навигация на страницу прогресса).
4. Обновить затронутые `*.test.tsx` под новый UX.
5. Зелёный CI (включая `gen:check` без перегенерации, `web-test`). Merge → отдельным PR
   `/opsx:archive` (sync+archive, по образцу #59/#67/#69).

Откат: revert PR (контракт/бэкенд не затронуты, миграций данных нет).
