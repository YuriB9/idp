## ADDED Requirements

### Requirement: Исходящая аутентификация запросов к реальному Harbor

Все исходящие HTTP-запросы Harbor-клиента к реальному Harbor ДОЛЖНЫ (MUST) нести аутентификацию
HTTP Basic заголовком `Authorization: Basic <base64(user:pass)>` со значениями из конфигурации
(`integrations.Config.HarborUsername`/`HarborPassword`, env `HARBOR_USERNAME`/`HARBOR_PASSWORD` или
`HARBOR_PASSWORD_FILE`). Пароль НЕ ДОЛЖЕН (MUST NOT) логироваться. Запросы к GitLab и Vault этой
аутентификацией НЕ затрагиваются (заголовок `Authorization` Harbor не протекает на другие
интеграции).

#### Scenario: Harbor-запрос несёт Basic-аутентификацию

- **GIVEN** сконфигурированные `HARBOR_USERNAME`/`HARBOR_PASSWORD`
- **WHEN** Harbor-клиент выполняет любой запрос к Harbor API
- **THEN** запрос содержит заголовок `Authorization: Basic` с этими значениями, и пароль не попадает
  в логи

#### Scenario: Креденшелы не применяются к другим интеграциям

- **WHEN** клиент GitLab или Vault выполняет запрос
- **THEN** заголовок `Authorization` Harbor не добавляется к этим запросам

### Requirement: Семантика реального Harbor v2.0 API в Harbor-клиенте

Harbor-клиент в реальном режиме ДОЛЖЕН (MUST) использовать семантику реального Harbor v2.0 API:
создание проекта (`POST /api/v2.0/projects` с `project_name`); создание robot-аккаунта
(`POST /api/v2.0/robots` с `level`/`permissions[]`/`duration`) с забором ФАКТИЧЕСКИХ `name` и
`secret` из ответа (а НЕ сконструированных) — именно они инъектируются в CI/CD-переменные GitLab;
резолвинг robot-аккаунта по ЧИСЛОВОМУ `id` для удаления (`DELETE /api/v2.0/robots/<id>` — поиск id
по имени через список, а НЕ обращение по имени); при необходимости резолвинг проекта по имени → `id`
(или заголовок `X-Is-Resource-Name`). Идемпотентность ДОЛЖНА (MUST) опираться на коды ответов Harbor
и при необходимости на предварительную проверку существования (GET-then-act), а НЕ на разбор текста
ошибок: повторное создание (`409`) и удаление отсутствующего (`404`) трактуются согласованно с кодами
реального Harbor. Клиент НЕ ДОЛЖЕН (MUST NOT) светить сырые тексты ошибок Harbor как контракт.

#### Scenario: Создание проекта и robot возвращает фактический секрет

- **GIVEN** реальный Harbor с admin-доступом
- **WHEN** воркфлоу провизии создаёт проект и robot-аккаунт
- **THEN** клиент создаёт проект, создаёт robot с телом v2.0 (`level`/`permissions[]`/`duration`) и
  возвращает ФАКТИЧЕСКИ выданные Harbor `name` и `secret` робота для инъекции в GitLab-переменные

#### Scenario: Удаление robot по числовому id

- **GIVEN** созданный robot-аккаунт проекта
- **WHEN** клиент отзывает/удаляет robot
- **THEN** клиент резолвит числовой `id` робота по имени (через список) и удаляет
  `DELETE /api/v2.0/robots/<id>`, а не обращается к роботу по имени

#### Scenario: Повторное создание и удаление отсутствующего идемпотентны

- **WHEN** проект/robot создаётся повторно либо удаляется отсутствующий проект/robot
- **THEN** операция трактуется как успех по кодам реального Harbor (`409` при повторном создании,
  `404` при удалении отсутствующего) без зависимости от текста ошибки

### Requirement: Семантика read-only проекта Harbor через отзыв robot

Операция перевода проекта Harbor в read-only (`SetReadOnly`, decommission, ADR-0012) ДОЛЖНА (MUST)
делать проект недоступным на запись через ОТЗЫВ robot-аккаунта (в Harbor нет project-level
`read_only` в метаданных проекта), а НЕ через несуществующее поле `read_only`. Результат ДОЛЖЕН
(MUST) быть наблюдаем через Harbor API (robot-аккаунт проекта отсутствует/отозван). Операция ДОЛЖНА
(MUST) быть идемпотентна (повторный отзыв/`404` — успех). Компенсация `SetWritable` восстанавливает
доступ воссозданием robot-аккаунта.

#### Scenario: Отзып robot делает проект недоступным на запись

- **GIVEN** проект с активным robot-аккаунтом
- **WHEN** вызывается перевод проекта в read-only
- **THEN** клиент отзывает robot-аккаунт проекта (по числовому id), после чего robot отсутствует и
  проект недоступен на запись, а сам проект сохраняется

#### Scenario: Повторный отзыв идемпотентен

- **GIVEN** проект с уже отозванным robot
- **WHEN** перевод в read-only вызывается повторно
- **THEN** операция завершается успешно как no-op (`404`/отсутствие robot — не ошибка)

### Requirement: Выбор реализации Harbor-клиента по конфигурации

`devinfra-worker` ДОЛЖЕН (MUST) выбирать реализацию Harbor-клиента по конфигурации: при заданных
`HARBOR_USERNAME`/`HARBOR_PASSWORD` (или `HARBOR_PASSWORD_FILE`) собирается клиент против реального
Harbor (с HTTP Basic и семантикой реального API); иначе сохраняется поведение по умолчанию (клиент
против WireMock-мока). In-memory стаб ДОЛЖЕН (MUST) оставаться доступным для дефолтного прогона
тестов без сети. GitLab и Vault сохраняют свой выбор реализации независимо от выбора Harbor-клиента.

#### Scenario: Без креденшелов — поведение по умолчанию

- **WHEN** `HARBOR_USERNAME`/`HARBOR_PASSWORD`/`HARBOR_PASSWORD_FILE` не заданы
- **THEN** `devinfra-worker` использует Harbor-клиент против мока, как прежде, без обращения к
  реальному Harbor

#### Scenario: С креденшелами — реальный Harbor

- **WHEN** `HARBOR_USERNAME`/`HARBOR_PASSWORD` заданы (профиль реального Harbor)
- **THEN** `devinfra-worker` использует Harbor-клиент с HTTP Basic и семантикой реального API против
  реального Harbor
