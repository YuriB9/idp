## ADDED Requirements

### Requirement: Реальный Harbor в отдельном compose-профиле

Стенд ДОЛЖЕН (MUST) предоставлять реальный Harbor как ОТДЕЛЬНЫЙ compose-override
(`deploy/compose/docker-compose.harbor.yml`), подключаемый поверх базовой локалки и НЕ входящий в
дефолтный `docker compose up`. Подключение профиля НЕ ДОЛЖНО (MUST NOT) ломать базовую локалку
(`docker-compose.yml`), e2e-override (`docker-compose.e2e.yml`), gitlab-профиль
(`docker-compose.gitlab.yml`), vault-профиль (`docker-compose.vault.yml`) и conformance-таргет.
Образы Harbor ДОЛЖНЫ (MUST) быть пиннуты по тегу версии. При активном профиле `devinfra-worker`
ДОЛЖЕН (MUST) направлять Harbor-вызовы на реальный Harbor, а GitLab и Vault ОСТАЮТСЯ моками (если их
собственные профили не активированы).

#### Scenario: Дефолтная локалка не поднимает реальный Harbor

- **WHEN** выполняется `docker compose -f deploy/compose/docker-compose.yml up`
- **THEN** реальный Harbor не запускается, а `devinfra-worker` ходит к WireMock-моку Harbor

#### Scenario: Профиль реального Harbor активируется отдельно

- **WHEN** стенд поднимается с override `docker-compose.harbor.yml`
- **THEN** запускается реальный Harbor (связка контейнеров, пиннутая версия), а `devinfra-worker`
  направляет Harbor-вызовы на него, оставляя GitLab и Vault моками

### Requirement: Health-gate и бюджет ожидания готовности Harbor

Сервис Harbor в compose ДОЛЖЕН (MUST) иметь health-gate (`GET /api/v2.0/health` или
`/api/v2.0/ping`), а зависящие сервисы/тесты ДОЛЖНЫ (MUST) дожидаться его готовности
ретраи-поллингом без фиксированных `sleep`. Бюджет ожидания ДОЛЖЕН (MUST) быть настраиваемым
(переменная окружения) с разумным дефолтом; поскольку Harbor — связка контейнеров с тяжёлым стартом
(зеркалит GitLab), дефолт ДОЛЖЕН (MUST) быть соразмерен бюджету GitLab, а не Vault.

#### Scenario: Тесты ждут готовности Harbor без sleep

- **GIVEN** поднимаемый реальный Harbor, ещё не прошедший инициализацию
- **WHEN** запускается интеграционный набор
- **THEN** набор дожидается готовности Harbor ретраи-поллингом в пределах настраиваемого бюджета и не
  использует фиксированных `sleep`-констант

### Requirement: Детерминированный сид Harbor для тестов

Стенд ДОЛЖЕН (MUST) детерминированно засевать реальный Harbor: admin-пароль — фиксированное значение
фикстуры стенда (`HARBOR_ADMIN_PASSWORD`, дефолт `Harbor12345`), при необходимости базовые сущности.
`devinfra-worker` ДОЛЖЕН (MUST) аутентифицироваться ИМЕННО этим admin-логином/паролем
(`HARBOR_USERNAME`/`HARBOR_PASSWORD`). Пароль НЕ ДОЛЖЕН (MUST NOT) хардкодиться в коде сверх фикстур
стенда. Per-service проекты и robot-аккаунты создаёт САМ клиент в воркфлоу провизии, а НЕ сид.

#### Scenario: Admin-доступ готов до провизии

- **GIVEN** засеянный реальный Harbor
- **WHEN** воркфлоу создаёт проект и robot для сервиса (`/api/v2.0/projects`, `/api/v2.0/robots`)
- **THEN** запросы успешны, потому что admin-креденшелы — известная фикстура стенда, и клиент несёт
  HTTP Basic admin

#### Scenario: Креденшелы сидируются детерминированно и переиспользуются worker

- **GIVEN** стенд с реальным Harbor
- **WHEN** Harbor стартует с `HARBOR_ADMIN_PASSWORD` и стартует `devinfra-worker`
- **THEN** `devinfra-worker` аутентифицируется этим же значением (`HARBOR_PASSWORD`/`Authorization:
  Basic`)

### Requirement: Интеграционные тесты воркфлоу против реального Harbor

ДОЛЖЕН (MUST) существовать интеграционный набор (build-тег `integration`), переиспользующий харнесс
`tests/e2e` (получение OIDC-токена, HTTP-клиент периметра, ожидание статусов без sleep, уникальные
имена), который гоняет воркфлоу через периметр и ассертит ФАКТИЧЕСКОЕ состояние в реальном Harbor
через Harbor API. Набор ДОЛЖЕН (MUST) активироваться только при заданной конфигурации реального
Harbor (отдельный gate-env, например `HARBOR_ADDR`/`HARBOR_USERNAME`/`HARBOR_PASSWORD`) и НЕ ДОЛЖЕН
(MUST NOT) влиять на прогон сквозных историй против моков (`make e2e`) и на gitlab/vault-наборы.
Прогон ДОЛЖЕН (MUST) быть только локальным ручным (не в CI). Набор НЕ ДОЛЖЕН (MUST NOT) ассертить
сырые тексты ошибок Harbor как контракт.

#### Scenario: Создание реально создаёт проект и robot

- **GIVEN** активный стенд с реальным Harbor
- **WHEN** воркфлоу создания услуги доходит до `active`
- **THEN** через Harbor API подтверждается, что проект существует, robot-аккаунт создан и секрет
  робота выдан

#### Scenario: Decommission делает проект недоступным на запись

- **GIVEN** активная услуга с созданным проектом и robot
- **WHEN** воркфлоу decommission доходит до `decommissioned`
- **THEN** через Harbor API подтверждается, что robot-аккаунт отозван (проект недоступен на запись),
  а проект сохраняется

#### Scenario: Transfer обновляет метаданные проекта

- **GIVEN** активная услуга с созданным проектом
- **WHEN** воркфлоу переноса доходит до `active`
- **THEN** через Harbor API подтверждается, что наблюдаемое поле метаданных проекта обновлено под
  целевой проект

#### Scenario: Набор пропускается без конфигурации реального Harbor

- **WHEN** интеграционный набор запускается без gate-env реального Harbor
- **THEN** тесты против реального Harbor пропускаются (skip), а прогон историй против моков
  (`make e2e`) не затрагивается

### Requirement: Makefile-цели локального прогона стенда с Harbor

Makefile ДОЛЖЕН (MUST) предоставлять цели подъёма стенда с реальным Harbor, сидирования, прогона
интеграционного набора и гашения стенда с очисткой томов (`harbor-up`/`harbor-seed`/`harbor-test`/
`harbor-down`/`harbor`) по образцу `gitlab-*`/`vault-*`. Цели ДОЛЖНЫ (MUST) выполнять прогон только
локально и НЕ ДОЛЖНЫ (MUST NOT) поднимать реальный Harbor в CI.

#### Scenario: Полный локальный цикл

- **WHEN** оператор выполняет цель полного цикла (подъём → сид → прогон → гашение)
- **THEN** стенд с реальным Harbor поднимается и сидируется, интеграционный набор прогоняется, а
  стенд гарантированно гасится с очисткой томов даже при падении тестов
