## Context

Создание сервиса сегодня двухфазное: `CreateService` вставляет запись каталога со
`status=CREATING` БЕЗ владельцев (`services/projects/internal/usecase/usecase.go`),
затем стартует `CreateServiceWorkflow` (`services/projects/provisioning`),
который провизионирует GitLab → Harbor → Vault → инъекцию секретов → активацию,
НЕ устанавливая роли владельцев. Владельцы появляются только позже, отдельной
операцией `SetServiceOwners` → `ChangeOwnersWorkflow`
(`services/projects/changeowners`): GitLab members → Vault policies → запись в
каталог (точка невозврата) → IDM owner roles, с Saga-компенсациями. Контракт
создания двойной: proto `CreateServiceRequest{project, name}` и openapi
`CreateServiceRequest required:[name]`. Колонка/таблица владельцев уже есть
(`service_owners`, `owners_version`), нормализация `normalizeOwners` и activities
синхронизации владельцев уже реализованы и переиспользуемы.

Задача — сделать владельцев обязательной частью создания и устанавливать их
атомарно, устранив окно «бесхозного» сервиса. Выбран **вариант B** (полное
установление владельца), см. решение D1.

Ограничения (обязательные): guarded-CAS статусов (ADR-0004), Saga-откат
(ADR-0005), split воркфлоу (ADR-0008), форма периметра без сырых ошибок
(ADR-0009), контракт владельцев/role-sync (ADR-0011), единый прогресс (ADR-0022),
комментарии в коде только на русском, чистая перегенерация кодогена (`gen:check`).

## Goals / Non-Goals

**Goals:**
- Обязательные владельцы в `CreateServiceRequest` на обоих контрактах (proto +
  openapi), чистая перегенерация buf + web, зелёный `gen:check`.
- Атомарная запись владельцев в каталог при вставке записи (`owners_version=1`),
  инвариант «нет записи без владельца».
- Реальное установление ролей владельца в воркфлоу создания (GitLab members,
  Vault policies, IDM owner roles) с Saga-компенсацией, переиспользуя activities
  `changeowners`.
- Синхронная валидация пустых владельцев на периметре → 400 без сырых деталей.
- Форма создания с обязательным полем владельцев, zod из кодогена, единый
  прогресс с обновлёнными подписями шагов.

**Non-Goals:**
- Изменение семантики ChangeOwners/Transfer/Decommission сверх необходимого.
- Проверка существования субъектов-владельцев в справочнике IDM при создании
  (держим консистентность с ChangeOwners — без существ. проверки в MVP, см. D4).
- Неявное авто-добавление создателя во владельцы (см. D5).
- SSE/WebSocket, редизайн портала сверх формы создания, K8s/прод, миграция
  исторических записей.

## Decisions

### D1. Вариант B (полное установление владельца) вместо A (только каталог)

**Выбор:** owners пишутся в каталог атомарно при вставке И воркфлоу создания
устанавливает реальные роли владельца во внешних системах.

**Почему:** мотивация задачи — устранить «бесхозный сервис». Вариант A пишет
владельца только в каталог; во внешних системах (GitLab/Vault/IDM) у владельца
нет доступа до отдельного `ChangeOwners` — то есть воспроизводится ровно то окно,
которое мы устраняем. Вариант B делает владение реальным «с первой секунды»,
консистентно с ADR-0011. Стоимость (интеграция шагов + компенсации + temporal-
тесты + правка степпера) оправдана, тем более что activities синхронизации
владельцев уже существуют и переиспользуются (`GitLabSyncMembers`,
`VaultSyncOwners`, `IDMSyncOwnerRoles`).

**Альтернатива A** отвергнута как неполная (см. выше). **Альтернатива «отдельный
воркфлоу владельцев после создания»** отвергнута: нарушает атомарность и
возвращает окно бесхозности.

### D2. Атомарная запись владельцев на уровне INSERT (точка невозврата каталога)

**Выбор:** вставка записи становится транзакционной — строка `services`
(status=`creating`) + строки `service_owners` + `owners_version = 1` в одной
транзакции (`insertService` поверх `Querier`/`TxRepo`). `CreateRecord` принимает
нормализованных владельцев; `CreateService` нормализует через `normalizeOwners` и
отклоняет пустой набор ДО вставки.

**Почему:** «установить атомарно с созданием» и «запись фиксируется первой»
(сохраняем существующий строгий порядок и best-effort→FAILED при сбое запуска).
В воркфлоу owners уже не пишутся повторно в каталог — они там с момента INSERT;
воркфлоу лишь синхронизирует внешние системы. Это смещает «точку невозврата
владельцев» на INSERT (как и для самой записи), что естественно: запись и её
владелец неразделимы.

**`owners_version=1`:** консистентно с первой `SetOwners` (0→1). Так первая
последующая смена владельцев ожидает `expected_version=1`. Достигается тем, что
INSERT при непустых владельцах ставит `owners_version=1` (а не 0).

**Альтернатива «писать owners в воркфлоу шагом CatalogSetOwners»** отвергнута:
тогда между INSERT и шагом воркфлоу существует запись без владельцев — нарушение
инварианта.

### D3. Порядок шагов воркфлоу создания и компенсации (вариант B)

Порядок активностей (зависимости ресурсов диктуют размещение шагов владельцев
ПОСЛЕ создания соответствующего ресурса):

1. GitLab create repo — `addComp(GitLabDeleteRepo)`
2. GitLab sync members (`add=owners, remove=[]`) — отдельная компенсация НЕ
   нужна: удаление репозитория (шаг 1) снимает и членство.
3. Harbor create — `addComp(HarborDelete)`
4. Vault setup AppRole — `addComp(VaultTeardown)`
5. Vault sync owners policies (`add=owners, remove=[]`) — компенсация поглощается
   teardown Vault (шаг 4).
6. Inject secrets
7. Activate (guarded-CAS `creating→active`)
8. IDM sync owner roles (`add=owners, remove=[]`) — ПОСЛЕ активации, best-effort
   с алертом при исчерпании ретраев (как IDM-шаг в `changeowners`), без
   молчаливого отката (каталог — источник правды, ADR-0005/0008).

**Почему такие компенсации:** членство GitLab и политики Vault — производные от
самих ресурсов; их полное удаление (delete repo / teardown Vault) уже снимает
назначения, поэтому отдельные restore-компенсации избыточны и упрощают Saga.
Сбой ДО активации → существующая ветка `fail()`: компенсации в обратном порядке
+ перевод записи в FAILED. IDM-роли вынесены ПОСЛЕ активации (как и в
changeowners IDM после точки невозврата): их сбой не должен откатывать уже
поднятый сервис.

**Реализация activities:** переиспользуем имена/типы из `changeowners`
(`SyncMembersInput{Ref, Add, Remove}`, `IDMSyncInput{Project, Add, Remove}`).
DevInfra worker уже регистрирует эти activities; убедиться, что воркфлоу создания
исполняется на том же worker'е с подключёнными опциями `WithOwners`/`WithIDMRoles`
(проверить регистрацию в `services/devinfra-worker`).

**`CreateServiceInput`** расширяется полем `Owners []string` (нормализованным),
прокидывается из usecase через `wfstarter.StartCreateService`.

### D4. Без проверки существования владельцев в справочнике IDM при создании

**Выбор:** не проверять существование субъектов в IDM при создании — как в
`ChangeOwners` (MVP без существ. проверки). **Почему:** консистентность с уже
принятым поведением смены владельцев; проверка существования — отдельная сквозная
задача (ADR-0016), вводить её только в создании создало бы расхождение контрактов.
Валидация ограничивается формой (непусто/без дублей/без пустых строк).

### D5. Авто-владелец: не добавлять неявно; форма МОЖЕТ предлагать создателя

**Выбор:** бэкенд берёт владельцев строго из запроса (без неявного добавления
создателя). Форма МОЖЕТ предлагать текущего пользователя как дефолт (UX), но это
не обязательность. **Почему:** требование — явное указание владельцев; неявное
добавление скрывало бы фактический состав и расходилось бы с декларативной
семантикой набора владельцев.

### D6. Контракт и валидация периметра

- proto: `repeated string owners = 3;` в `CreateServiceRequest`.
- openapi: `required:[name, owners]`, `owners` — `type: array`, `minItems: 1`,
  `items:{type:string, minLength:1}`.
- gateway `create`: распарсить `owners`, валидировать (непусто/без пустых/без
  дублей) тем же приёмом, что `setOwners`, → 400 через существующий маппинг;
  прокинуть в gRPC. gRPC-сторона дополнительно отклоняет пустой набор
  (`InvalidArgument`) как defense-in-depth (валидация и в usecase).

### D7. Портал: форма и степпер

- `CreateServicePage`: добавить поле владельцев (textarea), `parseOwners` из
  `OwnersCard` (вынести/переиспользовать экспорт). zod-схема `CreateServiceRequest`
  из кодогена становится источником валидации (owners как массив строк). Так как
  форма принимает текст, маппинг `text→owners[]` через `parseOwners` перед
  `mutate`; ошибка пустых владельцев — inline + тост (`lib/errors`).
- `workflow-steps.ts`: обновить подписи `OPERATION_STEPS.create` под порядок D3
  (добавить шаги назначения владельцев GitLab/Vault/IDM). `createPhase` и грубая
  гранулярность не меняются (всё ещё одна фаза на все шаги).

## Risks / Trade-offs

- **[Расширение `CreateServiceInput`/воркфлоу ломает детерминизм истории
  выполняющихся воркфлоу]** → в MVP нет долгоживущих in-flight воркфлоу создания
  между деплоями; новые поля только аддитивны, старые воркфлоу не реиграются.
  Mitigation: деплой при отсутствии активных созданий; новые шаги добавляются в
  конец логических секций.
- **[Компенсация членства/политик через delete ресурса предполагает, что delete
  идемпотентно снимает назначения]** → activities удаления уже идемпотентны
  (Saga-инвариант). Mitigation: temporal-тест проверяет, что при сбое до
  активации вызываются delete repo / teardown Vault.
- **[Сбой IDM-ролей после активации оставляет сервис active без ролей в IDM]** →
  допустимо и согласовано с changeowners (каталог — источник правды); алерт
  оператору, ретраи идемпотентны. Самовосстановление — повторным ChangeOwners.
- **[BREAKING контракт ломает всех вызывающих]** → перечислены в Migration; web и
  все e2e/harness обновляются в этом же change; `gen:check`/CI ловят расхождения.
- **[`gen:check` краснеет при ручной правке сгенерированного]** → перегенерация
  только инструментами (buf + web-кодоген + копия `public/openapi.yaml`),
  сгенерированное не править руками.

## Migration Plan

1. Ветка `change/service-create-requires-owner` от master.
2. Контракт: proto + openapi → перегенерация Go (buf) и web (openapi-typescript +
   openapi-zod-client + копия `public/openapi.yaml`); `gen:check` зелёный.
3. Бэкенд: репозиторий (транзакционный INSERT с owners + `owners_version=1`),
   usecase (`CreateService`/`CreateRecord` принимают owners, нормализация,
   отклонение пустых), `CreateServiceInput.Owners`, wfstarter, воркфлоу D3 +
   регистрация activities владельцев на worker'е создания.
4. Gateway: проброс + валидация owners.
5. Портал: форма + `workflow-steps` подписи.
6. Обновить вызывающих (BREAKING): `tests/e2e/{stories,errors,harness,
   gitlab_integration,vault_integration,harbor_integration}_test.go`.
7. Тесты: unit (usecase/репозиторий/валидация), gateway, web, temporal (воркфлоу +
   компенсации), e2e (создание с владельцами + отказ без владельцев).
8. PR с зелёным CI (go test всех модулей `-race -shuffle`, golangci-lint,
   govulncheck, gen:check, openapi-lint, web-test, integration). После merge —
   `/opsx:archive` отдельным PR sync+archive.

**Rollback:** обратимо откатом ветки/PR до merge. После merge — контракт BREAKING;
откат потребует обратного изменения контракта (нежелательно). Историческая
миграция данных не нужна (новых записей без владельцев просто не будет).
