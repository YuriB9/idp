## MODIFIED Requirements

### Requirement: REST-ручка создания сервиса

API-шлюз ДОЛЖЕН (MUST) реализовать `POST /api/projects/{project}/services` с
телом `{name, owners}`, который маппит запрос в gRPC
`projectsv1.CreateService(project, name, owners)` и при успехе возвращает HTTP 201
с телом по форме OpenAPI (идентификатор записи и стартовый статус `creating`).
Поле `owners` ДОЛЖНО (MUST) быть обязательным массивом с минимум одним непустым
элементом. Шлюз ДОЛЖЕН (MUST) синхронно валидировать владельцев (непустой массив,
без пустых строк, без дублей) ДО проксирования и при нарушении отвечать 400 через
существующий маппинг ошибок, НЕ раскрывая `err.Error()`. Шлюз НЕ ДОЛЖЕН (MUST NOT)
выполнять доменную логику создания сам — он лишь проксирует к `projects`.

#### Scenario: Успешный запуск создания с владельцами

- **GIVEN** валидные `project` и тело `{name, owners}` с непустым набором владельцев
- **WHEN** клиент шлёт `POST /api/projects/{project}/services`
- **THEN** шлюз вызывает gRPC `CreateService` с владельцами, и при `OK` отвечает
  201 с JSON, содержащим `id` и `status:"creating"` по схеме OpenAPI

#### Scenario: Конфликт имени сервиса

- **GIVEN** в проекте уже существует сервис с таким `name`
- **WHEN** клиент шлёт `POST /api/projects/{project}/services`
- **THEN** gRPC возвращает `FailedPrecondition`/`AlreadyExists`, и шлюз отвечает
  409 с JSON-ошибкой без раскрытия `err.Error()`

#### Scenario: Пустые или невалидные владельцы

- **WHEN** `owners` отсутствует, пуст, содержит пустую строку или дубли
- **THEN** шлюз отвечает 400 с JSON-ошибкой и НЕ вызывает gRPC, не раскрывая
  внутренних деталей

#### Scenario: Невалидное тело запроса

- **WHEN** тело отсутствует, не парсится или `name` пустой/не проходит валидацию
- **THEN** шлюз отвечает 400 с JSON-ошибкой и НЕ вызывает gRPC (либо gRPC
  вернул `InvalidArgument` → 400), не раскрывая внутренних деталей
