## ADDED Requirements

### Requirement: Инвариант «сервис без владельца невозможен» при создании

Платформа ДОЛЖНА (MUST) гарантировать, что запись сервиса в каталоге НИКОГДА не существует без владельца: владельцы устанавливаются уже при создании, а не только последующей сменой владельцев. Создание сервиса ДОЛЖНО (MUST) требовать минимум одного непустого владельца (после нормализации) и фиксировать его в записи каталога атомарно при вставке (`owners_version` стартует с 1, консистентно с первой `SetOwners`, переводящей версию 0→1). Установление ролей владельца во внешних системах (GitLab members, Vault policies, IDM owner roles) при создании ДОЛЖНО (MUST) переиспользовать сквозной механизм синхронизации владельцев (ADR-0011) с `add = owners`, `remove = []`. Семантика смены/переноса/вывода владельцев (ChangeOwners/Transfer/Decommission) НЕ изменяется.

#### Scenario: Созданный сервис сразу имеет владельца в каталоге

- **GIVEN** успешно созданный сервис с набором владельцев из запроса
- **WHEN** читается запись каталога сразу после создания (даже в статусе `creating`)
- **THEN** запись содержит непустой набор владельцев и `owners_version = 1`

#### Scenario: Владельцы создания нормализуются как при смене владельцев

- **GIVEN** запрос создания с владельцами, содержащими дубли, пустые строки и неупорядоченность
- **WHEN** usecase нормализует владельцев перед вставкой
- **THEN** применяется та же нормализация (`normalizeOwners`: trim/dedup/sort), что и в сценарии смены владельцев — детерминированный порядок, без пустых строк и дублей

#### Scenario: Версия владельцев при создании консистентна со сменой

- **GIVEN** созданный сервис с `owners_version = 1`
- **WHEN** позже выполняется первая смена владельцев с `expected_version = 1`
- **THEN** optimistic-concurrency совпадает по версии, и смена применяется без конфликта версии
