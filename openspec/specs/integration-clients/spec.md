## Requirements
### Requirement: Клиенты интеграций за узкими интерфейсами

Клиенты GitLab, Vault и Harbor ДОЛЖНЫ (MUST) объявляться узкими интерфейсами в `services/devinfra-worker/internal`, чтобы activities зависели от абстракций, а не от конкретных реализаций. Каждый интерфейс ДОЛЖЕН (MUST) предоставлять как операцию провизии, так и парную операцию компенсации (удаления/отзыва). Реализация по умолчанию ДОЛЖНА (MUST) работать против WireMock-моков (`GITLAB_BASE_URL`/`VAULT_BASE_URL`/`HARBOR_BASE_URL`); для дефолтного прогона тестов ДОЛЖЕН (MUST) существовать in-memory стаб.

#### Scenario: Activity зависит от интерфейса, не от реализации

- **WHEN** инспектируется код activity провизии
- **THEN** activity принимает интерфейс клиента интеграции (GitLab/Vault/Harbor), что позволяет подменять его in-memory стабом в тестах без сети

#### Scenario: Реализация против моков

- **GIVEN** поднятые WireMock-моки `mock-gitlab`/`mock-vault`/`mock-harbor`
- **WHEN** worker конфигурируется через `GITLAB_BASE_URL`/`VAULT_BASE_URL`/`HARBOR_BASE_URL`
- **THEN** клиенты интеграций направляют запросы к мокам и корректно разбирают их ответы

### Requirement: SSRF-guard на всех исходящих запросах интеграций

Все исходящие HTTP-запросы к GitLab/Vault/Harbor ДОЛЖНЫ (MUST) проходить SSRF-guard `pkg/ssrf`: `ValidateURL` при конфигурации/записи адреса и `GuardedDialContext` в HTTP-транспорте при отправке. Клиент интеграции НЕ ДОЛЖЕН (MUST NOT) выполнять запрос к адресу, не прошедшему guard (приватные/loopback/метаданные-эндпоинты блокируются согласно политике `pkg/ssrf`).

#### Scenario: Запрос к разрешённому адресу проходит

- **GIVEN** базовый URL интеграции, прошедший `ValidateURL`
- **WHEN** клиент выполняет запрос через транспорт с `GuardedDialContext`
- **THEN** запрос отправляется штатно

#### Scenario: Запрос к запрещённому адресу блокируется

- **GIVEN** базовый URL, указывающий на запрещённый политикой `pkg/ssrf` адрес (например, loopback/метаданные)
- **WHEN** клиент пытается выполнить запрос
- **THEN** запрос отклоняется SSRF-guard и не уходит во внешнюю сеть

### Requirement: Идемпотентность операций провизии и компенсации

Операции клиентов интеграций и их компенсации ДОЛЖНЫ (MUST) быть идемпотентны (создание репозитория, директории, Robot Account, политик, AppRole, инъекция переменных). Для GitLab идемпотентность ДОЛЖНА (MUST) опираться на коды ответов реального GitLab и при необходимости на предварительную проверку существования (GET-then-act), а НЕ на разбор текста сообщений об ошибке (текст не контракт и зависит от версии): повторное создание уже существующего репозитория ДОЛЖНО (MUST) трактоваться как успех (no-op), повторное удаление отсутствующего (`404`/`202`) — как успех, повторная архивация/перенос в уже целевое состояние — как успех. Повторный вызов операции провизии при уже существующем ресурсе НЕ ДОЛЖЕН (MUST NOT) приводить к ошибке или дубликату; повторный вызов компенсации при уже удалённом ресурсе ДОЛЖЕН (MUST) считаться успешным (no-op).

#### Scenario: Повторная провизия существующего ресурса

- **GIVEN** GitLab-репозиторий, уже созданный предыдущей попыткой
- **WHEN** операция создания вызывается повторно
- **THEN** клиент по предварительной проверке существования возвращает успех без создания дубликата и без зависимости от текста ошибки GitLab

#### Scenario: Повторная компенсация удалённого ресурса

- **GIVEN** Harbor-директория, уже удалённая компенсацией
- **WHEN** компенсация удаления вызывается повторно
- **THEN** операция завершается успешно как no-op (ресурс отсутствует — это не ошибка)

#### Scenario: Повторный перенос в уже целевую группу

- **GIVEN** репозиторий, уже находящийся в целевой группе
- **WHEN** операция переноса вызывается повторно
- **THEN** клиент трактует это как успех (no-op) по проверке текущего namespace, не возвращая ошибку

### Requirement: Инъекция секретов в CI/CD-переменные GitLab

После создания секретов в Vault и Robot Account в Harbor клиент интеграции ДОЛЖЕН (MUST) инъектировать выданные токены (Vault AppRole RoleID/SecretID и Harbor Robot-токен) в CI/CD-переменные GitLab-репозитория. Секреты НЕ ДОЛЖНЫ (MUST NOT) логироваться в открытом виде. Операция инъекции ДОЛЖНА (MUST) быть идемпотентна через upsert переменной (создание или перезапись по ключу), совместимый с реальным GitLab, не создавая дубликатов и не завися от текста ошибки «переменная существует».

#### Scenario: Токены попадают в переменные репозитория

- **GIVEN** созданные Vault AppRole и Harbor Robot Account
- **WHEN** выполняется инъекция секретов
- **THEN** в CI/CD-переменных GitLab-репозитория появляются соответствующие значения, и сами значения не пишутся в логи в открытом виде

#### Scenario: Повторная инъекция идемпотентна

- **GIVEN** переменные уже установлены предыдущей попыткой
- **WHEN** инъекция выполняется повторно
- **THEN** переменные перезаписываются (upsert) без ошибки и без дублирования

### Requirement: Исходящая аутентификация запросов к реальному GitLab

Все исходящие HTTP-запросы GitLab-клиента к реальному GitLab ДОЛЖНЫ (MUST) нести аутентификацию заголовком `PRIVATE-TOKEN` со значением из конфигурации (`integrations.Config.GitLabToken`, env `GITLAB_TOKEN`). Токен НЕ ДОЛЖЕН (MUST NOT) логироваться. Запросы к Vault и Harbor этой аутентификацией НЕ затрагиваются.

#### Scenario: GitLab-запрос несёт токен

- **GIVEN** сконфигурированный `GITLAB_TOKEN`
- **WHEN** GitLab-клиент выполняет любой запрос к GitLab API
- **THEN** запрос содержит заголовок `PRIVATE-TOKEN` с этим значением, и значение токена не попадает в логи

#### Scenario: Токен не применяется к другим интеграциям

- **WHEN** клиент Vault или Harbor выполняет запрос
- **THEN** заголовок `PRIVATE-TOKEN` GitLab не добавляется к этим запросам

### Requirement: Резолвинг namespace и владельца в идентификаторы GitLab

GitLab-клиент ДОЛЖЕН (MUST) резолвить путь группы (проект каталога) в числовой `namespace_id` и логин владельца в числовой `user_id` через GitLab API, и использовать числовые идентификаторы в телах запросов (`namespace_id` при создании репозитория, `user_id` при синхронизации участников). Результаты резолвинга МОГУТ (MAY) кэшироваться в пределах процесса. Если маппинг владельца на пользователя GitLab не задан, синхронизация этого участника ДОЛЖНА (MUST) безопасно пропускаться (не приводить к ошибке воркфлоу).

#### Scenario: Создание репозитория использует namespace_id

- **GIVEN** предсозданная группа с известным путём
- **WHEN** клиент создаёт репозиторий в этом namespace
- **THEN** путь группы резолвится в `namespace_id`, и создание шлёт `namespace_id` (число), а не строку namespace

#### Scenario: Синхронизация участников использует user_id

- **GIVEN** владелец, отображённый на существующего пользователя GitLab
- **WHEN** клиент добавляет участника в репозиторий
- **THEN** логин резолвится в `user_id`, и запрос шлёт `user_id` (число) с числовым `access_level`

#### Scenario: Незаданный маппинг владельца пропускается

- **GIVEN** владелец без отображения на пользователя GitLab
- **WHEN** выполняется синхронизация участников
- **THEN** этот владелец безопасно пропускается без ошибки воркфлоу

### Requirement: Выбор реализации GitLab-клиента по конфигурации

`devinfra-worker` ДОЛЖЕН (MUST) выбирать реализацию GitLab-клиента по конфигурации: при заданном `GITLAB_TOKEN` собирается клиент против реального GitLab (с аутентификацией и резолвингом идентификаторов); иначе сохраняется поведение по умолчанию (клиент против WireMock-моков). In-memory стаб ДОЛЖЕН (MUST) оставаться доступным для дефолтного прогона тестов без сети. Vault и Harbor ОСТАЮТСЯ моками независимо от выбора GitLab-клиента.

#### Scenario: Без токена — поведение по умолчанию

- **WHEN** `GITLAB_TOKEN` не задан
- **THEN** `devinfra-worker` использует GitLab-клиент против моков, как прежде, без обращения к реальному GitLab

#### Scenario: С токеном — реальный GitLab

- **WHEN** `GITLAB_TOKEN` задан (профиль реального GitLab)
- **THEN** `devinfra-worker` использует GitLab-клиент с аутентификацией и резолвингом идентификаторов против реального GitLab

### Requirement: Исходящая аутентификация запросов к реальному Vault

Все исходящие HTTP-запросы Vault-клиента к реальному Vault ДОЛЖНЫ (MUST) нести аутентификацию заголовком `X-Vault-Token` со значением из конфигурации (`integrations.Config.VaultToken`, env `VAULT_TOKEN` или `VAULT_TOKEN_FILE`). Токен НЕ ДОЛЖЕН (MUST NOT) логироваться. Запросы к GitLab и Harbor этой аутентификацией НЕ затрагиваются (заголовок `X-Vault-Token` не протекает на другие интеграции).

#### Scenario: Vault-запрос несёт токен

- **GIVEN** сконфигурированный `VAULT_TOKEN`
- **WHEN** Vault-клиент выполняет любой запрос к Vault API
- **THEN** запрос содержит заголовок `X-Vault-Token` с этим значением, и значение токена не попадает в логи

#### Scenario: Токен не применяется к другим интеграциям

- **WHEN** клиент GitLab или Harbor выполняет запрос
- **THEN** заголовок `X-Vault-Token` Vault не добавляется к этим запросам

### Requirement: Семантика реального Vault API в Vault-клиенте

Vault-клиент в реальном режиме ДОЛЖЕН (MUST) использовать семантику реального HashiCorp Vault API: секреты по KV v2 (`/v1/secret/data/...` с обёрткой `{"data":{...}}` на запись и разбором `data.data` на чтение); AppRole-роли и per-service ACL-политики (`/v1/sys/policies/acl/...`, `/v1/auth/approle/role/...`); identity-entity для владельцев (`/v1/identity/entity/name/...`). Идемпотентность ДОЛЖНА (MUST) опираться на коды ответов Vault и при необходимости на предварительную проверку существования (GET-then-act), а НЕ на разбор текста ошибок: запись (политика/роль/секрет/entity) трактуется как успех по 2xx (`200`/`204`), чтение/удаление отсутствующего (`404`) — как успех (no-op). Клиент НЕ ДОЛЖЕН (MUST NOT) светить сырые тексты ошибок Vault как контракт.

#### Scenario: AppRole и политика создаются идемпотентно

- **GIVEN** реальный Vault с включённым AppRole-движком
- **WHEN** воркфлоу провизии вызывает создание AppRole и политики повторно
- **THEN** повторная запись трактуется как успех (upsert/2xx) без дубликата и без зависимости от текста ошибки, а `role-id` читается из `data.role_id`

#### Scenario: Миграция секретов по KV v2

- **GIVEN** секреты по исходному пути `secret/data/<src>`
- **WHEN** клиент мигрирует пути source→target
- **THEN** чтение разбирает `data.data`, запись шлёт `{"data":{...}}` по целевому пути, создаётся целевая политика, исходный путь очищается (`404` при очистке — успех)

### Requirement: Семантика немедленного отзыва доступа Vault

Операция немедленного отзыва доступа Vault (`RevokeSecretID`, ADR-0012) ДОЛЖНА (MUST) отзывать доступ через перечисление и уничтожение secret-id, а НЕ через несуществующий «destroy all» одним вызовом. Клиент ДОЛЖЕН (MUST) перечислить активные `secret-id`-accessors роли (`LIST`/`GET ?list=true`) и уничтожить каждый через `secret-id-accessor/destroy`. Операция ДОЛЖНА (MUST) быть идемпотентна: отсутствие активных secret-id (пустой список / `404`) трактуется как успех (no-op). Результат отзыва ДОЛЖЕН (MUST) быть наблюдаем через Vault API (после отзыва набор активных accessors пуст).

#### Scenario: Отзыв уничтожает активные secret-id

- **GIVEN** AppRole-роль с выданными активными secret-id
- **WHEN** вызывается немедленный отзыв доступа
- **THEN** клиент перечисляет secret-id-accessors и уничтожает каждый, после чего набор активных accessors роли пуст

#### Scenario: Повторный отзыв идемпотентен

- **GIVEN** роль без активных secret-id (уже отозваны)
- **WHEN** отзыв вызывается повторно
- **THEN** операция завершается успешно как no-op (`404`/пустой список — не ошибка)

### Requirement: Выбор реализации Vault-клиента по конфигурации

`devinfra-worker` ДОЛЖЕН (MUST) выбирать реализацию Vault-клиента по конфигурации: при заданном `VAULT_TOKEN` (или `VAULT_TOKEN_FILE`) собирается клиент против реального Vault (с аутентификацией `X-Vault-Token` и семантикой реального API); иначе сохраняется поведение по умолчанию (клиент против WireMock-мока). In-memory стаб ДОЛЖЕН (MUST) оставаться доступным для дефолтного прогона тестов без сети. GitLab и Harbor сохраняют свой выбор реализации независимо от выбора Vault-клиента.

#### Scenario: Без токена — поведение по умолчанию

- **WHEN** `VAULT_TOKEN`/`VAULT_TOKEN_FILE` не заданы
- **THEN** `devinfra-worker` использует Vault-клиент против мока, как прежде, без обращения к реальному Vault

#### Scenario: С токеном — реальный Vault

- **WHEN** `VAULT_TOKEN` задан (профиль реального Vault)
- **THEN** `devinfra-worker` использует Vault-клиент с аутентификацией `X-Vault-Token` и семантикой реального API против реального Vault
