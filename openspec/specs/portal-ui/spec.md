## Requirements
### Requirement: SPA-портал поднимается как реальное приложение

Модуль `./web` ДОЛЖЕН (MUST) быть полноценным SPA на React 19 + Vite +
TypeScript с роутингом (react-router) и `QueryClient` (TanStack Query):
`index.html`, `main.tsx`, корневой `App`. ДОЛЖНЫ (MUST) присутствовать
npm-скрипты `dev`/`build`/`preview` и dev-прокси `/api` → gateway. OpenAPI
остаётся единственным источником правды для TS-клиента/zod (кодоген `gen` без
diff в CI).

#### Scenario: Сборка и запуск

- **WHEN** выполняется `npm run build`
- **THEN** Vite собирает SPA без ошибок типизации

#### Scenario: Dev-прокси периметра

- **GIVEN** запущенный gateway
- **WHEN** в dev-режиме (`npm run dev`) портал шлёт запрос на `/api/...`
- **THEN** dev-сервер Vite проксирует его на gateway, без CORS-ошибок в браузере

### Requirement: Экран списка сервисов проекта

Портал ДОЛЖЕН (MUST) показывать экран со списком сервисов проекта, получая
данные через `GET /api/projects/{project}/services`. Ответ ДОЛЖЕН (MUST)
валидироваться рантайм через сгенерированную из OpenAPI zod-схему (`.parse`);
дрейф контракта ДОЛЖЕН (MUST) падать явно.

#### Scenario: Отображение списка

- **GIVEN** периметр вернул валидный список сервисов
- **WHEN** открывается экран списка
- **THEN** сервисы отображаются с именем и статусом

#### Scenario: Дрейф контракта падает явно

- **GIVEN** ответ API не соответствует zod-схеме
- **WHEN** портал парсит ответ через `.parse`
- **THEN** выбрасывается ошибка валидации (несоответствие не маскируется тихо)

### Requirement: Форма создания сервиса

Портал ДОЛЖЕН (MUST) предоставлять форму создания сервиса на react-hook-form,
где источником правды валидации входных данных служит zod-схема из кодогена.
Отправка формы ДОЛЖНА (MUST) вызывать `POST /api/projects/{project}/services` и
по успеху переводить пользователя на экран прогресса создаваемого сервиса.

#### Scenario: Успешная отправка

- **GIVEN** заполнено валидное имя сервиса
- **WHEN** пользователь отправляет форму
- **THEN** портал шлёт `POST`, получает `id`/`status:"creating"` и переходит на
  экран прогресса

#### Scenario: Клиентская валидация имени

- **GIVEN** имя пустое или не проходит zod-схему
- **WHEN** пользователь пытается отправить форму
- **THEN** форма показывает ошибку валидации и НЕ шлёт запрос

#### Scenario: Конфликт имени с сервера

- **GIVEN** периметр вернул 409 (имя занято)
- **WHEN** обрабатывается ответ
- **THEN** форма показывает понятное сообщение о конфликте без раскрытия
  внутренних деталей сервера

### Requirement: Экран прогресса с поллингом статуса

Портал ДОЛЖЕН (MUST) иметь экран/индикатор прогресса, который через переиспользуемый
хук поллинга опрашивает `GET /api/projects/{project}/services/{name}` (TanStack
Query с интервалом) до достижения терминального статуса `active`, `failed` или
`decommissioned`, после чего опрос ДОЛЖЕН (MUST) останавливаться. Ход ДОЛЖЕН (MUST)
показываться ЕДИНЫМ компонентом ступенчатого прогресса (а не однострочной панелью):
шаги соответствуют активностям воркфлоу создания. Все ответы поллинга ДОЛЖНЫ (MUST)
валидироваться через zod `.parse`.

#### Scenario: Поллинг до ACTIVE со степпером

- **GIVEN** сервис в статусе `creating`
- **WHEN** экран прогресса поллит `GetService` через хук
- **THEN** при переходе в `active` опрос останавливается, все шаги степпера
  помечаются завершёнными, показывается успешный исход

#### Scenario: Поллинг до FAILED со степпером

- **GIVEN** сервис в статусе `creating`
- **WHEN** провизия завершилась откатом (Saga) и статус стал `failed`
- **THEN** опрос останавливается, степпер показывает неуспех операции и факт отката

#### Scenario: Промежуточный статус не завершает опрос

- **GIVEN** очередной ответ снова `creating`
- **WHEN** обрабатывается ответ поллинга
- **THEN** опрос продолжается с интервалом (терминал не достигнут), степпер
  продолжает показывать выполнение

### Requirement: Отображение и изменение владельцев в портале

Портал ДОЛЖЕН (MUST) отображать владельцев сервиса на экранах чтения (список/
карточка сервиса) и предоставлять форму изменения состава владельцев
(react-hook-form + zod как источник правды формы, мутации через TanStack Query).
Форма ДОЛЖНА (MUST) отправлять полный желаемый набор владельцев и текущую версию
`owners_version` (optimistic-concurrency). Ответы API ДОЛЖНЫ (MUST) валидироваться
через zod `.parse`. После успешного ЗАПУСКА смены владельцев ход операции ДОЛЖЕН
(MUST) показываться ЕДИНЫМ ступенчатым прогрессом (вырожденный/короткий степпер), а
НЕ тостом успеха. UI ДОЛЖЕН (MUST) обрабатывать синхронные `403` (нет права) и `409`
(конфликт версии) понятным тостом-сообщением и НЕ ДОЛЖЕН (MUST NOT) показывать сырые
внутренние ошибки. Все комментарии в коде — на русском языке.

#### Scenario: Просмотр владельцев

- **GIVEN** сервис с владельцами `{alice, bob}`
- **WHEN** пользователь открывает карточку/список сервиса
- **THEN** владельцы отображаются; данные ответа проходят zod `.parse`

#### Scenario: Успешный запуск изменения владельцев показывает степпер

- **GIVEN** форма изменения владельцев с валидным набором и текущей версией
- **WHEN** пользователь сохраняет изменения и мутация принята
- **THEN** ход смены владельцев показывается ступенчатым прогрессом; по завершении
  (рост `owners_version`) список владельцев и кэш TanStack Query обновляются

#### Scenario: Отказ доступа (403)

- **GIVEN** у пользователя нет права `change_owners`
- **WHEN** он отправляет форму изменения владельцев
- **THEN** UI показывает понятный тост об отказе доступа, без раскрытия внутренних
  деталей; степпер не активируется

#### Scenario: Конфликт версии (409)

- **GIVEN** владельцы были изменены конкурентно (версия устарела)
- **WHEN** пользователь сохраняет форму со старой версией
- **THEN** UI сообщает тостом о конфликте/устаревших данных и предлагает обновить
  состояние перед повтором

### Requirement: Действие «Вывести из эксплуатации» с подтверждением

Портал ДОЛЖЕН (MUST) предоставлять действие «Вывести из эксплуатации» на странице
сервиса (react-hook-form + zod, TanStack Query). Действие ДОЛЖНО (MUST) требовать
явного подтверждения с вводом имени сервиса и явно сообщать, что это decommission
(soft-delete, необратимость для доступа), а не purge. Действие ДОЛЖНО (MUST)
отправлять явное предусловие снятой нагрузки (`load_drained`). Кнопка/действие
ДОЛЖНЫ (MUST) быть недоступны или заблокированы для сервиса, который уже
`decommissioned` или в недопустимом статусе. После успешного ЗАПУСКА ход операции
ДОЛЖЕН (MUST) показываться ЕДИНЫМ ступенчатым прогрессом (шаги: отзыв
GitLab → Harbor → Vault → запись каталога; с пометкой точки невозврата), а НЕ
тостом. Ответы API ДОЛЖНЫ (MUST) валидироваться через zod (`.parse`).

#### Scenario: Подтверждение и ход вывода показываются степпером

- **GIVEN** активный сервис на странице портала
- **WHEN** пользователь подтверждает вывод (имя + снятая нагрузка) и мутация принята
- **THEN** отправляется `POST .../decommission`, ход показывается ступенчатым
  прогрессом с пометкой необратимости; по достижении `decommissioned` все шаги
  завершены, показан `decommissioned_at`

#### Scenario: Блокировка повторного/недопустимого действия

- **GIVEN** сервис уже в статусе `decommissioned` (или `creating`/`failed`)
- **WHEN** отображается страница сервиса
- **THEN** действие «Вывести из эксплуатации» недоступно/заблокировано

### Requirement: Обработка ошибок и индикация статуса в портале

Портал ДОЛЖЕН (MUST) корректно обрабатывать СИНХРОННЫЕ ошибки запуска операции
вывода из эксплуатации тостом (`lib/errors`): `403` (нет права) — отказ доступа;
`409` (конкурентный конфликт) — конфликт с предложением перечитать состояние; `422`
(предусловие не выполнено) — невыполненное предусловие. АСИНХРОННЫЙ ход и исход
(включая `failed` с откатом Saga) ДОЛЖНЫ (MUST) показываться степпером, а НЕ тостом.
Статус `decommissioned` ДОЛЖЕН (MUST) индицироваться бейджем статуса в списке и на
странице сервиса; `decommissioned_at` ДОЛЖЕН (MUST) отображаться для выведенных
сервисов.

#### Scenario: Отказ доступа (403) — тостом

- **GIVEN** у пользователя нет права `decommission`
- **WHEN** он пытается вывести сервис из эксплуатации
- **THEN** UI показывает тост об отказе доступа, статус не меняется, степпер не
  активируется

#### Scenario: Предусловие не выполнено (422) — тостом

- **GIVEN** нагрузка не снята
- **WHEN** API возвращает `422`
- **THEN** UI показывает понятный тост о невыполненном предусловии (нагрузка не снята)

#### Scenario: Конфликт (409) — тостом

- **GIVEN** статус сервиса изменился конкурентно
- **WHEN** API возвращает `409`
- **THEN** UI показывает тост о конфликте и предлагает перечитать состояние

### Requirement: Действие «Перенести сервис» с подтверждением и предупреждением о рисках

Портал ДОЛЖЕН (MUST) предоставлять действие «Перенести сервис» на странице сервиса
(react-hook-form + zod, TanStack Query). Действие ДОЛЖНО (MUST) требовать явного
подтверждения с вводом имени сервиса и явно предупреждать о НЕОБРАТИМОСТИ/рисках
(transfer репозитория GitLab и миграция путей Vault частично необратимы — точка
невозврата). Действие ДОЛЖНО (MUST) позволять указать target-проект. Кнопка/действие
ДОЛЖНЫ (MUST) быть недоступны для сервиса не в статусе `active`. После успешного
ЗАПУСКА ход переноса ДОЛЖЕН (MUST) показываться ЕДИНЫМ ступенчатым прогрессом (шаги:
начало переноса в каталоге → transfer GitLab → миграция Vault → метаданные
Harbor → фиксация каталога → перенос ролей; с пометкой точки невозврата), а НЕ
тостом. Ответы API ДОЛЖНЫ (MUST) валидироваться через zod (`.parse`).

#### Scenario: Подтверждение и ход переноса показываются степпером

- **GIVEN** активный сервис на странице портала
- **WHEN** пользователь подтверждает перенос (target-проект + имя) и мутация принята
- **THEN** отправляется `POST .../transfer` с `{target_project}`, ход показывается
  ступенчатым прогрессом с пометкой точки невозврата; статус проходит
  `transferring` → итоговый `active`

#### Scenario: Блокировка для недопустимого статуса

- **GIVEN** сервис в статусе `transferring` (или `creating`/`failed`/
  `decommissioned`)
- **WHEN** отображается страница сервиса
- **THEN** действие «Перенести сервис» недоступно/заблокировано

### Requirement: Обработка ошибок и индикация статуса переноса в портале

Портал ДОЛЖЕН (MUST) корректно обрабатывать СИНХРОННЫЕ ошибки запуска переноса
тостом (`lib/errors`): `403` (нет права на source или target) — отказ доступа; `409`
(занятое имя в target или конкурентный конфликт) — конфликт с предложением выбрать
другой проект/перечитать состояние; `422` (недопустимый исходный статус) —
невыполненное предусловие. АСИНХРОННЫЙ ход и исход (включая `failed`) ДОЛЖНЫ (MUST)
показываться степпером, а НЕ тостом. Статус `transferring` ДОЛЖЕН (MUST)
индицироваться бейджем статуса (`StatusBadge`) в списке и на странице сервиса.

#### Scenario: Отказ доступа (403) — тостом

- **GIVEN** у пользователя нет права `transfer` на source или `transfer_in` на target
- **WHEN** он пытается перенести сервис
- **THEN** UI показывает тост об отказе доступа, статус не меняется, степпер не
  активируется

#### Scenario: Конфликт занятого имени (409) — тостом

- **GIVEN** в target-проекте уже есть сервис с тем же именем
- **WHEN** API возвращает `409`
- **THEN** UI показывает тост о конфликте и предлагает выбрать другой target-проект

#### Scenario: Индикация transferring степпером и бейджем

- **GIVEN** сервис в процессе переноса
- **WHEN** отображается страница сервиса
- **THEN** `StatusBadge` показывает `transferring`, а ход переноса — ступенчатым
  прогрессом

### Requirement: Раздел «Роли и доступы» (IAM-админка) в портале

Портал ДОЛЖЕН (MUST) предоставлять горизонтальный раздел «Роли и доступы» (не
вложенный в project-scope), РАЗДЕЛЁННЫЙ на ТРИ самостоятельные страницы с
отдельными маршрутами под `/iam`: «Роли» (`/iam/roles`, каталог ролей и права
выбранной роли), «Права» (`/iam/permissions`, каталог прав), «Пользователи»
(`/iam/users`, субъекты с ролями и назначение роли). Старый маршрут `/iam` ДОЛЖЕН
(MUST) редиректить на дефолтный под-раздел (`/iam/roles`), не ломая существующие
ссылки. Каждая страница ДОЛЖНА (MUST) строиться на `PageHeader` и примитивах
дизайн-системы. Чтение и мутации ДОЛЖНЫ (MUST) идти через сгенерированный из OpenAPI
zodios-клиент с рантайм-валидацией ответов (`zod .parse`); сырые внутренние ошибки
клиенту НЕ показываются. Назначение/снятие роли ДОЛЖНО (MUST) использовать
react-hook-form + zod (валидация ввода) и TanStack-мутацию с инвалидацией
соответствующих запросов (субъекты/роли субъекта) после успеха. Общие запросы и
мутации (роли/права/субъекты) ДОЛЖНЫ (MUST) переиспользоваться между страницами через
общий модуль хуков и кэш react-query (без дублирования `listRoles`/`listPermissions`).

#### Scenario: Просмотр ролей, прав и субъектов

- **GIVEN** у субъекта есть право `(read, iam:global)`
- **WHEN** пользователь открывает страницы «Роли», «Права» и «Пользователи»
- **THEN** на «Роли» отображается каталог ролей и права выбранной роли, на «Права» —
  каталог прав, на «Пользователи» — список субъектов с их ролями; ответы проходят
  рантайм-валидацию zod

#### Scenario: Назначение и снятие роли с обновлением списка

- **GIVEN** пользователь с правом `(write, iam:global)` на странице «Пользователи»
- **WHEN** он назначает роль субъекту (форма), затем снимает её
- **THEN** выполняется соответствующая мутация, после успеха запросы субъекта/ролей
  инвалидируются и список обновляется, показывается индикация результата (тост)

#### Scenario: Валидация ввода формы

- **WHEN** пользователь отправляет форму назначения с пустым subject или без
  выбранной роли
- **THEN** zod-валидация react-hook-form блокирует отправку и показывает сообщение,
  запрос не уходит

#### Scenario: Редирект старого маршрута

- **GIVEN** пользователь переходит по старой ссылке `/iam`
- **WHEN** маршрут разрешается
- **THEN** происходит редирект на `/iam/roles`, содержимое страницы «Роли»
  отображается

### Requirement: Обработка отказа доступа и ошибок в разделе «Роли и доступы»

Раздел «Роли и доступы» ДОЛЖЕН (MUST) корректно обрабатывать `403` (нет права на
админку — раздел заблокирован либо показан явный отказ, без аварии приложения),
`404` (несуществующая роль при назначении/просмотре прав роли) и `400` (валидация),
показывая стабильные пользовательские сообщения и НЕ раскрывая сырые внутренние
детали. При `403` пользователь НЕ ДОЛЖЕН (MUST) видеть содержимое каталога
(fail-closed на UI согласуется с fail-closed периметра).

#### Scenario: Нет права на админку → отказ без содержимого

- **GIVEN** у субъекта нет права `(read, iam:global)` (периметр отвечает `403`)
- **WHEN** пользователь открывает раздел «Роли и доступы»
- **THEN** раздел показывает отказ доступа (или заблокирован), содержимое каталога
  не отображается, сырые ошибки не раскрываются

#### Scenario: Несуществующая роль при назначении → понятная ошибка

- **GIVEN** выбрана несуществующая роль (периметр отвечает `404`)
- **WHEN** пользователь подтверждает назначение
- **THEN** показывается стабильное сообщение об ошибке, список не повреждается

### Requirement: Управление каталогом ролей и прав в разделе «Роли и доступы»

Портал ДОЛЖЕН (MUST) в разделе «Роли и доступы» предоставлять структурное
управление каталогом: создание роли (`POST /iam/roles`), удаление роли
(`DELETE /iam/roles/{role}`), правку набора прав роли (attach/detach из списка
существующих прав), создание права (`POST /iam/permissions`). Формы ДОЛЖНЫ (MUST)
использовать react-hook-form + zod (валидация ввода: непустое имя роли, выбранное
право и т.п.) и TanStack-мутации с инвалидацией соответствующих запросов
(`["iam","roles"]`, `["iam","permissions"]`, права роли) после успеха. Ответы
ДОЛЖНЫ (MUST) проходить рантайм-валидацию zod `.parse`; сырые внутренние ошибки НЕ
показываются. Удаление существующего права/роли — через тот же сгенерированный из
OpenAPI клиент.

- **THEN** показан raw `sub` с пометкой «нет в каталоге», кнопки управления ролями
  доступны

#### Scenario: Курсорная пагинация субъектов

- **GIVEN** список субъектов длиннее одной страницы
- **WHEN** пользователь запрашивает следующую страницу
- **THEN** загрузка идёт по `next_page_token`, новые строки дописываются без потери
  предыдущих

#### Scenario: Нет права на справочник в модалке (403)

- **GIVEN** у субъекта нет `(read, iam:directory)` (справочник отвечает `403`)
- **WHEN** пользователь открывает модалку назначения
- **THEN** пикер скрыт с явным отказом, сырые ошибки не раскрываются

#### Scenario: Каталог справочника недоступен в модалке (503)

- **GIVEN** справочник отвечает `503`
- **WHEN** пользователь вводит строку поиска в модалке
- **THEN** показывается индикация «недоступен», поле поиска остаётся (запрос
  ретраебелен)

### Requirement: Сохранение fail-closed и деградации прав на каждой странице раздела

Каждая страница раздела «Роли и доступы» ДОЛЖНА (MUST) сохранять поведение прав и
обработку ошибок на всех трёх маршрутах (`/iam/roles`, `/iam/permissions`,
`/iam/users`): при `403` на админку (`read, iam:global`) — fail-closed (содержимое
скрыто, показан явный отказ); при `403` на `manage` — структурные действия
скрыты/заблокированы (просмотр при наличии `read` остаётся); на странице
«Пользователи» при `403` на `iam:directory` — пикер скрыт с отказом, остальное
работает; при `503` каталога — индикация «недоступен», поле поиска остаётся (запрос
ретраебелен). Результаты действий ДОЛЖНЫ (MUST) показываться тостами с единым
маппингом кодов (`ASSIGN_OVERRIDES`/`CATALOG_OVERRIDES`). Сырые внутренние ошибки
наружу НЕ показываются; ответы проходят `zod .parse`. Общий guard-компонент
fail-closed ДОЛЖЕН (MUST) переиспользоваться всеми тремя страницами.

#### Scenario: 403 на админку → fail-closed на каждой странице

- **GIVEN** у субъекта нет `(read, iam:global)` (периметр отвечает `403`)
- **WHEN** пользователь открывает любую из страниц `/iam/roles`, `/iam/permissions`,
  `/iam/users`
- **THEN** показывается отказ доступа, содержимое каталога не отображается, сырые
  ошибки не раскрываются

#### Scenario: 403 на manage → структурные действия скрыты

- **GIVEN** есть `(read, iam:global)`, но нет `(manage, iam:global)`
- **WHEN** пользователь открывает страницы «Роли» и «Права»
- **THEN** просмотр каталога доступен, формы создания/удаления/правки скрыты или
  заблокированы, сырые ошибки не раскрываются

#### Scenario: 403 на справочник → пикер скрыт, остальное работает

- **GIVEN** нет `(read, iam:directory)` (периметр отвечает `403`)
- **WHEN** пользователь открывает страницу «Пользователи»
- **THEN** пикер/поиск скрыт и заменён сообщением об отказе, просмотр и снятие
  существующих ролей остаются доступны

#### Scenario: 503 каталога → индикация, поиск остаётся

- **GIVEN** `/iam/directory/subjects` отвечает `503`
- **WHEN** пользователь ищет в пикере на странице «Пользователи»
- **THEN** показана индикация «каталог недоступен», поле поиска остаётся, просмотр и
  снятие существующих ролей работают

### Requirement: Переиспользуемый компонент ступенчатого прогресса (Stepper)

Дизайн-система портала (ADR-0017) ДОЛЖНА (MUST) предоставлять переиспользуемый
UI-примитив ступенчатого прогресса в `web/src/components/ui/`. Компонент ДОЛЖЕН
(MUST) принимать упорядоченный список шагов, каждый с подписью (на русском) и
состоянием `pending | running | done | failed`, и отрисовывать их единым стилем с
цветом+иконкой на состояние (по образцу `StatusBadge`). Компонент ДОЛЖЕН (MUST)
быть доступным: контейнер с `role` и `aria-live="polite"`, текущий/изменившийся шаг
объявляется ассистивным технологиям; анимации ДОЛЖНЫ (MUST) быть motion-safe
(уважать `prefers-reduced-motion`). Компонент НЕ ДОЛЖЕН (MUST NOT) раскрывать сырые
внутренние ошибки — только подписи шагов и (для `failed`) сообщение из единого
маппинга (`lib/errors`).

#### Scenario: Отрисовка состояний шагов

- **GIVEN** список шагов с состояниями `done`, `running`, `pending`
- **WHEN** компонент отрисовывается
- **THEN** завершённые шаги показаны как выполненные, текущий — как идущий
  (motion-safe-индикатор), последующие — как ожидающие

#### Scenario: Объявление изменения шага ассистивным технологиям

- **GIVEN** контейнер шагов с `aria-live="polite"`
- **WHEN** текущий шаг меняется (`running` переходит на следующий)
- **THEN** обновление подписи объявляется без перехвата фокуса

#### Scenario: Уважение reduced-motion

- **GIVEN** пользователь с `prefers-reduced-motion: reduce`
- **WHEN** отрисовывается идущий шаг
- **THEN** анимация спиннера/прогресса не проигрывается (статичная индикация)

### Requirement: Хук поллинга статуса сервиса

Портал ДОЛЖЕН (MUST) инкапсулировать поллинг статуса сервиса в переиспользуемом
хуке (`web/src/hooks/`), который через TanStack Query опрашивает
`GET /api/projects/{project}/services/{name}` с интервалом и ОСТАНАВЛИВАЕТ опрос по
достижении терминального статуса (`active` / `failed` / `decommissioned`). Хук НЕ
ДОЛЖЕН (MUST NOT) использовать `sleep`; остановка опроса ДОЛЖНА (MUST) выражаться
через `refetchInterval`, возвращающий `false` на терминале. Каждый ответ поллинга
ДОЛЖЕН (MUST) валидироваться zod `.parse` (через сгенерированный клиент). Хук
ДОЛЖЕН (MUST) переиспользоваться экраном прогресса и всеми карточками операций.

#### Scenario: Остановка на терминальном статусе

- **GIVEN** сервис в нетерминальном статусе (`creating`/`transferring`)
- **WHEN** хук поллит статус
- **THEN** опрос продолжается с интервалом до перехода в `active`/`failed`/
  `decommissioned`, затем `refetchInterval` становится `false` и опрос
  останавливается

#### Scenario: Дрейф контракта падает явно

- **GIVEN** ответ поллинга не соответствует zod-схеме
- **WHEN** хук обрабатывает ответ
- **THEN** `.parse` выбрасывает ошибку валидации (несоответствие не маскируется)

### Requirement: Клиентская модель шагов Temporal-воркфлоу (вариант B)

Портал ДОЛЖЕН (MUST) определять для каждой асинхронной операции над сервисом
(создание, смена владельцев, decommission, transfer) фиксированный упорядоченный
список шагов, чьи подписи соответствуют порядку активностей соответствующего
Temporal-воркфлоу (`services/projects/{provisioning,changeowners,decommission,
transfer}`). Текущая фаза ДОЛЖНА (MUST) выводиться КЛИЕНТСКИ из грубого статуса
сервиса (`creating`/`active`/`transferring`/`decommissioned`/`failed`) и доменных
полей (`decommissioned_at`, `owners_version`); контракт периметра НЕ меняется.
Поскольку периметр отдаёт только грубый статус (не per-activity прогресс Temporal),
модель ДОЛЖНА (MUST) честно отражать гранулярность: пока операция идёт — шаги
воркфлоу показаны как выполняемые (без ложного утверждения, какой именно шаг
завершён); при терминальном `failed` — операция помечается неуспешной с указанием
факта компенсирующего отката (Saga, ADR-0005), без атрибуции конкретного упавшего
шага. Сами переходы статусов остаются guarded-CAS на стороне сервиса projects
(ADR-0004); клиент только ЧИТАЕТ статус и не выполняет переходов.

#### Scenario: Вывод фазы создания из статуса

- **GIVEN** сервис в статусе `creating`
- **WHEN** строится модель шагов создания (GitLab → Harbor → Vault → инъекция
  секретов → активация)
- **THEN** шаги показаны как выполняемые; при переходе в `active` все шаги
  помечаются завершёнными

#### Scenario: Вырожденный степпер короткого воркфлоу (смена владельцев)

- **GIVEN** запущена смена владельцев (короткий воркфлоу, статус сервиса не меняется)
- **WHEN** строится модель шагов смены владельцев
- **THEN** показывается вырожденный степпер (выполнение → завершено по росту
  `owners_version` / успеху мутации), без ложной per-activity гранулярности

#### Scenario: Failed отражает откат без атрибуции шага

- **GIVEN** операция завершилась терминальным `failed`
- **WHEN** отрисовывается прогресс
- **THEN** операция помечена неуспешной с сообщением о выполненном откате (Saga),
  без указания конкретного упавшего шага (грубый статус не несёт этой информации)

### Requirement: Единый ступенчатый прогресс для всех операций над сервисом

Портал ДОЛЖЕН (MUST) отображать ход ВСЕХ асинхронных операций над сервисом
(создание, смена владельцев, decommission, transfer) ОДНИМ И ТЕМ ЖЕ компонентом
ступенчатого прогресса, питаемым хуком поллинга и клиентской моделью шагов. Тост
(ADR-0017) ДОЛЖЕН (MUST) использоваться ТОЛЬКО для мгновенных синхронных ошибок
запуска операции (валидация и 4xx прав/конфликта/предусловия через `lib/errors`);
ход и исход асинхронной операции (running/done/failed + факт Saga-отката) НЕ ДОЛЖЕН
(MUST NOT) показываться тостом — только степпером. Для операций с точкой невозврата
(decommission ADR-0012, transfer ADR-0013) степпер ДОЛЖЕН (MUST) визуально помечать
необратимость.

#### Scenario: Ход операции — степпером, а не тостом

- **GIVEN** пользователь успешно запустил decommission или transfer
- **WHEN** операция выполняется асинхронно
- **THEN** её ход показывается ступенчатым прогрессом (тем же компонентом, что и для
  создания), а не тостом в углу

#### Scenario: Синхронная ошибка запуска — тостом

- **GIVEN** запуск операции отклонён синхронно (`403`/`409`/`422`)
- **WHEN** мутация возвращает ошибку до старта воркфлоу
- **THEN** показывается тост с сообщением из единого маппинга (`lib/errors`), без
  раскрытия внутренних деталей; степпер не активируется

#### Scenario: Пометка точки невозврата

- **GIVEN** операция decommission или transfer
- **WHEN** отрисовывается её степпер
- **THEN** необратимость (отзыв доступа / частично необратимый перенос) визуально
  обозначена в прогрессе
