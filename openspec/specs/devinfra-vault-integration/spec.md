# devinfra-vault-integration Specification

## Purpose

Подключение реального HashiCorp Vault (dev-режим) к DevInfra worker для интеграционной проверки
воркфлоу (создание/смена владельцев/decommission/transfer) против настоящего Vault API — отдельным
compose-профилем, с детерминированным сидом и локальным ручным прогоном (ADR-0020). GitLab (реальный
или мок — по профилю) и Harbor остаются моками (граница MVP).

## Requirements

### Requirement: Реальный HashiCorp Vault в отдельном compose-профиле

Стенд ДОЛЖЕН (MUST) предоставлять реальный HashiCorp Vault как ОТДЕЛЬНЫЙ compose-override
(`deploy/compose/docker-compose.vault.yml`), подключаемый поверх базовой локалки и НЕ входящий в
дефолтный `docker compose up`. Подключение профиля НЕ ДОЛЖНО (MUST NOT) ломать базовую локалку
(`docker-compose.yml`), e2e-override (`docker-compose.e2e.yml`), gitlab-профиль
(`docker-compose.gitlab.yml`) и conformance-таргет. Образ Vault ДОЛЖЕН (MUST) быть пиннут по тегу
версии и запускаться в dev-режиме с фиксированным root-токеном (`VAULT_DEV_ROOT_TOKEN_ID`). При
активном профиле `devinfra-worker` ДОЛЖЕН (MUST) направлять Vault-вызовы на реальный Vault, а GitLab
и Harbor ОСТАЮТСЯ моками (если их собственные профили не активированы).

#### Scenario: Дефолтная локалка не поднимает реальный Vault

- **WHEN** выполняется `docker compose -f deploy/compose/docker-compose.yml up`
- **THEN** реальный Vault не запускается, а `devinfra-worker` ходит к WireMock-моку Vault

#### Scenario: Профиль реального Vault активируется отдельно

- **WHEN** стенд поднимается с override `docker-compose.vault.yml`
- **THEN** запускается реальный Vault (dev-режим, фиксированный root-токен), а `devinfra-worker`
  направляет Vault-вызовы на него, оставляя GitLab и Harbor моками

### Requirement: Health-gate и бюджет ожидания готовности Vault

Сервис Vault в compose ДОЛЖЕН (MUST) иметь healthcheck (`GET /v1/sys/health`), а зависящие
сервисы/тесты ДОЛЖНЫ (MUST) дожидаться его готовности ретраи-поллингом без фиксированных `sleep`.
Бюджет ожидания ДОЛЖЕН (MUST) быть настраиваемым (переменная окружения) с разумным дефолтом; поскольку
Vault в dev-режиме стартует за секунды, дефолт ДОЛЖЕН (MUST) быть существенно меньше бюджета GitLab.

#### Scenario: Тесты ждут готовности Vault без sleep

- **GIVEN** поднимаемый реальный Vault, ещё не прошедший инициализацию
- **WHEN** запускается интеграционный набор
- **THEN** набор дожидается готовности Vault ретраи-поллингом в пределах настраиваемого бюджета и не
  использует фиксированных `sleep`-констант

### Requirement: Детерминированный сид Vault для тестов

Стенд ДОЛЖЕН (MUST) детерминированно засевать реальный Vault одноразовым сид-шагом: root-токен —
фиксированное значение фикстуры стенда (`VAULT_DEV_ROOT_TOKEN_ID`), включённый AppRole auth-движок
(`vault auth enable approle`), смонтированный KV v2 на пути `secret/` (dev-дефолт). `devinfra-worker`
ДОЛЖЕН (MUST) аутентифицироваться ИМЕННО этим root-токеном (`VAULT_TOKEN`). Токен НЕ ДОЛЖЕН (MUST NOT)
хардкодиться в коде сверх фикстур стенда. Per-service политики и AppRole-роли создаёт САМ клиент в
воркфлоу провизии, а НЕ сид.

#### Scenario: AppRole-движок включён до провизии

- **GIVEN** засеянный реальный Vault
- **WHEN** воркфлоу создаёт AppRole для сервиса (`/v1/auth/approle/role/...`)
- **THEN** запрос успешен, потому что AppRole auth-движок предварительно включён сидом

#### Scenario: Токен сидируется детерминированно и переиспользуется worker

- **GIVEN** стенд с реальным Vault
- **WHEN** Vault стартует с `VAULT_DEV_ROOT_TOKEN_ID` и стартует `devinfra-worker`
- **THEN** `devinfra-worker` аутентифицируется этим же значением (`VAULT_TOKEN`/`X-Vault-Token`)

### Requirement: Интеграционные тесты воркфлоу против реального Vault

ДОЛЖЕН (MUST) существовать интеграционный набор (build-тег `integration`), переиспользующий харнесс
`tests/e2e` (получение OIDC-токена, HTTP-клиент периметра, ожидание статусов без sleep, уникальные
имена), который гоняет воркфлоу через периметр и ассертит ФАКТИЧЕСКОЕ состояние в реальном Vault через
Vault API. Набор ДОЛЖЕН (MUST) активироваться только при заданной конфигурации реального Vault
(отдельный gate-env, например `VAULT_ADDR`/`VAULT_TOKEN`) и НЕ ДОЛЖЕН (MUST NOT) влиять на прогон
сквозных историй против моков (`make e2e`) и на gitlab-набор. Прогон ДОЛЖЕН (MUST) быть только
локальным ручным (не в CI). Набор НЕ ДОЛЖЕН (MUST NOT) ассертить сырые тексты ошибок Vault как контракт.

#### Scenario: Создание реально создаёт AppRole и политику

- **GIVEN** активный стенд с реальным Vault
- **WHEN** воркфлоу создания услуги доходит до `active`
- **THEN** через Vault API подтверждается, что AppRole-роль и per-service политика существуют, а
  `role-id` читается

#### Scenario: Decommission отзывает secret-id

- **GIVEN** активная услуга с созданным AppRole
- **WHEN** воркфлоу decommission доходит до `decommissioned`
- **THEN** через Vault API подтверждается, что активные `secret-id`-accessors роли отозваны (набор пуст)

#### Scenario: Transfer мигрирует секреты source→target

- **GIVEN** активная услуга с секретами по исходному пути `secret/data/<src>`
- **WHEN** воркфлоу переноса доходит до `active`
- **THEN** через Vault API подтверждается, что секреты доступны по целевому пути `secret/data/<dst>`,
  создана целевая политика, а исходный путь очищен

#### Scenario: Смена владельцев отражена в identity/policies

- **GIVEN** активная услуга
- **WHEN** воркфлоу смены владельцев добавляет владельца
- **THEN** через Vault API подтверждается, что identity-entity владельца существует и несёт политику
  роли

#### Scenario: Набор пропускается без конфигурации реального Vault

- **WHEN** интеграционный набор запускается без gate-env реального Vault
- **THEN** тесты против реального Vault пропускаются (skip), а прогон историй против моков (`make e2e`)
  не затрагивается

### Requirement: Makefile-цели локального прогона стенда с Vault

Makefile ДОЛЖЕН (MUST) предоставлять цели подъёма стенда с реальным Vault, сидирования, прогона
интеграционного набора и гашения стенда с очисткой томов (`vault-up`/`vault-seed`/`vault-test`/
`vault-down`/`vault`) по образцу `gitlab-*`. Цели ДОЛЖНЫ (MUST) выполнять прогон только локально и НЕ
ДОЛЖНЫ (MUST NOT) поднимать реальный Vault в CI.

#### Scenario: Полный локальный цикл

- **WHEN** оператор выполняет цель полного цикла (подъём → сид → прогон → гашение)
- **THEN** стенд с реальным Vault поднимается и сидируется, интеграционный набор прогоняется, а стенд
  гарантированно гасится с очисткой томов даже при падении тестов
