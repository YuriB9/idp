# devinfra-gitlab-integration Specification

## Purpose

Подключение реального GitLab CE к DevInfra worker для интеграционной проверки воркфлоу
(создание/смена владельцев/decommission/transfer) против настоящего GitLab API — отдельным
compose-профилем, с детерминированным сидом и локальным ручным прогоном (ADR-0019). Vault и
Harbor остаются моками (граница MVP).

## Requirements

### Requirement: Реальный GitLab CE в отдельном compose-профиле

Стенд ДОЛЖЕН (MUST) предоставлять реальный GitLab CE как ОТДЕЛЬНЫЙ compose-override
(`deploy/compose/docker-compose.gitlab.yml`), подключаемый поверх базовой локалки и НЕ
входящий в дефолтный `docker compose up`. Подключение профиля НЕ ДОЛЖНО (MUST NOT) ломать
базовую локалку (`docker-compose.yml`), e2e-override (`docker-compose.e2e.yml`) и
conformance-таргет. Образ GitLab ДОЛЖЕН (MUST) быть пиннут по тегу версии. При активном
профиле `devinfra-worker` ДОЛЖЕН (MUST) направлять GitLab-вызовы на реальный GitLab, а
Vault и Harbor ОСТАЮТСЯ моками.

#### Scenario: Дефолтная локалка не поднимает GitLab

- **WHEN** выполняется `docker compose -f deploy/compose/docker-compose.yml up`
- **THEN** реальный GitLab не запускается, а `devinfra-worker` ходит к WireMock-моку GitLab

#### Scenario: Профиль реального GitLab активируется отдельно

- **WHEN** стенд поднимается с override `docker-compose.gitlab.yml`
- **THEN** запускается реальный GitLab CE, а `devinfra-worker` направляет GitLab-вызовы на него,
  оставляя Vault и Harbor моками

### Requirement: Health-gate и бюджет ожидания готовности GitLab

Реальный GitLab CE стартует медленно (минуты), поэтому сервис GitLab в compose ДОЛЖЕН (MUST)
иметь healthcheck, а зависящие сервисы/тесты ДОЛЖНЫ (MUST) дожидаться его готовности
ретраи-поллингом без фиксированных `sleep`. Бюджет ожидания ДОЛЖЕН (MUST) быть настраиваемым
(переменная окружения) с разумным дефолтом, покрывающим холодный старт GitLab.

#### Scenario: Тесты ждут готовности GitLab без sleep

- **GIVEN** поднимаемый реальный GitLab, ещё не прошедший инициализацию
- **WHEN** запускается интеграционный набор
- **THEN** набор дожидается готовности GitLab ретраи-поллингом в пределах настраиваемого
  бюджета и не использует фиксированных `sleep`-констант

### Requirement: Детерминированный сид GitLab для тестов

Стенд ДОЛЖЕН (MUST) детерминированно засевать реальный GitLab одноразовым сид-шагом:
root-пароль фиксированным значением фикстуры стенда, предсозданные группы под проекты
каталога (`demo`, `demo2`) и тест-пользователей под владельцев. Сид ДОЛЖЕН (MUST) выпускать
root-PAT фиксированным значением фикстуры (поскольку password-grant/ROPC в GitLab отключён,
выпуск выполняется через `gitlab-rails runner` внутри контейнера GitLab), и `devinfra-worker`
ДОЛЖЕН (MUST) аутентифицироваться ИМЕННО этим значением (`GITLAB_TOKEN`). Токен НЕ ДОЛЖЕН
(MUST NOT) хардкодиться в коде сверх фикстур стенда. CI-runner и пайплайны GitLab НЕ требуются
(репозитории создаются пустыми).

#### Scenario: Группы под проекты предсозданы

- **GIVEN** засеянный реальный GitLab
- **WHEN** воркфлоу создаёт репозиторий для проекта `demo`
- **THEN** репозиторий создаётся в предсозданной группе `demo` (резолвинг пути группы в
  `namespace_id`)

#### Scenario: Токен сидируется детерминированно и переиспользуется worker

- **GIVEN** стенд с реальным GitLab
- **WHEN** одноразовый сид-шаг отрабатывает и стартует `devinfra-worker`
- **THEN** сид выпускает root-PAT фиксированным значением фикстуры (через
  `gitlab-rails runner`), и `devinfra-worker` аутентифицируется этим же значением (`GITLAB_TOKEN`)

### Requirement: Интеграционные тесты воркфлоу против реального GitLab

ДОЛЖЕН (MUST) существовать интеграционный набор (build-тег `integration`), переиспользующий
харнесс `tests/e2e` (получение токена, HTTP-клиент периметра, ожидание статусов без sleep,
уникальные имена), который гоняет воркфлоу через периметр и ассертит ФАКТИЧЕСКОЕ состояние в
реальном GitLab через GitLab API. Набор ДОЛЖЕН (MUST) активироваться только при заданной
конфигурации реального GitLab (отдельный gate-env) и НЕ ДОЛЖЕН (MUST NOT) влиять на
существующий прогон сквозных историй против моков. Прогон ДОЛЖЕН (MUST) быть только локальным
ручным (не в CI).

#### Scenario: Создание реально создаёт репозиторий в нужной группе

- **GIVEN** активный стенд с реальным GitLab
- **WHEN** воркфлоу создания услуги в проекте `demo` доходит до `active`
- **THEN** через GitLab API подтверждается, что репозиторий существует в группе `demo`, а
  CI/CD-переменные репозитория заданы

#### Scenario: Decommission архивирует репозиторий

- **GIVEN** активная услуга с созданным репозиторием
- **WHEN** воркфлоу decommission доходит до `decommissioned`
- **THEN** через GitLab API подтверждается, что репозиторий архивирован (`archived=true`)

#### Scenario: Transfer переносит репозиторий в целевую группу

- **GIVEN** активная услуга в проекте `demo`
- **WHEN** воркфлоу переноса в `demo2` доходит до `active`
- **THEN** через GitLab API подтверждается, что репозиторий находится в группе `demo2` и
  отсутствует в группе `demo`

#### Scenario: Набор пропускается без конфигурации реального GitLab

- **WHEN** интеграционный набор запускается без gate-env реального GitLab
- **THEN** тесты против реального GitLab пропускаются (skip), а прогон историй против моков
  (`make e2e`) не затрагивается

### Requirement: Makefile-цели локального прогона стенда с GitLab

Makefile ДОЛЖЕН (MUST) предоставлять цели подъёма стенда с реальным GitLab, прогона
интеграционного набора и гашения стенда с очисткой томов по образцу `e2e-up`/`e2e-test`/
`e2e-down`. Цели ДОЛЖНЫ (MUST) выполнять прогон только локально и НЕ ДОЛЖНЫ (MUST NOT)
поднимать GitLab в CI.

#### Scenario: Полный локальный цикл

- **WHEN** оператор выполняет цель полного цикла (подъём → прогон → гашение)
- **THEN** стенд с реальным GitLab поднимается, интеграционный набор прогоняется, а стенд
  гарантированно гасится с очисткой томов даже при падении тестов
