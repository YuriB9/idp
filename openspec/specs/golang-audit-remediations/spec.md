## Requirements
### Requirement: Воспроизводимый govulncheck-гейт (D1)

CI ДОЛЖЕН (MUST) запускать govulncheck зафиксированной версией, а не из
`@latest`. Версия SHALL быть пинована явно (`@vX.Y.Z`) либо инструмент SHALL
запускаться из пинованного `tools/bin`. Гейт остаётся блокирующим и
`GOWORK=off`.

#### Scenario: Версия govulncheck зафиксирована

- **WHEN** проверяется джоба govulncheck в `.github/workflows/ci.yml`
- **THEN** в ней нет `govulncheck@latest`
- **AND** используется явно пинованная версия или бинарь из `tools/bin`

#### Scenario: Гейт остаётся блокирующим

- **WHEN** в зависимости модуля есть известная уязвимость
- **THEN** джоба govulncheck падает (CI красный)

### Requirement: Группировка Dependabot и согласованный кросс-модульный tidy (D2)

`.github/dependabot.yml` ДОЛЖЕН (MUST) содержать `groups:` для gomod-обновлений
общих зависимостей. Репозиторий SHALL предоставлять команду `make tidy-all`,
прогоняющую `go mod tidy` (`GOWORK=off`) по каждому затронутому модулю в порядке
зависимостей, чтобы bump общей зависимости тидился консистентно без diff (не
повторяя падение PR #75).

#### Scenario: Dependabot группирует gomod-обновления

- **WHEN** проверяется конфигурация Dependabot
- **THEN** для gomod задана секция `groups:`

#### Scenario: tidy-all держит модули согласованными

- **WHEN** общая зависимость бампнута и выполнен `make tidy-all`
- **THEN** `go mod tidy` (`GOWORK=off`) во всех затронутых модулях, включая
  `devinfra-worker`, не даёт diff

### Requirement: Выдача токена сервис-аккаунта без удержания мьютекса на время I/O (C1/R1)

Клиент Keycloak ДОЛЖЕН (MUST) НЕ удерживать мьютекс на время сетевого запроса
токена сервис-аккаунта. Дедупликация одновременных запросов токена SHALL
обеспечиваться double-checked-секцией или `singleflight`; при использовании
`singleflight` фоновая загрузка SHALL быть устойчива к отмене контекста лидера
(`context.WithoutCancel` при необходимости). Кэш токена в памяти сохраняется,
секрет не логируется.

#### Scenario: Сетевой запрос вне критической секции

- **WHEN** истёк кэшированный токен и несколько горутин одновременно вызывают
  справочник
- **THEN** сетевой запрос токена выполняется без удержания мьютекса
- **AND** конкурентные обращения не сериализуются на длительность запроса

#### Scenario: Корректность под гонками сохранена

- **WHEN** выполняется `go test -race` пакета identity
- **THEN** гонок не обнаружено, тесты зелёные

### Requirement: Ограниченно-параллельный резолв субъектов (P1)

Метод `Resolve` клиента Keycloak ДОЛЖЕН (MUST) выполнять резолв набора субъектов
ограниченно-параллельно (`errgroup.Group` с `SetLimit`), а не N последовательными
запросами. Порядок результата SHALL соответствовать порядку входных субъектов;
семантика «осиротевшего» субъекта (`Found=false` вместо ошибки) сохраняется.

#### Scenario: Параллельный резолв с ограничением

- **WHEN** `Resolve` вызывается для набора из нескольких субъектов
- **THEN** запросы выполняются параллельно с ограничением степени параллелизма

#### Scenario: Порядок и семантика сохранены

- **WHEN** часть субъектов отсутствует в каталоге
- **THEN** результат идёт в порядке входа, отсутствующие помечены `Found=false`

### Requirement: Сохранение внешнего поведения и зелёного CI

Ремедиации ДОЛЖНЫ (MUST) НЕ менять внешнее поведение (контракты gRPC/OpenAPI,
коды ответов, fail-closed-семантику RBAC и SSRF). Все Go-джобы CI
(`go test -race -shuffle=on`, `golangci-lint`, `govulncheck`,
`go mod tidy --check`, integration) SHALL оставаться зелёными.

#### Scenario: Внешнее поведение не изменилось

- **WHEN** прогоняются существующие тесты периметра и сервисов
- **THEN** они проходят без изменения ожидаемых ответов

#### Scenario: Новая зависимость согласована по модулям

- **WHEN** в `services/idm` добавляется зависимость (например, errgroup)
- **THEN** `go mod tidy` (`GOWORK=off`) в затронутых модулях не даёт diff
