# ADR-0023: Обязательные владельцы при создании сервиса и полное установление владения в воркфлоу

**Status:** Accepted
**Date:** 2026-06-27
**Change:** service-create-requires-owner

## Context

Создание сервиса было двухфазным: `CreateService` вставлял запись каталога со
`status=CREATING` БЕЗ владельцев, затем `CreateServiceWorkflow` провизионировал
GitLab → Harbor → Vault → инъекцию секретов → активацию, не устанавливая ролей
владельцев. Владельцы появлялись только позже, отдельной операцией
`SetServiceOwners` → `ChangeOwnersWorkflow` (GitLab members → Vault policies →
запись в каталог [точка невозврата] → IDM owner roles, Saga-компенсации,
ADR-0011). Между созданием и первой сменой владельцев сервис существовал «без
владельца» — нет ответственного субъекта ни в каталоге, ни во внешних системах.
Это противоречит контракту владения (ADR-0011): у каждого сервиса должен быть
владелец.

Контракт создания двойной: proto `CreateServiceRequest{project, name}` и openapi
`CreateServiceRequest required:[name]`. Таблица владельцев `service_owners`,
колонка `owners_version`, нормализация `normalizeOwners` и activities
синхронизации владельцев (`GitLabSyncMembers`/`VaultSyncOwners`/
`IDMSyncOwnerRoles`) уже существуют.

Ключевой вопрос: ЧТО значит «указать владельца при создании»?

- **Вариант A — только запись в каталог.** Owners нормализуются и пишутся в
  запись каталога атомарно при вставке (`owners_version=1`); воркфлоу создания НЕ
  меняется — роли во внешних системах не устанавливаются до отдельного
  `ChangeOwners`. Минимально, но владелец имеет доступ ТОЛЬКО в каталоге.
- **Вариант B — полное установление владельца.** Owners пишутся в каталог И
  воркфлоу создания устанавливает реальные роли (GitLab members → Vault policies →
  IDM owner roles) с Saga-компенсацией, согласованно с ADR-0011.

## Decision

Принят **вариант B**.

1. **Контракт (BREAKING).** `CreateServiceRequest` получает обязательное поле
   владельцев на обоих контрактах: proto `repeated string owners = 3;`; openapi
   `required:[name, owners]`, `owners` — массив `minItems:1`, элементы
   `minLength:1`. Перегенерация чистая (buf + web-кодоген), `gen:check` зелёный.

2. **Атомарная запись владельцев.** Вставка записи каталога становится
   транзакционной: строка `services` (status=`creating`) + строки
   `service_owners` + `owners_version = 1` в одной транзакции. Владельцы
   нормализуются (`normalizeOwners`) в usecase; пустой набор после нормализации
   отклоняется (`InvalidArgument`) ДО вставки и запуска воркфлоу. Сохраняется
   строгий порядок «запись фиксируется первой» и best-effort `creating→failed`
   при сбое запуска (guarded-CAS, ADR-0004). `owners_version=1` консистентно с
   первой `SetOwners` (0→1). Инвариант: записи каталога без владельца не
   существует.

3. **Установление ролей в воркфлоу создания.** `CreateServiceWorkflow` расширяется
   шагами назначения владельцев, переиспользуя activities `changeowners` с
   `add=owners, remove=[]`. Порядок: GitLab create repo → GitLab sync members →
   Harbor create → Vault setup → Vault sync policies → inject secrets → активация
   (`creating→active`) → IDM sync owner roles. Компенсации членства GitLab и
   политик Vault ПОГЛОЩАЮТСЯ существующими компенсациями удаления ресурса (delete
   repo / teardown Vault) — отдельные restore не нужны. IDM-роли — ПОСЛЕ
   активации, best-effort с алертом оператору при исчерпании ретраев, без
   молчаливого отката (каталог — источник правды, ADR-0005/0008).

4. **Без проверки существования владельцев в IDM** при создании — консистентно с
   `ChangeOwners` (MVP без существ. проверки, ADR-0016). Создатель НЕ добавляется
   во владельцы неявно; форма может предлагать его как дефолт (UX), не
   обязательность.

5. **Периметр и портал.** Gateway прокидывает `owners`, синхронно валидирует
   (непусто/без пустых/без дублей) → 400 без сырых деталей (ADR-0009). Форма
   создания получает обязательное поле владельцев (reuse `parseOwners`), zod из
   кодогена. Подписи `OPERATION_STEPS.create` (ADR-0022) обновляются под новый
   порядок активностей; грубая гранулярность «одна фаза на все шаги» сохраняется.

## Consequences

- Владелец реально владеет сервисом «с первой секунды»; окно «бесхозного
  сервиса» устранено и в каталоге, и во внешних системах.
- BREAKING для всех вызывающих создание (web, e2e/harness, integration-тесты) —
  обновляются в этом же change.
- Воркфлоу создания усложняется (шаги владельцев + порядок), но переиспользует
  существующие idемпотентные activities и компенсации; temporal-тесты покрывают
  happy-path и откат до активации.
- Сбой IDM-ролей после активации оставляет сервис `active` без ролей в IDM
  (допустимо, как в ChangeOwners) — самовосстановление повторным ChangeOwners.
- Вариант A отвергнут: оставляет владельца без фактического доступа во внешних
  системах до отдельного ChangeOwners, воспроизводя устраняемую проблему.

Реализует/уточняет: ADR-0011 (owners/role-sync), ADR-0004 (guarded-CAS),
ADR-0005 (Saga), ADR-0008 (split воркфлоу), ADR-0009 (форма периметра),
ADR-0016 (справочник субъектов IDM), ADR-0022 (единый прогресс).
