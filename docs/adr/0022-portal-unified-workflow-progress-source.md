# ADR-0022: Единый ступенчатый прогресс операций в портале и источник «шагов воркфлоу»

**Status:** Accepted
**Date:** 2026-06-27
**Change:** portal-unified-workflow-progress

## Context

В портале четыре асинхронные операции над сервисом запускают Temporal-воркфлоу в
сервисе projects с фиксированным порядком активностей (реальные плечи DevInfra —
ADR-0019/0020/0021):

- создание: GitLab → Harbor → Vault → инъекция секретов → `active`;
- смена владельцев: GitLab → Vault → каталог (`owners_version++`) → IDM (статус не меняется);
- decommission: load-drained → GitLab archive → Harbor read-only → Vault revoke → каталог;
- transfer: каталог begin (`active→transferring`) → GitLab transfer → Vault migrate →
  Harbor metadata → каталог commit → роли IDM.

UX был рассогласован: создание показывало однострочную панель на `ServiceProgressPage`,
остальные операции — мутацию с тостом в углу (`Owners/Decommission/TransferCard`).
Дизайн-система (ADR-0017) не имела примитива ступенчатого прогресса.

Ключевой архитектурный вопрос: ОТКУДА брать «шаги от темпорала»?

- **Вариант A** — реальные per-activity шаги Temporal: воркер/projects отдают список
  шагов (через Temporal Query/DescribeWorkflowExecution или прогресс, репортируемый в
  каталог) → новое поле/эндпоинт периметра → изменение контракта (openapi + кодоген +
  zod/TS-клиент). Богаче, но это backend+contract работа, пересекающая Saga-откат
  (ADR-0005), guarded-CAS (ADR-0004), split воркфлоу (ADR-0008), точки невозврата
  (ADR-0012/0013); `gen:check` меняется (нужна чистая перегенерация).
- **Вариант B** — шаги выводятся клиентски из ГРУБОГО статуса (`ServiceStatus:
  [creating, active, decommissioned, failed, transferring]`) + доменных полей
  (`owners_version`, `decommissioned_at`). Контракт периметра не меняется; «шаги» —
  крупные фазы, не истинная per-activity гранулярность.

## Decision

1. **Вводится единый презентационный примитив ступенчатого прогресса** (`ui/stepper.tsx`)
   в дизайн-системе портала (ADR-0017) с состояниями `pending|running|done|failed`,
   доступностью (`role`/`aria-live="polite"`) и motion-safe-анимациями. Им показывается
   ход ВСЕХ операций над сервисом (создание/owners/decommission/transfer) вместо
   панель-для-создания/тост-для-остального.

2. **Источник шагов — вариант B.** Шаги — фиксированная клиентская модель на операцию
   (порядок = порядок активностей воркфлоу из `services/projects/*`); текущая фаза
   выводится из грубого статуса + доменных полей. Контракт периметра, кодоген и
   `openapi.yaml` НЕ меняются (`gen:check` зелёный без перегенерации).

3. **Честная гранулярность.** Периметр несёт только грубый статус, поэтому B не
   утверждает, какая именно активность идёт/упала. Для create/transfer промежуточные
   статусы (`creating`/`transferring`) дают фазовость; для decommission и смены владельцев
   промежуточного статуса нет → степпер показывает «выполняется → завершено/ошибка».
   `failed` отражает факт компенсирующего отката (Saga, ADR-0005) без атрибуции шага.

4. **Судьба тостов.** Тост остаётся ТОЛЬКО для синхронных ошибок запуска операции
   (валидация и 4xx прав/конфликта/предусловия через `lib/errors`); асинхронный ход и
   исход показывает степпер, а не тост.

Выбор B обоснован соразмерностью цели (унификация UX достигается на фронте), сохранением
зелёного `gen:check` и быстрым инкрементом в серии завершённых работ.

## Consequences

- Чисто клиентское изменение: контракт периметра, кодоген, бэкенд и воркфлоу не
  затрагиваются; миграций данных нет; откат — revert PR.
- UI не показывает истинную per-activity гранулярность Temporal и конкретный упавший шаг
  (ограничение B, зафиксировано в spec и в самом UI).
- Сервис projects остаётся источником истины статусов (guarded-CAS, ADR-0004); клиент
  только читает статус поллингом (без `sleep`, ADR-0018); двойной запуск защищён CAS→409.

## Alternatives considered

- **Вариант A (расширение контракта полем `steps[]`).** Отвергнут для этой задачи как
  непропорциональный цели и затрагивающий контракт/кодоген/бэкенд. Зафиксирован как
  естественное продолжение отдельной задачей: добавить `steps[] {key,label,state}` в
  `ServiceSummary` (или under-status-эндпоинт), наполнять из воркера (heartbeat-прогресс
  в каталог или Temporal Query из projects), чистая перегенерация кодогена; Saga-откат —
  пометкой `failed`/`done` на шагах. При выборе A эта ADR помечается `Superseded by`.
- **Полный отказ от тостов.** Отвергнут: синхронный отказ прав/конфликта до старта
  воркфлоу нужно показать немедленно, степпер для этого не подходит.
