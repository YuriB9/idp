# ADR-0010: Модель RBAC сервиса IDM и стратегия кэширования решений

**Status:** Accepted
**Date:** 2026-06-19
**Change:** idm-rbac-min

## Context

ADR-0003 зафиксировал, что авторизация прав — отдельный сервис IDM (gRPC
`CheckAccess`, Postgres + DragonflyDB-кэш), но НЕ описал саму модель данных и
стратегию кэширования. На этапе наполнения IDM (Этап 1 docs/IDP_MVP_plan.md)
нужно зафиксировать: какова минимальная модель ролей/прав, как принимается
решение, как кэшируются решения с защитой от stampede (БЛОК 5) и как кэш
инвалидируется, чтобы реализация была согласованной и расширяемой без смены
gRPC-контракта.

## Decision

- **Модель — нормализованный минимальный RBAC** в Postgres: `roles`,
  `permissions(action, resource)`, `role_permissions` (M:N), `subject_roles`
  (`subject` = `sub` из JWT). Решение `allowed=true` ⇔ существует цепочка
  `subject_roles → role_permissions → permissions` с совпадением `(action,
  resource)`; иначе **deny-by-default**. `resource` — строка целевого ресурса
  (например, `project:demo`), матчинг строгий (без wildcard в MVP). Контракт
  `proto/idm/v1` не меняется.
- **Кэш решений в DragonflyDB** (клиент `redis/go-redis/v9`): ключ
  `idm:decision:<gen>:{subject}:{resource}:{action}`, значение allow/deny,
  конечный TTL (включая negative caching отказов). Промах кэша → чтение
  Postgres под `singleflight` по ключу решения (один запрос в БД на N
  конкурентных одинаковых промахов, anti-stampede).
- **Инвалидация** — версионным префиксом `<gen>` (служебный ключ
  `idm:cache:gen`), бампаемым при изменении ролей/прав/привязок, плюс точечное
  удаление по субъекту для адресных изменений. Старые решения становятся
  недостижимыми без сканов.
- **Fail-closed**: недоступная БД → отказ; недоступный кэш → деградация к
  прямому чтению Postgres (а не молчаливое разрешение); наружу — только
  `allowed/reason`, без `err.Error()`. `AUTH_DISABLED` — только локалка.

## Consequences

**Положительные:** честный RBAC, расширяемый (роли/права/ресурсы) без смены
gRPC-контракта; кэш с предсказуемым TTL и защитой от stampede; корректная
инвалидация без сканов; поведение fail-closed соответствует ADR-0003 и БЛОК 2.

**Отрицательные:** версионный bump инвалидирует весь кэш решений при любом
изменении ролей — приемлемо для редких seed/админ-правок MVP, но грубо для
частых изменений (точечное удаление по субъекту смягчает); двойной вызов
`CheckAccess` (периметр + projects, defense-in-depth) добавляет латентность,
амортизируемую кэшем; negative caching создаёт ограниченное TTL-окно «залипания»
отказа после выдачи права.

**Альтернативы (отклонены):** denormalized право в `subject_roles` — проще
чтение, но дублирование и сложная инвалидация; кэширование ролей субъекта с
вычислением решения в памяти — сложнее инвалидация, чем кэш готового решения.

**Связь:** реализует и уточняет ADR-0003 (модель auth/RBAC); использует goose
из `./tools` (ADR-0007) для миграций IDM.
