# ADR-0013: Семантика переноса сервиса, точка невозврата и двусторонняя авторизация

**Status:** Accepted
**Date:** 2026-06-20
**Change:** transfer-service

## Context

Пятый и последний сквозной сценарий MVP — «Перенос сервиса» (transfer),
docs/IDP_MVP_plan.md (Этап 3, «Перенос сервиса»; порядок реализации — пункт 5,
самый рискованный, делается последним). Перенос меняет проект-владельца сервиса с
переносом репозитория GitLab в новую группу, миграцией путей/политик Vault,
обновлением метаданных/прав Harbor, переносом ролей владельцев в IDM и обновлением
каталога. Сегодня доменного пути переноса нет: нет RPC, нет workflow, нет ручки
периметра/портала, нет RBAC-действий переноса, нет операции смены колонки
`project` и нет соответствующих activities в DevInfra worker.

Перенос — особый риск: transfer GitLab и миграция Vault частично необратимы, а
операция затрагивает ДВА проекта. Нужно зафиксировать: семантику переноса
(сохранение id записи, смена колонки `project`, допустимые исходные статусы,
поведение при занятом имени в target, транзитный статус), форму контракта и
идемпотентность, двустороннюю авторизацию (какие действия и где точки `CheckAccess`),
точное расположение точки невозврата и политику алерта, перенос ролей IDM с
инвалидацией. Опирается на ADR-0001 (Temporal), ADR-0003 (auth/fail-closed),
ADR-0004 (guarded-CAS), ADR-0005/0008 (Saga, разделение определения/исполнения
workflow), ADR-0009 (периметр), ADR-0010 (RBAC и кэш), ADR-0011 (владельцы/
синхронизация ролей), ADR-0012 (soft-delete, разведение 409/422).

## Decision

- **Семантика переноса: смена `project`, сохранение id и владельцев.** Перенос
  меняет колонку `project` записи каталога `source→target`; **id записи
  СОХРАНЯЕТСЯ**, владельцы переезжают вместе с записью (FK `service_id`,
  `service_owners` не переписывается), `decommissioned_at` остаётся `NULL`.
  Уникальность `(project, name)` требует свободной пары `(target_project, name)` —
  занято → конфликт. Допустимый исходный статус — **только `active`**; прочие
  (`creating`/`failed`/`decommissioned`/`transferring`) → отказ-предусловие или
  конфликт. Имя сервиса при переносе сохраняется (без авто-переименования).
- **Транзитный статус `transferring` — вводится (аддитивно).** Добавляется
  `SERVICE_STATUS_TRANSFERRING = 5` (аддитивное значение enum) и расширяется
  `CHECK status IN (...,transferring)` (обратимая миграция goose). Обоснование:
  защита от конкурентных операций на время длинного рискованного переноса
  (guarded-CAS «забирает» сервис атомарно: `active→transferring`) и наблюдаемость
  (UI/оператор видят перенос; «зависание» в `transferring` после сбоя — явный
  сигнал к ручному разбору). Стоимость мала (не ломает wire/переходы). В отличие от
  decommission (где транзитного статуса не вводили), для самого рискованного
  сценария это оправдано.
- **Контракт `TransferService(project, name, target_project)`, идемпотентность.**
  Новый RPC помечается BREAKING в комментарии (добавление метода в существующий
  сервис). Идемпотентность: повторный вызов на уже перенесённом сервисе
  (`project=target`, `active`) → success (no-op), новый workflow не стартует.
  Отдельного expected-условия нет: целевое состояние однозначно по
  `(target_project, name)`, гонки закрывает транзитный статус + детерминированный
  `WorkflowID`.
- **Каталог: две guarded-CAS-фазы в транзакции (ADR-0004).** Фаза начала
  (`CatalogBeginTransfer`): в одной транзакции проверка свободы `(target, name)`
  (занято → `ErrConflict`) + guarded-CAS `active→transferring`; `RowsAffected==0` →
  разбор статуса (`transferring`→идёт; `creating`/`failed`/`decommissioned`→
  `ErrPrecondition`; конкурентная смена→`ErrConflict`). Фаза фиксации
  (`CatalogCommitTransfer`): guarded-CAS `transferring→active` + `project=target`
  (с повторной проверкой свободы); `RowsAffected==0`→`ErrConflict`; идемпотентна.
  Компенсация `CatalogAbortTransfer`: guarded-CAS `transferring→active`.
- **Двусторонняя авторизация: `transfer` (source) И `transfer_in` (target).**
  Перенос требует двух прав: `transfer` на `project:<source>` (вынести) и
  `transfer_in` на `project:<target>` (принять). Оба проверяются `CheckAccess` в
  ДВУХ точках (gateway + projects, defense-in-depth). Отказ/недоступность/ошибка
  IDM по ЛЮБОМУ из двух проектов → `PermissionDenied`/403 (fail-closed), без
  побочных эффектов и без раскрытия деталей. Это критичное security-решение:
  нельзя «вынести» сервис в чужой проект без права `transfer_in` на target.
- **Workflow «Перенос» (Saga, ADR-0005/0008)** в отдельном пакете
  `services/projects/transfer`, детерминированный `WorkflowID =
  "transfer:<source>:<name>"`. Порядок: (0) `CatalogBeginTransfer`
  (`active→transferring`, компенсируемо) → (1) **`GitLabTransferRepo` — ТОЧКА
  НЕВОЗВРАТА (необратимо)** → (2) `VaultMigratePaths` (копия секретов + новые
  политики + очистка старых) → (3) `HarborUpdateMetadata` → (4)
  `CatalogCommitTransfer` (`transferring→active`, `project=target`) → (5)
  `TransferOwnerRoles` (перенос ролей IDM + инвалидация). Сбой ДО точки невозврата
  (включая срыв запуска шага 1 после CAS начала) → идемпотентная компенсация
  `CatalogAbortTransfer`, внешние системы не трогаем. Сбой ПОСЛЕ (включая конфликт
  guarded-CAS фиксации) → НЕ молчаливый откат: форвард-only ретраи идемпотентных
  шагов, при исчерпании — алерт оператору структурным логом; сервис может остаться
  в `transferring` до ручного довыполнения. Каталог = целевой источник правды.
- **Точка невозврата на transfer GitLab.** PONR выбран на первом необратимом шаге —
  transfer репозитория GitLab: в MVP чистая компенсация (transfer-back) не
  моделируется (меняются namespace/URL), а полу-перенесённый репозиторий
  недопустим. Раньше PONR только каталог (компенсируемо), позже — форвард-only.
- **Перенос ролей владельцев IDM.** После фиксации каталога для каждого
  субъекта-владельца: `RevokeRole(owner:project:<source>)` +
  `AssignRole(owner:project:<target>)` + `InvalidateSubject` по всем затронутым
  субъектам (примитивы из ADR-0011, идемпотентны). В отличие от decommission (роль
  per-project не отзывалась во избежание over-revoke сиблингов, ADR-0012), при
  переносе сервис целиком покидает source-проект, поэтому отзыв в source + выдача в
  target корректны. Не оставлять устаревших `allow` в кэше.
- **Маппинг ошибок (переиспользует ADR-0012).** `PermissionDenied→403`,
  `NotFound→404`, занятое имя в target/конкурентный конфликт guarded-CAS
  (`ErrConflict→Aborted`/`AlreadyExists`)→409, недопустимый исходный статус
  (`ErrPrecondition→FailedPrecondition`)→422, невалидный запрос
  (`InvalidArgument`)→400, прочее→500 (без деталей). Новых правил `httpFromGRPC` не
  требуется.
- **Периметр (ADR-0009): `POST /projects/{project}/services/{name}/transfer`**,
  тело `{target_project}`. POST-действие на под-ресурсе (как `decommission`).
  Идемпотентно. gateway проверяет права на оба проекта перед проксированием.

## Consequences

**Положительные:** id записи и владельцы сохраняются (бесшовная смена владельца);
транзитный статус даёт защиту от гонок и наблюдаемость; идемпотентный,
гонко-устойчивый контракт; двусторонняя авторизация предотвращает «вынос» сервиса
в чужой проект; единый стиль Saga/guarded-CAS с принятыми ADR; явная PONR + алерт
вместо молчаливого отката необратимых шагов; корректный перенос ролей с
инвалидацией кэша; переиспользование разведения 409/422 (ADR-0012) без новых правил
маппинга.

**Отрицательные:** окно несогласованности после точки невозврата (репозиторий/
секреты перенесены, каталог ещё `transferring`) до форвард-ретрая/ручного
вмешательства; сервис может «зависнуть» в `transferring` при исчерпании ретраев;
перенос ролей предполагает сидированную роль `owner:project:<target>` (динамическое
создание ролей вне MVP); BREAKING-расширение контракта требует синхронной
регенерации Go/TS; добавление `transferring` в `CHECK` — миграция каталога.

**Альтернативы (отклонены):** одношаговая прямая смена `project` без транзитного
статуса (нет защиты от гонок и наблюдаемости для рискованного потока); одно
действие `transfer` только на source (позволяет вынести сервис в чужой проект без
прав на target); PONR позже transfer GitLab при моделировании transfer-back
(в MVP чистая компенсация group-transfer не моделируется → полу-перенесённый
репозиторий недопустим); откат после точки невозврата (необратимые шаги →
молчаливый откат запрещён ADR-0005); авто-переименование при занятом имени в target
(скрывает конфликт, неожиданно для пользователя); не отзывать роль владельца в
source (оставляет устаревший доступ, т.к. сервис покинул проект).

**Связь:** реализует Этап 3 docs/IDP_MVP_plan.md («Перенос сервиса», пункт 5
порядка реализации); опирается на
ADR-0001/0003/0004/0005/0008/0009/0010/0011/0012; использует goose из `./tools`
(ADR-0007).
