# ADR-0001: Temporal как оркестратор провизии ресурсов

**Status:** Accepted
**Date:** 2026-06-18

## Context

Каждая user story (создание / изменение владельцев / перенос / удаление сервиса) — это распределённая транзакция через несколько внешних API (GitLab, Vault, Harbor). При сетевом сбое нельзя оставлять ресурсы в полусозданном состоянии. Нужны: ретраи, видимость прогресса, компенсации (Saga), идемпотентность.

В предыдущем проекте на близком стеке оркестрация была на голом AMQP — это потребовало вручную писать supervisor-петли реконнекта консьюмеров, ack/nack/requeue-семантику и было источником значимой доли инцидентов («healthy, но мёртв»).

## Considered Options

1. **Temporal** — durable workflow, встроенные ретраи activity, история, компенсации в коде workflow.
2. **AMQP + собственная оркестрация** — как в прошлом проекте.
3. **Прямые синхронные вызовы из API** — без durable-слоя.

## Decision

Используем **Temporal**. Сервис проектов содержит API + Temporal worker (раздельные процессы); провизия идёт через workflows, внешние вызовы — через activities.

## Consequences

**Положительные:** durable-исполнение и ретраи из коробки; реконнект/ack-семантика уходят в SDK (снимаем главный класс инцидентов прошлого проекта); компенсации описываются явно в workflow; история даёт аудит и видимость прогресса для UI.

**Отрицательные:** новая операционная зависимость (Temporal cluster); команде нужно освоить модель workflow/activity и детерминизм workflow-кода.

**Адаптация уроков:** уроки прошлого проекта про консьюмеры/ack/реконнект НЕ копируются дословно — перекладываются на RetryPolicy/таймауты/heartbeat activity и non-retryable `ApplicationError` для ветки компенсации.
