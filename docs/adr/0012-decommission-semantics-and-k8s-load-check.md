# ADR-0012: Семантика вывода из эксплуатации (soft-delete) и проверка снятой нагрузки K8s

**Status:** Accepted
**Date:** 2026-06-20
**Change:** decommission-service


## Context

Четвёртый сквозной сценарий MVP — «Удаление / вывод из эксплуатации сервиса»
(soft delete / decommission), docs/IDP_MVP_plan.md (Этап 3, «Удаление сервиса»,
строки ~171–178). Статус `DECOMMISSIONED` уже есть в enum `proto/projects/v1` и в
`CHECK` каталога, а `repository.TransitionStatus` механически выражает переход
`ACTIVE→DECOMMISSIONED` (ADR-0004), но доменной операции soft-delete (сохранение
данных, идемпотентность, `decommissioned_at`), workflow обратных операций, ручки
периметра и действия RBAC нет. Decommission ≠ delete: это archive + отзыв доступа,
а не удаление ресурса; часть отзывов доступа необратима. Нужно зафиксировать:
семантику soft-delete (допустимые исходные статусы, `decommissioned_at`, судьбу
владельцев/ролей), форму контракта и идемпотентность, политику отказов workflow с
частично необратимыми шагами и — ОБЯЗАТЕЛЬНО — закрыть открытый вопрос плана о
проверке снятой нагрузки из K8s при отсутствии K8s-worker в MVP. Опирается на
ADR-0001 (Temporal), ADR-0003 (auth/fail-closed), ADR-0004 (guarded-CAS),
ADR-0005/0008 (Saga, разделение определения/исполнения workflow), ADR-0009
(периметр), ADR-0010 (RBAC и кэш), ADR-0011 (владельцы/синхронизация ролей).

## Decision

- **Soft-delete с сохранением данных.** Decommission переводит сервис
  `ACTIVE→DECOMMISSIONED` guarded-CAS-запросом (`UPDATE ... WHERE id=$id AND
  status='active'`, `RowsAffected==0 →` разбор статуса), проставляет
  `decommissioned_at` (новая nullable-колонка, обратимая миграция goose). Строка
  сервиса и владельцы НЕ удаляются (данные сохраняются, возможен будущий restore).
- **Допустимые исходные статусы и идемпотентность.** Допустим только `active`.
  Повторный вызов на `decommissioned` → идемпотентный success (no-op).
  `creating`/`failed` → отказ-предусловие. Конкурентная смена статуса → конфликт.
  Контракт `DecommissionService(project, name, load_drained)` идемпотентен.
- **Проверка снятой нагрузки K8s — явный чек-флаг/предусловие (НЕ прямой запрос к
  кластеру).** В MVP K8s-worker и управляемого кластера нет; «прямой запрос» был
  бы фиктивным/замоканным. Вызывающая сторона ЯВНО подтверждает снятие нагрузки
  (флаг `load_drained`), доменный слой трактует это как предусловие, проверяемое
  предварительным шагом (activity `EnsureLoadDrained` за интерфейсом `LoadChecker`)
  ДО любых отзывов доступа. Не выполнено → `FailedPrecondition` (HTTP 422) без
  побочных эффектов. Интерфейс `LoadChecker` — архитектурная граница: будущий
  K8s-worker подменит реализацию реальным запросом к кластеру без изменения ядра
  workflow и контракта.
- **Отзыв доступа во внешних системах (сервис-скоупно), роль владельцев per-project
  НЕ отзывается.** «Немедленное прекращение доступа» обеспечивается отзывом в
  GitLab (archive + revoke), Harbor (read-only + robot revoke) и Vault (отзыв
  активных SecretID/токенов — ключевое необратимое действие). Per-project роль
  `owner:project:<project>` при decommission одного сервиса НЕ отзывается — иначе
  она лишила бы доступа к остальным активным сервисам проекта (over-revoke).
  Программный отзыв роли (`RevokeRole`+`InvalidateSubject`) остаётся примитивом и
  будет подключён при переходе к сервис-скоупным ролям (Open Question).
- **Workflow «Вывод из эксплуатации» (Saga, ADR-0005/0008)** в отдельном пакете
  `services/projects/decommission`, детерминированный `WorkflowID =
  "decommission:<project>:<name>"`. Порядок: (0) предусловие снятой нагрузки → (1)
  GitLab archive+revoke → (2) Harbor read-only+robot revoke → (3) **Vault revoke
  SecretID — ТОЧКА НЕВОЗВРАТА (необратимо)** → (4) guarded-CAS
  `ACTIVE→DECOMMISSIONED` в каталоге. Сбой ДО точки невозврата → идемпотентные
  компенсации (Harbor→writable, GitLab→unarchive), каталог не трогаем. Сбой ПОСЛЕ
  (включая конфликт guarded-CAS) → НЕ молчаливый откат: форвард-only ретраи
  идемпотентных шагов, при исчерпании — алерт оператору структурным логом; каталог
  = целевой источник правды.
- **Маппинг ошибок: разведение конфликта (409) и предусловия (422).** Принимается
  gRPC-каноничное разделение: `codes.Aborted` = конкурентный конфликт (guarded-CAS)
  → 409; `codes.FailedPrecondition` = семантическое предусловие (нагрузка не снята /
  недопустимый исходный статус) → 422. В projects `repository.ErrConflict → Aborted`
  (включая путь владельцев `change-owners` — итоговый HTTP 409 сохраняется); в
  gateway `httpFromGRPC` добавляет `Aborted→409` и уточняет `FailedPrecondition→422`.
- **Новое действие `decommission`** на ресурсе `project:<project>`; проверка IDM
  `CheckAccess` в двух точках (gateway + projects, defense-in-depth, fail-closed) —
  как для `create`/`change_owners` (ADR-0003/0010).
- **Периметр (ADR-0009): `POST /projects/{project}/services/{name}/decommission`**
  (а не `DELETE`), тело `{load_drained}`. POST-действие выбрано, т.к. это
  soft-delete (данные сохраняются), `DELETE` подразумевал бы purge. Идемпотентно.

## Consequences

**Положительные:** честная, тестируемая граница проверки нагрузки без имитации
несуществующего кластера и с заделом под K8s-worker; сохранность данных и
возможность будущего restore; идемпотентный, гонко-устойчивый контракт и
soft-delete; немедленное прекращение доступа на уровне внешних систем
(сервис-скоупно); единый стиль Saga/guarded-CAS с принятыми ADR; корректное
разведение 409/422; fail-closed RBAC сохранён.

**Отрицательные:** окно несогласованности после точки невозврата (доступ отозван,
каталог ещё `active`) до успешного форвард-ретрая/ручного вмешательства; per-project
роль владельцев не отзывается (доступ режется только во внешних системах) до
введения сервис-скоупных ролей; смена `ErrConflict→Aborted` требует обновить тесты
projects/gateway (включая `change-owners`); BREAKING-расширение контракта требует
синхронной регенерации Go/TS.

**Альтернативы (отклонены):** прямой запрос к K8s-кластеру в MVP (фиктивен —
кластера/worker нет); `DELETE` как hard-delete или как soft-delete (двусмысленно с
purge); откат после точки невозврата (необратимый отзыв Vault → молчаливый откат
запрещён ADR-0005); отзыв per-project роли владельцев при decommission одного
сервиса (over-revoke доступа к сиблинг-сервисам); общий `FailedPrecondition→409`
без разведения конфликта и предусловия (теряется различимость 409/422).

**Связь:** реализует Этап 3 docs/IDP_MVP_plan.md («Удаление сервиса», пункт 4
порядка реализации); опирается на ADR-0001/0003/0004/0005/0008/0009/0010/0011;
использует goose из `./tools` (ADR-0007).
