# ADR-0014: IAM-админка — модель полномочий, читающий контракт и границы scope

**Status:** Accepted
**Date:** 2026-06-20
**Change:** iam-roles-portal

## Context

RBAC (IDM, ADR-0010) «слеп» со стороны UI: роли/права/привязки существуют только в
БД и сидируются миграциями, а портал умеет лишь project-scoped доменные операции
над сервисами. Нужно дать в портале просмотр ролей/прав и управление назначением
ролей субъектам (assign/revoke). Это ПЕРВЫЙ горизонтальный (не project-scoped)
сценарий периметра: gateway-helper `authorize` жёстко формировал
`resource="project:"+project`. Сама админка привилегированна — листинг ВСЕХ
субъектов/ролей и тем более мутации нельзя отдавать без права. Опирается на
ADR-0003 (auth/fail-closed), ADR-0009 (периметр/OpenAPI/TS), ADR-0010 (RBAC и
кэш), ADR-0011 (синхронизация ролей), ADR-0012/0013 (маппинг ошибок).

## Decision

- **Модель полномочий: раздельные `read`/`write` на горизонтальном `iam:global`.**
  `read` — все читающие операции каталога; `write` — assign/revoke. Укладывается в
  существующую модель (action+resource, strict-match, deny-by-default). Наименьшие
  привилегии: аудитор получает только `read`; `write` строго опаснее. Неявного
  наследования нет (ни между `read`/`write`, ни из/в project-действия).
- **Точки CheckAccess (fail-closed).** gateway вызывает `CheckAccess` перед КАЖДОЙ
  IAM-ручкой: GET → `(read, iam:global)`, POST/DELETE → `(write, iam:global)`.
  Отказ/недоступность/ошибка IDM → `403`, без проксирования; деталь — в лог по
  ключу slog `err`.
- **Обобщение gateway authorize.** Вводится `authorizeResource(resource, action)`;
  прежний `authorize(project, action)` — тонкая обёртка
  (`resource="project:"+project`). Нулевой регресс project-вызовов
  (create/list/read/owners/decommission/transfer).
- **Читающий контракт `IamAdminService` (аддитивно).** Новый сервис в
  `proto/idm/v1`: `ListRoles`, `ListPermissions`, `GetRolePermissions(role)`,
  `ListSubjectsWithRoles(page_size, page_token)`, `GetSubjectRoles(subject)`;
  сообщения `Role{name}`, `Permission{action,resource}`,
  `SubjectRoles{subject,roles[]}`. Изменение wire-совместимо (новый сервис +
  сообщения, существующие RPC не трогаются). Идентификатор роли — `name` (не `id`).
- **Пагинация.** Роли/права/права роли/роли субъекта — без пагинации (малые
  ограниченные справочники). `ListSubjectsWithRoles` — keyset по `subject` (ASC,
  серверный кламп `page_size`, непрозрачный `page_token`), как services (ADR-0009);
  offset отклонён (нестабилен при конкурентных вставках).
- **Перечисление субъектов — `DISTINCT subject` из `subject_roles`.** Реестра
  пользователей нет; субъект известен админке тогда и только тогда, когда у него
  есть роль. Субъекты без ролей не видны — принято (назначить роль можно любому
  subject по строке).
- **Идемпотентность и коды assign/revoke на периметре.** Переиспользуют
  идемпотентные `RoleAdminService.AssignRole`/`RevokeRole`. Повтор assign и revoke
  отсутствующей связки → `200`; несуществующая роль → `404`; пустые поля → `400`.
  Ответ несёт актуальный набор ролей субъекта (`{subject, roles[]}`), а не `204`:
  клиенту нужен свежий список для рантайм-валидации zod и обновления UI (как
  `setOwners`). `409`/`422` не применяются (нет конфликта состояния).
- **Инвалидация кэша.** Мутации идут через `usecase.RoleManager` →
  `InvalidateSubject(subject)` (не оставлять устаревший allow). Читающие методы —
  без побочных эффектов на кэш (только SELECT).
- **Маппинг ошибок (reuse ADR-0012/0013).** `PermissionDenied→403`,
  `NotFound→404`, `InvalidArgument→400`, прочее→500 (без деталей). Новых правил
  `httpFromGRPC` не требуется.
- **Без новых таблиц и guarded-CAS.** Change read-only по модели; единственная
  миграция — обратимый локальный seed роли `iam-admin` (`read`/`write` на
  `iam:global`) субъекту `demo-user`.

## Consequences

**Положительные:** привилегированный листинг и мутации защищены отдельным правом
(fail-closed); раздельные `read`/`write` дают наименьшие привилегии; аддитивный
wire-совместимый контракт; обобщённый authorize открывает горизонтальные сценарии
без регресса project-вызовов; идемпотентные assign/revoke с актуальным телом ответа
удобны UI; корректная инвалидация кэша; переиспользование маппинга ошибок без новых
правил.

**Отрицательные:** субъекты без ролей невидимы в списке (компромисс из-за
отсутствия реестра пользователей); горизонтальный ресурс `iam:global` — единая
точка полномочий (грубее проектной гранулярности); добавление сервиса требует
синхронной регенерации Go/TS (`gen:check`).

**Альтернативы (отклонены):** одно действие `manage_iam` (нельзя выдать
только-чтение аудитору; смешивает листинг и мутацию); per-endpoint действия
(избыточно для MVP); offset-пагинация субъектов (нестабильна); `204` без тела у
assign/revoke (UI лишается свежего набора ролей для zod-валидации/рендера);
наследование `iam:global` из project-прав (привилегия утекла бы неявно);
динамический CRUD ролей/прав и аудит-лог (вне MVP-scope).

**Связь:** реализует Этап 3 docs/IDP_MVP_plan.md (RBAC/IAM); опирается на
ADR-0003/0009/0010/0011/0012/0013; использует goose из `./tools` (ADR-0007).
