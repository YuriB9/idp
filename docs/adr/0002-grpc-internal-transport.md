# ADR-0002: gRPC/protobuf для внутренних вызовов

**Status:** Accepted
**Date:** 2026-06-18

## Context

Целевая C4-архитектура предписывает внутри платформы gRPC/protobuf, а https/json — только на стыке с порталом (через Oauth2-Proxy) и с внешними системами. MVP должен ложиться на целевую архитектуру без переписывания при росте.

## Considered Options

1. **gRPC/protobuf сразу** — как в целевой.
2. **REST для скорости, gRPC позже** — быстрее на старте, но миграция контрактов и клиентов позже.

## Decision

Внутренние вызовы (API-шлюз ↔ IDM, API-шлюз ↔ сервис проектов) — **gRPC/protobuf с первого дня**. .proto — источник правды контракта, кодоген Go-стабов. На периметре (портал ↔ API-шлюз) — OpenAPI + https/json, кодоген TS-клиента.

## Consequences

**Положительные:** соответствие целевой архитектуре; строгие контракты; отсутствие миграции REST→gRPC позже.

**Отрицательные:** чуть выше порог входа и сложнее отладка, чем у REST; нужен общий interceptor-стек (recovery, request-id, otel, auth) — заводим в `pkg`.

**Контракт:** изменения .proto помечаются BREAKING; wire-совместимость отслеживается.
