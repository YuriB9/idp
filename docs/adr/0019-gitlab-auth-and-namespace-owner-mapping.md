# ADR-0019: Модель аутентификации worker→GitLab и маппинг namespace/owner в GitLab

**Status:** Accepted
**Date:** 2026-06-22
**Change:** devinfra-real-gitlab

## Context

До этого изменения все три управляемые системы DevInfra (GitLab/Vault/Harbor) — WireMock-
моки (`deploy/mocks/mappings/*`). HTTP-клиент GitLab (`services/devinfra-worker/internal/
integrations/http.go`) писался под мок и расходится с реальным GitLab CE API: метод `call()`
не ставит аутентификацию; `CreateRepo` шлёт `namespace` строкой (реальный
`POST /api/v4/projects` ждёт `namespace_id` число); `SyncMembers` шлёт `username` (реальный
API ждёт `user_id` число); идемпотентные коды (409/404) угаданы под мок.

Это изменение (граница MVP) подключает РЕАЛЬНЫЙ GitLab CE вместо мока ТОЛЬКО для GitLab-плеча
(Vault и Harbor остаются моками), не меняя контракт периметра, proto и сгенерированный код.
Каталог проектов использует `project` как namespace (`demo`/`demo2`); владельцы — UUID
Keycloak (`subjAlice`/`subjBob`), которых в GitLab нет. Нужно зафиксировать аутентификацию
worker→GitLab и способ отображения namespace/owner на сущности GitLab. Опирается на ADR-0001
(Temporal/activities), ADR-0005 (Saga-откат создания — компенсация `DeleteRepo`), ADR-0012
(decommission → archive), ADR-0013 (transfer → GitLab transfer, точка невозврата), ADR-0018
(стенд-override и харнесс E2E).

## Decision

- **Аутентификация — `PRIVATE-TOKEN` со статическим root PAT.** Каждый запрос worker→GitLab
  несёт заголовок `PRIVATE-TOKEN: <token>` из конфигурации (`integrations.Config.GitLabToken`,
  env `GITLAB_TOKEN`). Токен не логируется; на Vault/Harbor не распространяется. Выбран
  статический PAT, а не OAuth/Bearer-обмен: проще сидировать детерминированно и не требует
  refresh для одного сервис-актора. Секрет — фикстура стенда, в код не хардкодится сверх неё;
  и сид GitLab, и worker берут одно env-значение.

- **Маппинг namespace→`group_id` — предсозданные группы + lookup с кэшем.** Сид стенда
  предсоздаёт группы с путями `demo`/`demo2`. Клиент резолвит путь группы в `group_id` через
  `GET /api/v4/groups/:path` и шлёт `namespace_id`; результат кэшируется в процессе.
  Автосоздание групп отклонено: namespace = проект каталога, GitLab лишь отражает его.

- **Маппинг владелец→`user_id` — предсозданные тест-пользователи + фикстура-маппинг.** Сид
  создаёт тест-пользователей; UUID-владелец отображается на GitLab-логин фикстурой стенда.
  Клиент резолвит логин в `user_id` через `GET /api/v4/users?username=...` и шлёт `user_id`.
  Незаданный маппинг владельца безопасно пропускается (не валит воркфлоу).

- **Идемпотентность — по кодам и GET-then-act, не по тексту ошибок.** Тексты ошибок GitLab не
  контракт и зависят от версии. Повторное создание/перенос проверяется предварительным GET;
  инъекция переменных — upsert (`PUT .../variables/:key`); удаление — `404`/`202` как успех.

- **Выбор реализации по наличию `GITLAB_TOKEN`.** Задан токен → клиент против реального
  GitLab; иначе — клиент против моков (поведение по умолчанию). In-memory стаб остаётся для
  дефолтного прогона. SSRF-guard для стендового GitLab на приватном http-адресе выключается
  `SSRF_DISABLED=true` (как у моков); в проде guard включён.

## Consequences

- Реальный GitLab CE поднимается отдельным compose-override (тяжёлый образ, медленный старт) —
  только локально/ручно, не в CI и не в дефолтной локалке; нужен health-gate и расширенный
  бюджет ожидания.
- Клиент усложняется резолвингом (group/user) и предварительными GET — это сетевые вызовы;
  кэш групп смягчает накладные расходы.
- Компенсации наблюдаемы против реального GitLab: `DeleteRepo` (откат создания, ADR-0005),
  `TransferRepo` как частично необратимый (точка невозврата, ADR-0013).
- Контракт периметра/proto/кодоген не затрагиваются (gen:check остаётся зелёным).
